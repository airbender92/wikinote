# 11.1 OAuth 令牌是什么？（精简翻译+要点）
## 中文翻译
令牌是所有 OAuth 交互的**核心**。
客户端从授权服务器获取令牌，然后提交给受保护资源。
授权服务器创建令牌并发放给客户端，同时在令牌中绑定资源所有者的委托授权与客户端权限。
受保护资源从客户端接收令牌并进行验证，将令牌附带的权限与客户端的请求进行匹配校验。

令牌代表**授权委托的结果**：它是资源所有者、客户端、授权服务器、受保护资源、权限范围（scope）以及所有授权相关信息的集合。
当客户端需要在不打扰用户的情况下刷新访问令牌时，会使用另一种令牌：**刷新令牌**。

OAuth 令牌是整个 OAuth 生态的核心机制，可以说**没有令牌就没有 OAuth**。甚至 OAuth 的非官方 Logo 都设计成了实体公交令牌的样子。

尽管如此强调令牌，但 OAuth **完全没有规定令牌内部应该包含什么内容**。
正如我们在第 2、3 章讨论的，OAuth 客户端**不需要知道令牌本身的任何信息**。
客户端只需要知道：如何从授权服务器拿令牌、如何在资源服务器使用令牌。
但**授权服务器和资源服务器必须理解令牌**：授权服务器要知道如何生成令牌，资源服务器要知道如何识别和验证令牌。

为什么 OAuth 核心协议会省略如此基础的内容？
因为不规定令牌格式，OAuth 才能灵活适用于各种部署环境、不同风险等级与不同需求。
OAuth 令牌可以过期、可被吊销、永久有效，或根据场景组合特性；
可以代表特定用户、系统内所有用户，甚至不代表任何用户；
可以有内部结构、可以是随机字符串、可以加密保护，也可以是以上方式的组合。
这种灵活性与模块化，让 OAuth 比 WS-*、SAML、Kerberos 等**硬性规定令牌格式**的安全协议更容易适配各类系统。

在实际系统中，有几种常见的令牌创建与验证方式，各有优缺点与适用场景。
在本书第 3、4、5 章的练习里，我们创建的令牌是**随机字母数字串**，类似这样：
`s9nR4qv7qVadTUssVD5DqA7oRLJ2xonn`

授权服务器生成令牌后，会把令牌信息存在共享数据库。
受保护资源收到令牌时，在同一个数据库里查询该令牌的权限。
这类令牌**内部不携带任何信息**，只作为数据库查询的索引键。
这是完全合法且常用的实现方式，优点是**令牌体积小、熵值高（安全性强）**。

但在很多场景下，授权服务器与受保护资源**共享数据库并不现实**（尤其是一台授权服务器保护多个下游资源时）。
那我们该用什么方案？
本章会介绍另外两种通用方案：**结构化令牌** 和 **令牌内省（Introspection）**。

# 11.2 结构化令牌：JWT（JSON Web Token）核心知识点总结
## 一、JWT 要解决什么问题？
- 之前的随机令牌需要**授权服务器和资源服务器共享数据库**
- JWT 让令牌**自己携带所有必要信息**，不用查库
- 授权服务器把信息“打包”进令牌，客户端不用理解，资源服务器可直接解析

## 二、JWT 结构
- 格式：**Header.Payload.Signature**（三部分用 `.` 分隔）
- 每一部分都是 **Base64URL 编码**
- 可以安全放在 URL、Header、表单参数里

### 1）Header（头部）
说明令牌类型、算法
示例：
```json
{ "typ": "JWT", "alg": "none" }
```

### 2）Payload（载荷）
存放要传递的信息，称为 **Claims（声明）**

## 三、JWT 标准声明（必须记住）
- **iss**：签发者（通常是授权服务器 URL）
- **sub**：主题（用户ID）
- **aud**：受众（哪些资源服务器可以用）
- **exp**：过期时间（Unix 秒）
- **nbf**：在此时间之前不可用
- **iat**：签发时间
- **jti**：令牌唯一ID（防重放）

## 四、为什么用 Base64URL？
- 能**安全放在 HTTP 各种位置**（header、url、form）
- 不需要额外转义
- 避免中间件意外修改 JSON

## 五、JWT 特点（OAuth 视角）
- 对客户端**完全不透明**，客户端代码不用改
- 资源服务器**不需要共享数据库**
- 可自包含、自验证、可过期
- 适合分布式、多资源服务场景

## 六、本章实现逻辑
1. 授权服务器生成 JWT（Header + Payload → Base64URL 拼接）
2. 设置 iss / sub / aud / exp / iat / jti
3. 资源服务器收到 JWT 后：
   - 按 `.` 拆分
   - 解码 Payload
   - 校验 iss、aud、iat、exp
4. 校验通过就认为令牌合法

## 七、本章重要结论
- JWT = **自带信息的安全令牌**
- 不需要共享数据库，适合分布式系统
- 客户端完全无感知，不用修改代码
- 资源服务器直接解析、校验即可

# 一句话核心答案
**JWT 使用 Base64URL 编码，是为了让令牌能安全地放在任何地方（URL、Header、表单、数据库），不需要额外转义，也不会被系统改错。**

# 完整通俗解释（逐段讲明白）
## 为什么不能直接用 JSON？
JSON 长这样：
```json
{ "sub": "123", "name": "Alice" }
```
里面有 **{ } " : , 空格** 等符号。

这些符号**不能直接放在 HTTP 里**，比如：
- URL 里不能直接用 `{ } "`
- 请求头里不能乱加特殊字符
- 表单参数会把特殊字符转义掉

如果你直接把 JSON 塞进去，系统会自动把它变成：
```
%7B%22sub%22%3A%22123%22%2C%22name%22%3A%22Alice%22%7D
```
又乱又容易出错，还会出现**双重转义**（越转越错）。

---

## Base64URL 解决了什么？
它把 JSON 转换成**只包含以下安全字符**：
- 字母 A-Z a-z
- 数字 0-9
- `-` `_`

这些字符**在任何地方都合法**：
- URL 查询参数 ✅
- HTTP 请求头 ✅
- 表单字段 ✅
- 数据库存储 ✅
- 各种编程语言字符串 ✅

→ **放哪里都不会被转义、不会出错、不会被改错。**

---

## 额外好处：不会被中间件改错
很多服务器、网关、代理会自动“修理”JSON，比如：
- 加空格
- 改排序
- 转义字符
- 重新格式化

一旦被改，签名就失效，JWT 就废了。

但 Base64URL 是**一串普通字符串**，网关/代理不会去动它，保证 JWT 从头到尾**原汁原味**。

---

# 最终超级浓缩版（必背）
## 为什么 JWT 要用 Base64？
1. **让令牌能安全放在任何地方**（URL、Header、表单、数据库）
2. **避免特殊字符被转义**（不会出现 %7B %22 这种乱码）
3. **防止双重编码**（不会越转越错）
4. **避免被中间件改错**（保持原样，保证签名有效）

# 11.2.2 JWT claims（JWT 标准声明）
## 一句话概括
**Claims = JWT 载荷里存放的“标准字段”**，用来放令牌的身份、时间、权限等信息。
JWT 本身可以存任何 JSON，但这些标准字段让所有系统都能统一识别。

---

# 7 个最重要的标准 Claim（必须记住）
## 1. **iss**
Issuer → **签发者**
谁发的令牌？（一般是授权服务器 URL）

## 2. **sub**
Subject → **主题**
令牌是属于谁的？（用户唯一 ID）

## 3. **aud**
Audience → **受众**
这个令牌是给谁用的？（资源服务器地址）
可以是一个字符串，也可以是数组（多个接收方）。

## 4. **exp**
Expiration → **过期时间**
Unix 时间戳（秒），到点就失效。

## 5. **nbf**
Not Before → **在此时间之前不可用**
令牌还没生效，没到时间不能用。

## 6. **iat**
Issued At → **签发时间**
令牌什么时候生成的。

## 7. **jti**
JWT ID → 令牌唯一 ID
防止重放攻击、防止重复、防止伪造。

---

# 超简版记忆口诀
**iss 谁发的
sub 属于谁
aud 给谁用
exp 何时过期
nbf 何时生效
iat 何时签发
jti 唯一标识**

---

# 关键知识点（书里的重点）
1. **所有 claim 都是可选的**，但业务系统可以强制要求必须带某些字段
2. 除了标准字段，**你可以随便加自定义字段**，比如 name、email、admin、scope
3. 标准字段是为了**跨系统兼容**，所有服务都认识
4. 时间都是 **Unix 时间戳（秒）**，不是毫秒
5. **aud 可以是数组**，表示令牌可以给多个服务用
6. **jti 是随机字符串**，防重放、防碰撞

# 二、授权服务器做了什么？
## 1. 删掉原来的随机令牌
```js
var access_token = randomstring.generate();
```

## 2. 生成 JWT 固定头部
```js
var header = { 'typ': 'JWT', 'alg': 'none' };
```
- `typ: JWT`：说明这是 JWT
- `alg: none`：**没有签名**（教学用，真实环境必须签名）

## 3. 生成 JWT 载荷（最重要）
```js
var payload = {
  iss: 'http://localhost:9001/',    // 谁签发
  sub: code.user.sub,              // 属于哪个用户
  aud: 'http://localhost:9002/',   // 给哪个资源服务器用
  iat: Math.floor(Date.now()/1000),// 签发时间
  exp: Math.floor(Date.now()/1000)+5*60, // 5分钟过期
  jti: randomstring.generate(8)    // 唯一ID
};
```

## 4. 拼接成 JWT
```js
var access_token =
  base64url编码(header) + "." +
  base64url编码(payload) + ".";
```
格式：
```
头部.载荷.（签名为空）
```

# 三、资源服务器做了什么？
## 1. 拆分 JWT
```js
var tokenParts = inToken.split('.');
```

## 2. 解码载荷
```js
var payload = JSON.parse(base64url.decode(tokenParts[1]));
```

## 3. 逐项校验（必须全过）
```js
1. iss === 授权服务器地址      ✅
2. aud === 自己的地址          ✅
3. iat <= 当前时间             ✅
4. exp >= 当前时间             ✅
```

## 4. 全部通过 → 令牌合法
资源服务器直接用 payload 里的信息做权限判断，**不用查库**。

# 11.3 核心一句话
**前面实现的无签名 JWT 极度不安全，任何人都能伪造/篡改。
必须用 JOSE 套件给令牌加签名/加密，才能在生产环境使用。**

---

# 一、作者坦白：我们刚做了一件极度不安全的事
上一节我们做的 **无签名 JWT（alg:none）** 有致命漏洞：
- 客户端拿到明文令牌
- **可以随便改里面的内容**（比如把过期时间改到10年后、把普通用户改成管理员）
- 资源服务器没有任何办法验证令牌是否被篡改
- 攻击者甚至能**凭空伪造一个令牌**，直接骗过资源服务器

→ **这在生产环境完全不能用！**

---

# 二、解决方案：JOSE 套件
**JOSE = JSON Object Signing and Encryption**
一套给 JSON 数据做**签名 + 加密**的标准集合。

它包含三大核心技术：
1. **JWS（JSON Web Signature）**
   给 JWT 签名，**防篡改、防伪造**（最常用）
2. **JWE（JSON Web Encryption）**
   给 JWT 加密，**防止内容被偷看**
3. **JWK（JSON Web Key）**
   用 JSON 格式存储密钥（公钥/私钥）

我们之前的无签名 JWT = **JWS 的一个特殊无签名版本**。

---

# 三、本节要讲两种最常用的签名方式
1. **对称签名：HMAC（如 HS256）**
   - 一个密钥同时用于签名 + 验证
   - 简单、速度快
   - 适合授权服务器和资源服务器是同一方

2. **非对称签名：RSA（如 RS256）**
   - 私钥签名，公钥验证
   - 公钥可以公开分发
   - 适合分布式、多第三方资源服务器

3. 支持**多把密钥同时存在**（平滑过渡）

- 让授权服务器暴露一个 `/jwks` 接口
- 返回 JWK Set 格式的**公钥**
- 资源服务器启动时去下载公钥
- 注意：**绝对不能泄露私钥（d 字段）！**

# 最终极简总结（你记这个就够）
### 问题：
密钥更新 → 要复制到所有服务，太麻烦。

### 解决方案：
**授权服务器发布 JWK Set（公钥集合）到公开 URL**
资源服务器自动下载、自动更新。

### 特点：
- 自动、安全、可扩展
- 生产环境标准做法
- 只暴露公钥，绝不暴露私钥

# 整段内容 **一句话总结**
**资源服务器用授权服务器的公钥验证 JWT 签名，
签名通过后，再做 iss / aud / exp 等校验，
全部通过才算合法令牌。**

---

# 逐段超简单解释

## 1. 用公钥验证签名
```js
var publicKey = jose.KEYUTIL.getKey(rsaKey);
```
把**公钥**（rsaKey，没有私钥 d 字段）加载成库能识别的格式。

---

## 2. 核心：验证 JWT 签名
```js
if (jose.jws.JWS.verify(inToken, publicKey, [header.alg])) {
   // 签名验证成功！
}
```

### 这行代码在做什么？
- 检查这个 JWT **是不是授权服务器用私钥签的**
- 检查**头部/载荷有没有被篡改过**
- 检查**算法是否匹配（RS256）**

→ **签名通过 = 令牌没被改、是合法签发的**

---

## 3. 签名通过后，还要做什么？
**必须继续做之前所有的合法性检查！**
都要写在 `if` 里面：

- 检查 `iss` 签发者是否正确
- 检查 `aud` 是不是发给自己的
- 检查 `iat` 签发时间是否合理
- 检查 `exp` 是否过期

这些校验**一个都不能少**！

---

## 4. 校验通过后
把解析出来的 payload 交给业务代码，判断：
- 用户是谁（sub）
- 有没有权限（scope）
- 哪个客户端（client_id）

---

## 5. 书里的小练习
让你自己给 JWT 添加自定义字段，比如：
- `scope` 权限
- `client_id` 客户端ID
- `user_role` 用户角色

然后资源服务器读取这些值做权限控制。

# 最关键的 3 个知识点（必背）
1. **RS256 验签只用公钥，不需要私钥**
2. **先验签名，再验内容**
3. **签名通过只代表没被篡改，还要检查过期、签发者等**

---

# 最终极简总结（记这个就够）
**公钥验签名 → 防篡改
验完签名再查 iss / aud / exp → 防滥用
全部通过 → 令牌合法**

# 11.4 令牌内省（Token Introspection）超通俗解释
我给你**一句话抓住核心 + 简单白话 + 必背知识点**，保证立刻懂👇

---

# 整段的 **核心一句话**
**JWT 自己带信息（自包含）有两大缺点：太大、没法吊销。
所以诞生了 Token Introspection —— 让资源服务器去授权服务器“在线查令牌状态”。**

---

# 逐句白话解释
## 1. 自包含令牌（JWT）的缺点
> Packing the information about the token into the token itself does have its drawbacks.

把所有信息都塞进令牌里（比如 JWT），是有缺点的。

## 2. 缺点①：令牌会变得很大
> The tokens themselves can grow to be quite large…

JWT 要放：
- iss / sub / aud / exp / iat
- scope / client_id
- 签名信息

→ **令牌变得很长**，每次请求都要带着，浪费流量。

## 3. 缺点②：没法吊销（超级重要！）
> …difficult to revoke active tokens once they’ve been created and sent into the wild.

JWT 一旦发出去：
- **只要没过期，就能一直用**
- 即使用户注销、退出、改密码
- 授权服务器**管不了已经发出去的 JWT**
- 想让令牌立刻失效？**做不到！**

这就是 **自包含令牌最大的安全缺陷**。

---

# 4. 解决方案：Token Introspection（令牌内省）
## 它是什么？
**资源服务器不解析令牌，
而是把令牌发给授权服务器问：
“这个令牌有效吗？信息是什么？”**

流程：
1. 客户端带令牌访问资源服务器
2. 资源服务器**不自己解析**
3. 资源服务器调用授权服务器的 **introspection 接口**
4. 授权服务器返回：有效/无效、用户信息、权限、过期时间…
5. 资源服务器根据结果决定是否放行

---

# 5. 这样解决了什么？
## ✅ 解决“无法吊销”
授权服务器有令牌状态，想让它失效就失效。

## ✅ 解决“令牌太大”
令牌可以是**短随机字符串**，不用带任何信息。

## ✅ 更安全
令牌内容不会泄露在客户端/传输中。

---

# 6. 极简总结（必背）
## JWT（自包含令牌）
- 优点：不用查库、快、分布式友好
- 缺点：**令牌大、无法吊销**

## Token Introspection（令牌内省）
- 优点：**可吊销、令牌小、更安全**
- 缺点：需要调用授权服务器，慢一点

---

# 11.5 令牌吊销（Token Revocation）超通俗完整解释
（完全贴合原文、不漏知识点、大白话）

## 整段**一句话核心**
JWT 等自包含令牌**不能主动失效，只能等过期**；
OAuth 专门出了**令牌吊销标准**，允许客户端主动告诉授权服务器：
**这个令牌立刻作废，不要再让它生效了。**

---

## 逐段拆解解释
### 1. OAuth 令牌原本的生命周期
令牌正常流程：
授权服务器创建 → 客户端使用 → 资源服务器验证
令牌可以**自然过期**，也可以由**用户/管理员手动在授权服务器吊销**。

刷新令牌（Refresh Token）可以用来换新的访问令牌。
之前我们学了两种验证方式：
- JWT 自验证
- 令牌内省 Introspection

---

### 2. 提出问题（原文重点）
有时候**客户端自己知道再也不需要令牌了**：
- 用户退出登录
- APP 卸载
- 用户撤销授权
- 客户端发现被盗、异常风险

那客户端**必须等到令牌自然过期吗？**
之前的 OAuth 核心协议**没有任何机制**，能让客户端主动通知授权服务器作废令牌。

---

### 3. 解决方案：OAuth Token Revocation 令牌吊销规范
这个标准就是用来补全这个缺口的：
**允许客户端主动、提前、主动管理令牌生命周期**。

客户端可以主动调用授权服务器的吊销接口，通知：
> 这个 access_token / refresh_token 立刻失效！

---

### 4. 哪些场景会主动吊销？（原文举例）
1. 原生 APP 在用户设备上**被卸载**
2. 用户在客户端页面**主动撤销授权**
3. 客户端检测到异常、被盗风险，想要**止损**，防止权限继续被滥用

不管什么原因，吊销协议都允许客户端通知授权服务器：
**已签发的令牌，禁止再使用。**

---

# 最重要的知识点（考试/面试必背）
1. **自然过期 = 被动失效**
2. **令牌吊销 = 主动提前失效**
3. JWT 自包含令牌**本身无法吊销**，必须配合吊销接口 + 内省查询才能实现吊销
4. 吊销是客户端主动发起，不是授权服务器单方面操作
5. 既可以吊销 Access Token，也可以吊销 Refresh Token
6. 吊销后：再用这个令牌访问，直接拒绝

---

# 极简背诵总结
OAuth 令牌默认只能**等到过期**；
令牌吊销协议允许**客户端主动作废令牌**；
用于退出登录、卸载APP、撤销授权、安全止损；
是完整令牌生命周期管理的最后一块拼图。
