# 11.1 OAuth 令牌是什么？（精简翻译+要点）
## 中文翻译
令牌是所有 OAuth 交互的**核心**。
客户端从授权服务器获取令牌，然后提交给受保护资源。
授权服务器创建令牌并发放给客户端，同时在令牌中绑定资源所有者的委托授权与客户端权限。
受保护资源从客户端接收令牌并进行验证，将令牌附带的权限与客户端的请求进行匹配校验。

令牌代表**授权委托的结果**：它是资源所有者、客户端、授权服务器、受保护资源、权限范围（scope）以及所有授权相关信息的集合。
当客户端需要在不打扰用户的情况下刷新访问令牌时，会使用另一种令牌：**刷新令牌**。

OAuth 令牌是整个 OAuth 生态的核心机制，可以说**没有令牌就没有 OAuth**。甚至 OAuth 的非官方 Logo 都设计成了实体公交令牌的样子。

尽管如此强调令牌，但 OAuth **完全没有规定令牌内部应该包含什么内容**。
正如我们在第 2、3 章讨论的，OAuth 客户端**不需要知道令牌本身的任何信息**。
客户端只需要知道：如何从授权服务器拿令牌、如何在资源服务器使用令牌。
但**授权服务器和资源服务器必须理解令牌**：授权服务器要知道如何生成令牌，资源服务器要知道如何识别和验证令牌。

为什么 OAuth 核心协议会省略如此基础的内容？
因为不规定令牌格式，OAuth 才能灵活适用于各种部署环境、不同风险等级与不同需求。
OAuth 令牌可以过期、可被吊销、永久有效，或根据场景组合特性；
可以代表特定用户、系统内所有用户，甚至不代表任何用户；
可以有内部结构、可以是随机字符串、可以加密保护，也可以是以上方式的组合。
这种灵活性与模块化，让 OAuth 比 WS-*、SAML、Kerberos 等**硬性规定令牌格式**的安全协议更容易适配各类系统。

在实际系统中，有几种常见的令牌创建与验证方式，各有优缺点与适用场景。
在本书第 3、4、5 章的练习里，我们创建的令牌是**随机字母数字串**，类似这样：
`s9nR4qv7qVadTUssVD5DqA7oRLJ2xonn`

授权服务器生成令牌后，会把令牌信息存在共享数据库。
受保护资源收到令牌时，在同一个数据库里查询该令牌的权限。
这类令牌**内部不携带任何信息**，只作为数据库查询的索引键。
这是完全合法且常用的实现方式，优点是**令牌体积小、熵值高（安全性强）**。

但在很多场景下，授权服务器与受保护资源**共享数据库并不现实**（尤其是一台授权服务器保护多个下游资源时）。
那我们该用什么方案？
本章会介绍另外两种通用方案：**结构化令牌** 和 **令牌内省（Introspection）**。

# 11.2 结构化令牌：JWT（JSON Web Token）核心知识点总结
## 一、JWT 要解决什么问题？
- 之前的随机令牌需要**授权服务器和资源服务器共享数据库**
- JWT 让令牌**自己携带所有必要信息**，不用查库
- 授权服务器把信息“打包”进令牌，客户端不用理解，资源服务器可直接解析

## 二、JWT 结构
- 格式：**Header.Payload.Signature**（三部分用 `.` 分隔）
- 每一部分都是 **Base64URL 编码**
- 可以安全放在 URL、Header、表单参数里

### 1）Header（头部）
说明令牌类型、算法
示例：
```json
{ "typ": "JWT", "alg": "none" }
```

### 2）Payload（载荷）
存放要传递的信息，称为 **Claims（声明）**

## 三、JWT 标准声明（必须记住）
- **iss**：签发者（通常是授权服务器 URL）
- **sub**：主题（用户ID）
- **aud**：受众（哪些资源服务器可以用）
- **exp**：过期时间（Unix 秒）
- **nbf**：在此时间之前不可用
- **iat**：签发时间
- **jti**：令牌唯一ID（防重放）

## 四、为什么用 Base64URL？
- 能**安全放在 HTTP 各种位置**（header、url、form）
- 不需要额外转义
- 避免中间件意外修改 JSON

## 五、JWT 特点（OAuth 视角）
- 对客户端**完全不透明**，客户端代码不用改
- 资源服务器**不需要共享数据库**
- 可自包含、自验证、可过期
- 适合分布式、多资源服务场景

## 六、本章实现逻辑
1. 授权服务器生成 JWT（Header + Payload → Base64URL 拼接）
2. 设置 iss / sub / aud / exp / iat / jti
3. 资源服务器收到 JWT 后：
   - 按 `.` 拆分
   - 解码 Payload
   - 校验 iss、aud、iat、exp
4. 校验通过就认为令牌合法

## 七、本章重要结论
- JWT = **自带信息的安全令牌**
- 不需要共享数据库，适合分布式系统
- 客户端完全无感知，不用修改代码
- 资源服务器直接解析、校验即可

# 背后的关键知识点（必须掌握）
1. **JWT 看起来是乱码，但不是随机串**
   它只是被 Base64URL 编码了，解码后是有结构的 JSON。

2. **JWT 用 . 分割成三部分**
   - 第一部分：Header（头部）
   - 第二部分：Payload（载荷）
   - 第三部分：Signature（签名，11.3 讲）

3. **分割后可以分别处理**
   资源服务器拿到 JWT 后，直接按 `.` 拆分 → 解码 → 读取信息，**不用查数据库**。

4. **示例里最后一个点是空的**
   因为这是**未签名的 JWT**，签名部分留空了，所以格式是：
   ```
   头部.载荷.
   ```

# 一句话核心答案
**JWT 使用 Base64URL 编码，是为了让令牌能安全地放在任何地方（URL、Header、表单、数据库），不需要额外转义，也不会被系统改错。**

---

# 完整通俗解释（逐段讲明白）
## 为什么不能直接用 JSON？
JSON 长这样：
```json
{ "sub": "123", "name": "Alice" }
```
里面有 **{ } " : , 空格** 等符号。

这些符号**不能直接放在 HTTP 里**，比如：
- URL 里不能直接用 `{ } "`
- 请求头里不能乱加特殊字符
- 表单参数会把特殊字符转义掉

如果你直接把 JSON 塞进去，系统会自动把它变成：
```
%7B%22sub%22%3A%22123%22%2C%22name%22%3A%22Alice%22%7D
```
又乱又容易出错，还会出现**双重转义**（越转越错）。

---

## Base64URL 解决了什么？
它把 JSON 转换成**只包含以下安全字符**：
- 字母 A-Z a-z
- 数字 0-9
- `-` `_`

这些字符**在任何地方都合法**：
- URL 查询参数 ✅
- HTTP 请求头 ✅
- 表单字段 ✅
- 数据库存储 ✅
- 各种编程语言字符串 ✅

→ **放哪里都不会被转义、不会出错、不会被改错。**

---

## 额外好处：不会被中间件改错
很多服务器、网关、代理会自动“修理”JSON，比如：
- 加空格
- 改排序
- 转义字符
- 重新格式化

一旦被改，签名就失效，JWT 就废了。

但 Base64URL 是**一串普通字符串**，网关/代理不会去动它，保证 JWT 从头到尾**原汁原味**。

---

# 最终超级浓缩版（必背）
## 为什么 JWT 要用 Base64？
1. **让令牌能安全放在任何地方**（URL、Header、表单、数据库）
2. **避免特殊字符被转义**（不会出现 %7B %22 这种乱码）
3. **防止双重编码**（不会越转越错）
4. **避免被中间件改错**（保持原样，保证签名有效）

# 第一段解释（Header 部分）
### 原文：
This header is always a JSON object and it’s used to describe information about the rest of the token.
### 翻译：
**JWT 的 Header 固定是一个 JSON 对象**，作用是**描述这个令牌本身的信息**。

---

### 原文：
The typ header tells the application processing the rest of the token what to expect in the second section, the payload.
### 翻译：
`typ` 这个字段告诉解析器：**第二部分（Payload）是什么类型的数据**。

---

### 原文：
In our example, we’re told that it’s a JWT.
### 翻译：
我们例子里写的是：
`"typ": "JWT"`
意思就是：**这是一个 JWT 令牌**。

---

### 原文：
This also includes the alg header with the special value none to indicate that this is an unsigned token.
### 翻译：
Header 里还有一个 **alg** 字段，表示**用什么签名算法**。
我们例子里写的是：
`"alg": "none"`
意思就是：**这个 JWT 没有签名（未加密、未签名）**。

# 第二段解释（Payload 部分）
### 原文：
The second section is the payload of the token itself, and it’s serialized in the same way as the header: Base64URL-encoded JSON.
### 翻译：
JWT 的**第二部分叫 Payload（载荷）**，
和 Header 一样，也是 **JSON → Base64URL 编码** 而来。

---

### 原文：
Because this is a JWT, the payload can be any JSON object, and in our previous example it’s a simple set of user data.
### 翻译：
Payload **可以放任意 JSON 数据**。
我们例子里放的是用户信息：
```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "admin": true
}
```

---

# 最后一句关键知识点
### 原文：
Specifically, it’s Base64 encoding with a URL-safe alphabet and no padding characters
### 翻译：
JWT 使用的不是普通 Base64，而是 **URL 安全版 Base64（叫 Base64URL）**
特点：
- 没有 `+` `/` 这些 URL 不安全字符
- 没有末尾填充的 `=` 符号
→ **可以直接放在网址里不乱码**

# 最精简总结（你记这个就够）
1. **JWT 第一部分 = Header**
   - 描述令牌类型、签名算法
   - 示例：`{"typ":"JWT","alg":"none"}`

2. **JWT 第二部分 = Payload**
   - 放真正要传的数据（用户信息、权限、过期时间等）
   - 也是 JSON → Base64URL

3. **编码方式 = Base64URL**
   - 网址安全
   - 无特殊字符
   - 无填充符

# 11.2.2 JWT claims（JWT 标准声明）
## 一句话概括
**Claims = JWT 载荷里存放的“标准字段”**，用来放令牌的身份、时间、权限等信息。
JWT 本身可以存任何 JSON，但这些标准字段让所有系统都能统一识别。

---

# 7 个最重要的标准 Claim（必须记住）
## 1. **iss**
Issuer → **签发者**
谁发的令牌？（一般是授权服务器 URL）

## 2. **sub**
Subject → **主题**
令牌是属于谁的？（用户唯一 ID）

## 3. **aud**
Audience → **受众**
这个令牌是给谁用的？（资源服务器地址）
可以是一个字符串，也可以是数组（多个接收方）。

## 4. **exp**
Expiration → **过期时间**
Unix 时间戳（秒），到点就失效。

## 5. **nbf**
Not Before → **在此时间之前不可用**
令牌还没生效，没到时间不能用。

## 6. **iat**
Issued At → **签发时间**
令牌什么时候生成的。

## 7. **jti**
JWT ID → 令牌唯一 ID
防止重放攻击、防止重复、防止伪造。

---

# 超简版记忆口诀
**iss 谁发的
sub 属于谁
aud 给谁用
exp 何时过期
nbf 何时生效
iat 何时签发
jti 唯一标识**

---

# 关键知识点（书里的重点）
1. **所有 claim 都是可选的**，但业务系统可以强制要求必须带某些字段
2. 除了标准字段，**你可以随便加自定义字段**，比如 name、email、admin、scope
3. 标准字段是为了**跨系统兼容**，所有服务都认识
4. 时间都是 **Unix 时间戳（秒）**，不是毫秒
5. **aud 可以是数组**，表示令牌可以给多个服务用
6. **jti 是随机字符串**，防重放、防碰撞

# 本节核心一句话
把之前**随机字符串令牌**改成 **JWT 结构化令牌**，让资源服务器**不用查数据库**，直接解析令牌就能验证权限。

# 一、整体流程（超级简单）
1. **授权服务器**：生成 JWT（Header + Payload → Base64URL 拼接）
2. **发给客户端**：客户端完全不用改，照样传输
3. **资源服务器**：拆分 JWT → 解码 → 校验字段 → 通过就放行

---

# 二、授权服务器做了什么？
## 1. 删掉原来的随机令牌
```js
var access_token = randomstring.generate();
```

## 2. 生成 JWT 固定头部
```js
var header = { 'typ': 'JWT', 'alg': 'none' };
```
- `typ: JWT`：说明这是 JWT
- `alg: none`：**没有签名**（教学用，真实环境必须签名）

## 3. 生成 JWT 载荷（最重要）
```js
var payload = {
  iss: 'http://localhost:9001/',    // 谁签发
  sub: code.user.sub,              // 属于哪个用户
  aud: 'http://localhost:9002/',   // 给哪个资源服务器用
  iat: Math.floor(Date.now()/1000),// 签发时间
  exp: Math.floor(Date.now()/1000)+5*60, // 5分钟过期
  jti: randomstring.generate(8)    // 唯一ID
};
```

## 4. 拼接成 JWT
```js
var access_token =
  base64url编码(header) + "." +
  base64url编码(payload) + ".";
```
格式：
```
头部.载荷.（签名为空）
```

---

# 三、资源服务器做了什么？
## 1. 拆分 JWT
```js
var tokenParts = inToken.split('.');
```

## 2. 解码载荷
```js
var payload = JSON.parse(base64url.decode(tokenParts[1]));
```

## 3. 逐项校验（必须全过）
```js
1. iss === 授权服务器地址      ✅
2. aud === 自己的地址          ✅
3. iat <= 当前时间             ✅
4. exp >= 当前时间             ✅
```

## 4. 全部通过 → 令牌合法
资源服务器直接用 payload 里的信息做权限判断，**不用查库**。

---

# 四、本节最重要的 5 个知识点（考试/面试必背）
1. **客户端完全不用改代码**
   JWT 对客户端是透明的，客户端不需要理解令牌内容。

2. **资源服务器不再需要共享数据库**
   所有信息都在令牌里，分布式系统必备。

3. **时间必须用秒（Unix 时间戳）**
   JS 默认是毫秒，必须 ÷1000。

4. **本节 JWT 没有签名（alg:none）**
   仅教学演示，生产环境必须加签名。

5. **校验 4 个字段就足够安全**
   iss（签发者）、aud（受众）、iat（签发时间）、exp（过期时间）。

- 授权服务器打包 JWT → 发给客户端
- 客户端无感传输
- 资源服务器拆包 → 校验4项 → 放行
- **无数据库、无共享存储、分布式友好**
