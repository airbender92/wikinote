# 10.1 什么是 Bearer Token（持有者令牌）？
## 翻译
OAuth 2.0 抛弃了 OAuth 1.0 复杂的签名机制，**完全依靠 TLS（HTTPS）保证安全**。
Bearer Token 的定义非常简单：
**谁拿到这个令牌，谁就可以用它。**

就像：
- 公交车票
- 游乐园门票
- 电影票

**不需要验证身份，只要有票就能进。**

## 技术上看 Bearer Token
- 是一串**明文字符串**
- **不需要签名、不需要密钥**
- 安全性**完全依赖 HTTPS**
- 和 Cookie 很像，但**不受同源策略保护**

## OAuth 1.0 为什么被淘汰？
因为太复杂：
- 需要计算签名
- 参数顺序、编码、URL 格式必须完全一致
- 大小写、编码不同都会导致验签失败
- 开发难度极高、极易出bug

所以 OAuth 2.0 直接简化为：
## **Bearer Token + HTTPS = 全部安全**

---

# 10.2 Bearer Token 的风险
Bearer Token 一旦被盗，攻击者**可以直接使用**，因为它不校验持有者身份。

主要威胁：
1. **Token 伪造**：攻击者自己造假令牌
2. **Token 重放**：盗走旧令牌反复使用
3. **Token 重定向**：把给A服务器的token用在B服务器
4. **Token 信息泄露**：token里包含敏感信息

---

# 10.3 如何保护 Bearer Token？
## 核心原则：**永远必须用 HTTPS**
没有 HTTPS，OAuth 2.0 没有任何安全性。

---

## 10.3.1 客户端怎么保护？
1. **申请最小权限 scope**
   不需要的权限绝对不申请
2. **Token 只存在内存，不要持久化**
   防止被盗
3. **用完就删**

---

## 10.3.2 授权服务器怎么保护？
1. **Token 必须是高随机、高熵值（至少128位）**
2. **存储 Token 哈希，不存原文**
   即使数据库泄露，攻击者也无法使用
3. **Access Token 有效期必须短**（几分钟~几小时）
4. 使用 **Refresh Token** 续期
5. 完整日志审计

---

## 10.3.3 资源服务器怎么保护？
1. 严格校验 Token
2. 不日志记录 Token
3. 最小权限校验
4. 限流防暴力破解

---

# 10.4 Authorization Code（授权码）
授权码的好处：
- 不通过浏览器传递 Access Token
- 更安全
- 一次性使用
- 必须配合 client_secret / PKCE 才能兑换

---

# 10.4.1 PKCE（最重要！）
## 公共客户端（前端、移动端）无法隐藏 client_secret
所以容易被劫持授权码。

**PKCE = 防止授权码劫持的终极方案**

## PKCE 原理（超级简单）
1. 客户端生成一个随机字符串：**code_verifier**
2. 客户端计算它的 SHA256 哈希：**code_challenge**
3. 授权请求时，只发送 **code_challenge**
4. 授权服务器保存它
5. 兑换 Token 时，客户端发送 **code_verifier**
6. 服务器重新计算哈希，比对是否一致

## 结果：
**即使劫持了 code，没有 code_verifier 也无法兑换 Token！**

这就是 **公共客户端（前端/APP）的必选安全机制**。

# 本章终极总结（背会就精通 OAuth 令牌安全）
1. **Bearer Token = 谁拿谁能用，完全依赖 HTTPS**
2. **OAuth 2.0 放弃复杂签名，简化为 TLS + 令牌**
3. **Access Token 必须短有效期**
4. **Refresh Token 长有效期，只用于授权服务器**
5. **Token 存储必须加密/哈希，不存明文**
6. **PKCE = 公共客户端防授权码劫持的唯一方案**
7. **没有 HTTPS，OAuth 2.0 等于完全不安全**

# 一、核心内容：Bearer Token 承载令牌（全书第10章）
## 1. 什么是 Bearer Token？
- **谁拿到令牌，谁就能用**，不需要额外密钥、签名、身份验证
- 像**公交票、公园门票**：有票就能进，不查持票人是谁
- 基于 **TLS/HTTPS** 保证安全，OAuth2 不再用 OAuth1 复杂的签名机制
- 与 Cookie 相似：明文字符串、无内置签名、靠 TLS 保护

## 2. Bearer Token 的四大风险
1. **令牌伪造**：攻击者造出假令牌获取权限
2. **令牌重放**：用过期/旧令牌再次访问
3. **令牌跨站滥用**：把给A服务的令牌用在B服务
4. **令牌泄露**：日志、URL、传输中被窃取

## 3. 如何保护 Bearer Token（最重要）
### 客户端要做
- 只申请**最小权限 scope**
- 令牌存在**临时内存**，不要持久化
- 绝对不要把令牌放在 URL 里

### 授权服务器要做
- 存储**令牌哈希**，不存原文
- **访问令牌有效期尽量短**
- 用刷新令牌延长权限，不延长访问令牌
- 完整审计日志，不记录令牌原文

### 资源服务器要做
- 严格校验 scope 和权限
- 从日志中清除令牌
- 做限流、防暴力测试

# 二、核心安全机制：PKCE（最重要！）
## 是什么？
- 全称：**Proof Key for Code Exchange**
- 读音：pixie
- 作用：**防止授权码劫持**，专门保护**公共客户端**（前端、原生APP、无密钥客户端）

## 原理（超级简单版）
1. 客户端生成一个随机密钥：`code_verifier`
2. 客户端算出它的哈希值：`code_challenge`
3. 授权时只发给服务器 **challenge**
4. 拿授权码换令牌时，再发给服务器 **verifier**
5. 服务器重新算哈希，比对一致才发放令牌

→ **即使劫持了授权码，没有 verifier 也换不到令牌**

# 三、结合你给的 3 个真实漏洞案例
这些案例直接印证了本章说的 **scope 校验缺失、权限过大、客户端不隔离** 问题：

## 1. GitHub 漏洞（document 1）
- 问题：OAuth 令牌**没有任何 scope**，却能读取用户的私密 Gist
- 根本原因：**资源服务器未校验 scope**
- 修复：接口必须检查令牌是否有 `gist` 权限

## 2. Facebook 私人照片泄露（document 2）
- 问题：任何拥有 `user_photos` 权限的 APP，都能读取用户**私人同步相册**
- 根本原因：接口只校验用户，不校验**调用方APP**
- 修复：把官方APP加入白名单，第三方禁止访问

## 3. RFC 7636（document 3）
- 就是 **PKCE 官方标准**
- 用于防止授权码拦截攻击
