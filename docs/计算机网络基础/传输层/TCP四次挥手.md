下面用**标准、清晰、考试友好**的方式，把 **TCP 四次挥手（断开连接）** 完整讲清楚。

---

## 一、什么是 TCP 四次挥手？
TCP 连接是**全双工**的，断开时需要：
- 客户端 → 服务器 关闭
- 服务器 → 客户端 关闭

因此需要 **四次报文交互**，才能完全断开连接。

---

## 二、四次挥手过程（核心）

假设：
- 客户端主动关闭连接
- 服务器被动关闭

### 第一次挥手（客户端 → 服务器）
- **报文**：FIN = 1
- **作用**：
  - 客户端请求关闭连接
  - 表示“我不再发送数据”

📌 客户端状态：
- 进入 **FIN-WAIT-1**

---

### 第二次挥手（服务器 → 客户端）
- **报文**：ACK = 1
- **作用**：
  - 服务器确认收到 FIN
  - 客户端 → 服务器方向关闭完成

📌 服务器状态：
- 进入 **CLOSE-WAIT**

📌 客户端状态：
- 进入 **FIN-WAIT-2**

⚠️ 此时服务器仍可向客户端发送数据

---

### 第三次挥手（服务器 → 客户端）
- **报文**：FIN = 1
- **作用**：
  - 服务器也准备关闭连接
  - 表示“我不再发送数据”

📌 服务器状态：
- 进入 **LAST-ACK**

---

### 第四次挥手（客户端 → 服务器）
- **报文**：ACK = 1
- **作用**：
  - 客户端确认服务器 FIN
  - 连接完全关闭

📌 客户端状态：
- 进入 **TIME-WAIT**
- 等待 2MSL 后进入 **CLOSED**

📌 服务器状态：
- 进入 **CLOSED**

✅ 至此，**TCP 连接彻底断开**

---

## 三、四次挥手示意图（文字版）

```
客户端                      服务器
  | ---- FIN ------------> |
  | <---- ACK ------------ |
  |                        |
  | <---- FIN ------------ |
  | ---- ACK ------------> |
```

---

## 四、为什么是四次挥手？

### 关键原因：
- TCP 是 **全双工通信**
- 每一方向的关闭需要：
  - 一方发送 FIN
  - 另一方发送 ACK

📌 ACK 和 FIN 不能合并：
- 服务器可能还有数据未发送完

---

## 五、TIME-WAIT 状态的作用（常考）

客户端在 TIME-WAIT 等待 **2MSL** 的原因：
1. 确保最后一个 ACK 能被服务器收到
2. 防止旧连接的数据包影响新连接

---

## 六、一句话总结（考试 / 背诵版）

> TCP 通过四次挥手断开连接，第一次客户端发送 FIN，第二次服务器发送 ACK，第三次服务器发送 FIN，第四次客户端发送 ACK，最终双方进入 CLOSED 状态，完成连接释放。
