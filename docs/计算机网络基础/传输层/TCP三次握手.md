下面用**清晰、标准、考试友好**的方式，把 **TCP 三次握手（建立连接）** 讲清楚。

---

## 一、什么是 TCP 三次握手？
TCP 是**面向连接**的协议，通信前必须通过**三次报文交互**，建立一条可靠的 TCP 连接。

> 目的：  
> ✅ 确认双方的发送和接收能力  
> ✅ 同步双方的初始序号  
> ✅ 防止已失效的连接请求造成错误

---

## 二、三次握手过程（核心）

假设：
- 客户端：主动发起连接（Client）
- 服务器：被动监听（Server）

### 第一次握手（客户端 → 服务器）
- **报文**：SYN = 1
- **作用**：
  - 客户端请求建立连接
  - 发送初始序号 seq = x

📌 状态变化：
- 客户端进入：**SYN-SENT**

---

### 第二次握手（服务器 → 客户端）
- **报文**：SYN = 1，ACK = 1
- **作用**：
  - 服务器同意建立连接
  - 确认客户端的序号（ack = x + 1）
  - 发送自己的初始序号 seq = y

📌 状态变化：
- 服务器进入：**SYN-RCVD**

---

### 第三次握手（客户端 → 服务器）
- **报文**：ACK = 1
- **作用**：
  - 客户端确认服务器的连接
  - ack = y + 1

📌 状态变化：
- 客户端进入：**ESTABLISHED**
- 服务器进入：**ESTABLISHED**

✅ 至此，**TCP 连接建立完成**

---

## 三、三次握手示意图（文字版）

```
客户端                      服务器
  | ---- SYN(seq=x) -------> |
  |                          |
  | <-- SYN(seq=y), ACK -----|
  |                          |
  | ---- ACK(ack=y+1) -----> |
```

---

## 四、为什么必须是三次？（常考）

### ❓ 两次握手行不行？
❌ 不行，原因：
- 可能接收到 **已失效的旧连接请求**
- 服务器会误以为连接已建立
- 造成资源浪费或数据错误

### ✅ 三次握手解决了：
- 确认双方的发送和接收能力
- 同步初始序号
- 防止历史连接请求造成错误连接

---

## 五、一句话总结（考试 / 背诵版）

> TCP 通过三次握手建立连接，第一次客户端发送 SYN，第二次服务器回应 SYN+ACK，第三次客户端发送 ACK，双方进入 ESTABLISHED 状态，完成连接建立。
