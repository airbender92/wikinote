流量控制：停止等待协议与滑动窗口协议（GBN、SR）

流量控制的核心是协调发送方与接收方的速率，避免接收方缓冲区溢出。其实现依赖两类经典协议：停止等待协议（最简单的流量控制，滑动窗口的特例）和滑动窗口协议（连续发送多帧，提高效率），后者又分为Go-Back-N（GBN，后退N帧） 和Selective Repeat（SR，选择重传）。  

一、停止等待协议（Stop-and-Wait Protocol）

定义

发送方每发送1帧后停止，等待接收方返回确认（ACK），收到ACK后再发送下一帧。本质是滑动窗口大小为1的特殊情况（发送窗口=1，接收窗口=1）。  

核心原理

1. 发送方行为：  
   • 发送1帧（编号i），启动定时器；  

   • 若收到ACKi（确认帧i正确接收），关闭定时器，发送下一帧i+1；  

   • 若定时器超时（未收到ACKi），重发帧i（应对丢包或ACK丢失）。  

2. 接收方行为：  
   • 收到正确帧i（无差错、序号正确），发ACKi，交付网络层；  

   • 若帧出错（CRC校验失败），丢弃该帧，不发ACK（发送方超时后重传）。  

3. 序号与确认：  
   • 用1位序号（0/1）区分帧（避免新旧帧混淆），如帧0→ACK0，帧1→ACK1，循环使用。  

示例流程

  
发送方：发帧0 → 等ACK0 → 收ACK0 → 发帧1 → 等ACK1 → ...  
接收方：收帧0（正确）→ 发ACK0 → 收帧1（正确）→ 发ACK1 → ...  
（若帧0丢失：发送方超时重发帧0；若ACK0丢失：发送方超时重发帧0）  
  

优缺点

• 优点：实现简单（无需缓存多帧），逻辑清晰；  

• 缺点：效率低（信道利用率≈ \frac{T_{trans}}{T_{trans} + 2T_{prop}}，T_{trans}为帧传输时间，T_{prop}为传播时延）。当RTT（往返时延）远大于T_{trans}时（如卫星通信），大部分时间在等待ACK，带宽浪费严重。  

典型应用

早期低速串行通信（如RS-232）、对实时性要求低的简单链路（如传感器数据采集）。  

二、滑动窗口协议（Sliding Window Protocol）

定义

发送方维护一个发送窗口（允许连续发送的帧序号范围），接收方维护一个接收窗口（允许接收的帧序号范围）。窗口随确认动态“滑动”，实现连续发送多帧，大幅提高信道利用率（窗口大小N越大，效率越高）。  

核心概念

• 发送窗口：[send\_base, send\_next-1]，其中send\_base是首个未确认帧序号，send\_next是下一个待发帧序号。窗口内帧可连续发送，无需等待ACK。  

• 接收窗口：[recv\_base, recv\_base+W_r-1]，其中recv\_base是期望接收的帧序号，W_r是接收窗口大小。窗口内帧允许接收（按顺序或乱序，取决于协议）。  

• 滑动机制：收到ACK后，send\_base右移（缩小发送窗口）；正确接收帧后，recv\_base右移（缩小接收窗口）。  

三、Go-Back-N（GBN，后退N帧协议）

定义

累积确认+接收窗口=1的滑动窗口协议：发送方连续发N帧（发送窗口W_t=N），接收方仅按顺序接收（接收窗口W_r=1），若中间某帧出错，丢弃后续所有帧，发最后一次正确帧的ACK（累积确认），发送方超时后重传所有未确认帧（从出错帧开始）。  

核心原理

1. 发送方：  
   • 发送窗口[send\_base, send\_base+W_t-1]，可连续发窗口内所有帧；  

   • 收到ACKk（累积确认，表示k及之前帧均已接收），则send\_base=k+1（窗口滑动）；  

   • 若某帧超时（未收到ACK），重传所有未确认帧（从send\_base开始）。  

2. 接收方：  
   • 接收窗口W_r=1（仅接收recv\_base序号的帧）；  

   • 若收到帧i=recv\_base（正确），发ACKi，recv\_base=i+1；  

   • 若收到帧i≠recv\_base（出错或乱序），丢弃该帧，重发ACK(recv\_base-1)（最后一次正确帧的ACK）。  

示例（窗口大小N=5，帧2出错）

  
发送方发送帧0~4 → 接收方收帧0（正确）→ 发ACK0，收帧1（正确）→ 发ACK1，收帧2（出错）→ 丢弃帧2、3、4，发ACK1（累积确认帧1）；  
发送方超时未收ACK2→ 重传帧2、3、4（从send_base=2开始）。  
  

优缺点

• 优点：实现简单（接收方无需缓存乱序帧），累积确认减少ACK数量；  

• 缺点：效率低（出错后重传多帧，尤其窗口大时），不适合高误码率场景。  

典型应用

误码率低的局域网（如早期以太网半双工模式）。  

四、Selective Repeat（SR，选择重传协议）

定义

独立确认+接收窗口>1的滑动窗口协议：发送方连续发N帧（发送窗口W_t=N），接收方可乱序接收并缓存正确帧（接收窗口W_r=N，通常W_t=W_r=N），发对每个正确帧的独立ACK，发送方仅重传出错帧。  

核心原理

1. 发送方：  
   • 发送窗口[send\_base, send\_base+W_t-1]，连续发窗口内帧；  

   • 收到ACKi（独立确认帧i正确），标记帧i为已确认，若i=send\_base则滑动窗口（更新send\_base为最小未确认帧序号）；  

   • 若帧i超时（未收到ACKi），仅重传帧i。  

2. 接收方：  
   • 接收窗口[recv\_base, recv\_base+W_r-1]，缓存窗口内所有正确帧（乱序）；  

   • 若收到帧i在窗口内且正确，缓存并发ACKi；若i=recv\_base，则滑动窗口（交付连续帧给网络层）；  

   • 若帧i不在窗口内（过期帧），发ACKi（避免发送方重传）。  

示例（窗口大小N=5，帧2出错，帧3、4正确）

  
发送方发帧0~4 → 接收方收帧3（正确，缓存）→ 发ACK3，收帧4（正确，缓存）→ 发ACK4，收帧2（出错）→ 丢弃，不发ACK；  
发送方收ACK3、ACK4 → 窗口滑动（send_base=3？不，仅帧3、4确认，send_base仍为2），帧2超时→ 仅重传帧2；  
接收方收帧2（正确）→ 发ACK2，此时recv_base=0，帧0、1已收，交付0、1、2、3、4给网络层。  
  

关键约束：窗口大小与序号空间

为避免新旧帧混淆（如序号循环后，新帧被误认为旧帧），需满足 W_t + W_r ≤ 2^n（n为序号位数，序号范围0\sim2^n-1）。通常取W_t=W_r=N≤2^{n-1}（如n=3，序号0~7，N≤4）。  

优缺点

• 优点：效率高（仅重传出错帧），适合高误码率场景；  

• 缺点：实现复杂（需缓存乱序帧、排序、独立确认），接收方需更大缓冲区。  

典型应用

无线局域网（Wi-Fi，IEEE 802.11）、卫星通信（高误码率链路）。

核心结论

• 停止等待是流量控制的基础，通过“一发一收”保证可靠，但效率低；  

• 滑动窗口通过“连续发送+窗口滑动”提高效率，GBN和SR是两种策略：  

  • GBN用“累积确认+后退重传”简化实现，适合误码率低的场景；  

  • SR用“独立确认+选择重传”优化效率，适合高误码率场景，但实现复杂。  

三者均属于ARQ（自动重传请求）协议，结合流量控制与差错控制（重传机制），是现代数据链路层可靠传输的核心。
