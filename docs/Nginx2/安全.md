左侧：客户端请求 `R`

规则：

## 二、全文完整翻译&核心作用
恶意攻击者会用自动化脚本发起海量请求（密码暴力破解、DDoS攻击），快速耗尽服务器CPU、内存、带宽资源。
导致正常用户网站卡顿、打不开、报错。

Nginx 限流（Rate Limiting）就是解决这个问题的安全方案：
限制**单个用户IP在指定时间内的最大HTTP请求数**。

### 限流4大核心价值（考试简答题）
1. **安全防护**：减缓暴力破解密码、账号撞库攻击速度
2. **DDoS防护**：拦截异常高频恶意流量，抵御CC/DDoS攻击
3. **防止服务过载**：避免后端被突发海量请求打垮
4. **IP访问管控**：配合IP黑白名单限制恶意主机访问

## 三、Nginx 限流算法：漏桶算法 Leaky Bucket（必考）
Nginx 原生限流基于**漏桶算法**：
1. 流入水 = 客户端进来的请求
2. 桶 = FIFO先进先出请求队列
3. 漏水 = 服务器匀速处理请求的速度
4. 桶溢出 = 请求超过阈值，直接丢弃拒绝（返回503）

原理：不管请求进来多快，服务器**永远匀速处理**，不会被突发流量打垮。

## 四、两大核心限流指令（考试重中之重）
### 1. limit_req_zone 定义限流内存区域（http块配置）
语法：
```nginx
limit_req_zone key zone=名称:大小 rate=速率;
```
教材示例：
```nginx
limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;
```
逐字段解释：
- `$binary_remote_addr`：限流Key，**客户端IP地址**（最常用）
- `zone=mylimit:10m`：定义共享内存区域`mylimit`，分配10MB内存存储限流状态
- `rate=10r/s`：限流速率 → **每秒最多10个请求**

> 考点：Nginx毫秒级精准限流
> 10r/s = 每**100毫秒放行1个请求**，小于100ms连续请求直接触发限流

### 2. limit_req 应用限流规则（server/location块）
语法：
```nginx
limit_req zone=区域名 burst=队列大小 [nodelay];
```
教材示例：
```nginx
limit_req zone=mylimit burst=20 nodelay;
```

## 五、Burst 突发队列 + Nodelay 无延迟（本节最难考点）
### burst 是什么？
`burst=20`：桶的队列长度为20
超出匀速速率的突发请求，先放入burst队列排队，而不是直接拒绝。
示例：`rate=10r/s burst=20`
- 瞬间21个请求：前1个立刻处理，剩余20个进入burst队列排队匀速处理
- 瞬间22个请求：队列满了，**第22个直接拒绝返回503**

### nodelay 是什么？
- 不加nodelay：队列里的请求**排队延迟处理**，响应慢
- 加`nodelay`：burst队列里的请求**立刻放行处理**，不排队延迟；只拒绝超出burst的溢出请求
> 生产环境几乎都搭配 `nodelay` 使用

## 六、完整标准限流配置（教材原版，直接背）
```nginx
http {
    # 定义限流区域：按IP限流，每秒10请求，10M内存
    limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;

    server {
        listen 80;
        server_name www.example.com;

        location /api/ {
            # 应用限流：突发队列20，无延迟立刻处理
            limit_req zone=mylimit burst=20 nodelay;
            proxy_pass http://backend_server;
        }
    }
}
```

## 三、对应你例子的完整执行步骤（100%匹配原文）
> rate=10r/s、burst=20
1.  **Request #1 第1个请求**
    队列空闲 → **立刻转发到后端上游服务 Immediate to Upstream**，占用1个槽位，100ms后释放
2.  **Requests #2-21 第2~21个请求（一共20个）**
    还有剩余队列槽位 → **进入Burst突发队列排队缓存 Queued in Burst**，填满20个队列槽位
3.  **Request #22 第22个请求**
    burst队列20个槽位**全部占满** → **直接拒绝，返回503错误 Rejected with 503**

## 四、结合漏桶算法理解（考试必背）
1.  `rate=10r/s` = 桶底部**匀速漏水速度**：每100ms漏出1个请求，槽位释放1个
2.  `burst=20` = 桶的**最大容量**：最多缓存20个突发请求
3.  桶没满：突发请求进队列缓存，不报错
4.  桶满溢出：新请求直接丢弃，返回503

## 五、加上 nodelay 的区别（考点）
你这个流程图是**不带nodelay**的原生逻辑：
- 队列里#2~21请求是**排队延迟处理**，等100ms槽位释放才依次转发

带 `nodelay`：
- burst队列里的请求**立刻转发处理**，不排队等待；只统计占用槽位，槽满才拒绝，响应零延迟。

# Nginx 两级限流（Two-Stage）+ 地域IP管控（geo）完整图文解析
完全对照你这张**时间轴流程图** + 教材原文，**考试直接背、直接复制**👇

## 一、先看懂这张限流流程图（核心考点）
配置：`rate=5r/s burst=12 delay=8`
时间轴 0.0 ~ 3.0：
1. **绿色箭头 (Allowed, no delay)**：**放行且无延迟**
    *   前8个额外请求（`delay=8`），**立刻**放行，不排队。
2. **蓝色箭头 (Delayed)**：**延迟排队**
    *   接下来4个请求（`12 - 8 = 4`），进入队列**延迟处理**，保证速率在 `5r/s`。
3. **橙色箭头 (Rejected)**：**直接拒绝**
    *   超出 `burst=12` 总队列的请求，**直接返回 503 错误**。

> **考点**：`delay` 是 Nginx 1.15.7 版本引入的**二级限流**核心参数！

---
## 二、全文翻译 & 二级限流（Two-Stage）原理
### 1. 什么是二级限流？
Nginx 1.15.7+ 引入 `delay` 参数，实现**分阶段限流**。
配置：`limit_req zone=myzone burst=12 delay=8;`
*   **基础速率 (rate=5r/s)**：每100ms放行1个请求。
*   **突发队列 (burst=12)**：最多缓存12个突发请求。
*   **延迟阈值 (delay=8)**：**前8个**突发请求**立刻放行**，不延迟；**超出8个的部分**（第9~12个）才排队延迟。

### 2. 为什么要这样设计？
*   普通用户瞬间加载多个资源（如网页加载 4-6 个图片），**不会因为突发流量被误判为攻击**，体验流畅。
*   恶意攻击者高频疯狂请求，**会被匀速限制或直接拒绝**，保护服务器。

### 3. 流程对应
*   **0~1.0s**：21点请求 → 前8个（绿色）立刻放行，后13个进入队列。
*   **1.0~2.0s**：匀速漏水，处理队列里的请求。
*   **2.0~3.0s**：橙色箭头请求超出总队列（12个），**直接报错 503**。

## 三、Nginx 完整配置示例（教材原版）
```nginx
http {
    # 定义限流区域：每秒5个请求，10M内存
    limit_req_zone $binary_remote_addr zone=my_limit:10m rate=5r/s;

    server {
        listen 80;
        server_name example.com;

        location / {
            # 二级限流：突发队列12，前8个无延迟，剩余排队
            limit_req zone=my_limit burst=12 delay=8;
            proxy_pass http://backend;
        }
    }
}
```

## 四、多级限流 + IP白名单（Allowlist）案例
考试爱考：**不同IP不同限流策略**（内部员工/合作伙伴限流宽，公网限流严）

### 核心逻辑
1. **geo 指令**：根据IP判断归属，给变量赋值。
2. **map 指令**：配合 geo，生成不同的限流 Key。
3. **双重 limit_req**：白名单IP走宽松规则，非白名单IP走严格规则。

### 配置代码（考试满分答案）
```nginx
http {
    # 1. 定义 geo 变量：标记IP是否在白名单
    geo $limit_allow {
        default 1;          # 默认非白名单
        10.0.0.0/8 0;       # 内网IP段，标记为0（不限流/宽松）
        192.168.0.0/24 0;   # 内网IP段
    }

    # 2. 映射 key：根据 $limit_allow 决定用哪个IP作为限流键
    map $limit_allow $limit_key {
        0 "";               # 白名单，返回空key，不触发第一个限流
        1 $binary_remote_addr; # 非白名单，用IP限流
    }

    # 3. 定义两个限流区域
    limit_req_zone $limit_key          zone=req_zone_public:10m rate=5r/s;  # 公网：严限5r/s
    limit_req_zone $binary_remote_addr    zone=req_zone_wl:10m    rate=15r/s; # 白名单：宽限15r/s

    server {
        listen 80;
        server_name example.com;

        location / {
            # 4. 应用双重限流
            # 非白名单IP：匹配 req_zone_public，限速5r/s，burst=10
            limit_req zone=req_zone_public burst=10 nodelay;
            
            # 白名单IP：不匹配上面的（key为空），匹配下面的 req_zone_wl，限速15r/s
            limit_req zone=req_zone_wl    burst=20 nodelay;

            proxy_pass http://backend;
        }
    }
}
```
**考点解释**：
*   **白名单IP (10.x.x.x)**：`$limit_allow=0`，`$limit_key=""`。
    *   第一个 `limit_req` 匹配不到（key为空），不生效。
    *   第二个 `limit_req` 生效，按 **15r/s** 放行。
*   **公网IP**：`$limit_allow=1`，`$limit_key=IP`。
    *   两个 `limit_req` 都匹配。Nginx 取**更严格的**（5r/s < 15r/s），最终按 **5r/s** 限流。

## 五、geo 指令：IP 地域/白名单管控（Restricting Access）
### 1. 作用
根据客户端 IP 地址给变量赋值。
常用于：**IP 白名单/黑名单、地区封禁 (Geo-blocking)、路由分流**。

### 2. 语法与示例
```nginx
http {
    geo $ip_region {
        default "OTHER";      # 默认其他
        192.168.0.0/24 "LOCAL"; # 本地网段
        203.0.113.0/24 "CORP";  # 公司网段
        10.0.0.0/8 "INTERNAL";  # 内网
    }

    server {
        location /admin {
            # 仅允许公司/内网访问
            if ($ip_region !~ "CORP|INTERNAL") {
                return 403; # 拒绝访问
            }
            proxy_pass http://backend;
        }
    }
}
```

### 3. 高级用法（考试加分项）
*   **Geo-blocking**：集成 MaxMind GeoIP2 数据库，封禁特定国家 IP。
*   **Rate Limiting by Region**：结合 `geo` 和 `limit_req`，给不同地区设置不同限流策略。
*   **Traffic Routing**：不同地区流量转发到最近的数据中心。

# 一、核心概念（一句话）
Nginx 作为反向代理时：
- **客户端 ↔ Nginx** 可以用 HTTPS 加密
- **Nginx ↔ 后端服务** 默认是**明文 HTTP**
→ 不安全！必须配置 **Nginx → 后端也走 TLS/HTTPS**

这就是本节内容：**Securing Traffic to Upstream Servers**

# 二、Nginx 连接后端 HTTPS 必须配置的指令（必考）
## 1. `proxy_pass https://...`
必须用 https 协议转发

## 2. `proxy_ssl_certificate`
Nginx 作为**客户端**出示给后端的证书

## 3. `proxy_ssl_certificate_key`
Nginx 客户端私钥

## 4. `proxy_ssl_trusted_certificate`
信任的 CA 根证书（用来验证后端证书是否合法）

## 5. `proxy_ssl_verify on`
**开启后端证书校验**（默认 off，不校验！）

## 6. `proxy_ssl_protocols`
指定 TLS 版本（TLSv1.2 TLSv1.3）

## 7. `proxy_ssl_server_name on`
开启 SNI，多域名 HTTPS 必须加

---

# 三、完整配置（考试直接默写）
```nginx
location / {
    proxy_pass https://backend_server;

    proxy_ssl_certificate     /etc/nginx/client.pem;
    proxy_ssl_certificate_key /etc/nginx/client.key;

    proxy_ssl_trusted_certificate /etc/nginx/ca.crt;
    proxy_ssl_verify   on;
    proxy_ssl_protocols TLSv1.2 TLSv1.3;
    proxy_ssl_server_name on;
}
```

# 四、最容易考的区别：
# ssl_certificate VS proxy_ssl_certificate
## 1. ssl_certificate
- 作用：**客户端 → Nginx**
- 位置：`server` 块
- 用途：给浏览器看的服务器证书

## 2. proxy_ssl_certificate
- 作用：**Nginx → 后端服务器**
- 位置：`location` / `upstream`
- 用途：Nginx 作为客户端出示给后端的证书
- 用于**双向 TLS（mTLS）**

# 五、TCP 流量加密（Stream 模块）
如果 Nginx 代理 **TCP 协议**（数据库、邮件、自定义TCP）
用 **stream {}** 块，不是 http {}

```nginx
stream {
    upstream tcp_backend {
        server 10.0.0.1:3306;
    }

    server {
        listen 3306;
        proxy_pass tcp_backend;

        proxy_ssl on;  # 开启TCP TLS
        proxy_ssl_certificate /etc/nginx/client.pem;
        proxy_ssl_certificate_key /etc/nginx/client.key;
        proxy_ssl_verify on;
    }
}
```
