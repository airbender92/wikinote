# 译文：Nginx 介绍（专业流畅版）
Nginx，读作 **Engine-X**，是 Web 服务器技术领域的一次**范式革新**。它最初由 Igor Sysoev 在 2004 年开发，目的是解决传统服务器在**高并发场景**下的性能瓶颈。如今，Nginx 已迅速成为全球最流行、使用最广泛的 Web 服务器之一。凭借轻量架构、极致性能和丰富功能，它成为 Netflix、Airbnb、Dropbox 等全球超大型流量网站的首选技术。

但 Nginx 究竟是什么？它又凭什么脱颖而出？

简单来说，Nginx 是：
- 一款广受欢迎的**开源 Web 服务器与反向代理**，以高性能、低内存占用、高并发处理能力著称。
- 一款通用的 **HTTP 服务器**，被大型网站和应用用来高效承载海量 Web 流量。
- 一款支持邮件协议（IMAP、POP3、SMTP）的代理服务器，同时也是面向 HTTP、TCP、UDP 的**反向代理与负载均衡器**。

Nginx 如此受欢迎，主要依靠**三大核心优势**：
1. **高效与稳定**
    在高并发压力下依然表现出色，无论是静态资源服务、反向代理动态请求，还是负载均衡，都能提供顶尖性能与可靠性。

2. **通用与可扩展**
    架构模块化，拥有丰富的第三方模块生态，可以高度定制。既能做反向代理、缓存、SSL 安全加固，也能通过 fastcgi 等模块成为完整的应用服务器，灵活性远超同类产品。

3. **微服务与容器化适配**
    伴随微服务与容器化浪潮，Nginx 成为现代基础设施的关键组件。它轻量、高效，能无缝融入 Docker、Kubernetes 等云原生技术，是分布式系统中编排复杂应用、管理流量的**必备工具**。

# 译文：反向代理服务器
**反向代理服务器**（Reverse Proxy Server）是介于客户端和Web服务器之间的**中介**（硬件或软件形态）。

其工作流程如下：
1. 客户端向反向代理服务器发送 **HTTP请求**。
2. 反向代理服务器将该请求转发给后端的Web服务器。
3. Web服务器处理请求后，将 **HTTP响应** 返回给反向代理服务器。
4. 最后，反向代理服务器把响应数据作为自己的响应，回传给客户端。

常用的反向代理服务器包括：**Apache**、**HAProxy** 以及 **Nginx**。

### 核心图解解析
这张图直观呈现了Nginx作为反向代理的流转链路：
- **Client（客户端）**：用户浏览器，发起请求的起点。
- **Reverse Proxy Server（反向代理服务器）**：图中带Nginx标志的绿色节点，负责**中转请求**，隐藏后端真实服务器IP。
- **Web Server（Web服务器）**：后端真实处理业务的服务器，对外不可直接访问。

### 关键作用
- **隐藏后端真实地址**：提升系统安全性，避免后端服务器直接暴露在公网。
- **负载均衡**：将请求分发到多台后端服务器，提升整体并发处理能力。
- **缓存静态资源**：减轻后端服务器压力，加快响应速度。
- **SSL终端**：统一处理HTTPS加密/解密，简化后端服务器配置。

# 译文：负载均衡器
**负载均衡器（Load Balancer）**是一种设备、软件或服务，它会将客户端流量分发到多台后端服务器/资源上。

它的核心目的，是避免单台服务器因流量过大而不堪重负。负载均衡有多种分发算法，我们会在第3章《负载均衡》中详细介绍。

---

### 图解解析
这张图清晰展示了Nginx作为负载均衡器的工作流程：


1.  **Clients（客户端）**：多个用户同时发起请求，流量通过Internet汇聚。
2.  **Load Balancer（负载均衡器）**：Nginx在这里作为流量入口，它根据预设的算法，把请求均匀/按规则转发给后端服务器。
3.  **Webservers（后端服务器集群）**：多台真实的业务服务器，共同承接流量，避免单点压力过大。

---

### 核心价值
- **提升系统吞吐量**：多台服务器共同承载，大幅提升并发处理能力。
- **避免单点故障**：如果一台服务器挂了，负载均衡器会自动把流量切到健康的节点上，保证服务不中断。
- **实现弹性伸缩**：可以根据流量增减，动态增加或减少后端服务器数量。
- **简化客户端配置**：用户只需要访问负载均衡器的地址，不用关心后端有多少台服务器。

---

### 三者关系梳理
现在我们可以把Web服务器、反向代理、负载均衡器串起来理解：
- **Web服务器**：负责直接处理HTTP请求，生成并返回响应。
- **反向代理**：作为客户端和Web服务器的中介，转发请求并隐藏后端。
- **负载均衡器**：反向代理的进阶形态，它把请求分发到多台Web服务器上，实现高可用和高性能。

**Nginx 正是同时具备这三种角色能力的工具**，这也是它在现代架构中不可或缺的原因。

# 译文：Nginx 架构解析
Nginx **事件驱动、异步非阻塞**的架构，赋予了它卓越的性能与极高的可靠性。我们可以用一个披萨店的类比来轻松理解它的核心工作方式：

想象你开了一家披萨店，店里有：
- **店长（Master Process 主进程）**
- **店员（Worker Processes 工作进程）**
- **主厨（Cache Manager 缓存管理器）**
- **备料员（Cache Loader 缓存加载器）**

店长的职责是接订单、管理店员；而真正制作披萨的是店员。店里会提前备好热门披萨的食材，存放在指定区域（缓存）。当顾客点单时，店员可以直接取用备好的食材，比从头开始做快得多——这就像 Nginx 里的 `Cache Loader` 预加载缓存的作用。主厨（Cache Manager）则负责监控和维护缓存库存，确保热门食材充足，让资源利用更高效。

---

## 对应到 Nginx 架构
就像披萨店有店长和多名店员，Nginx 也由 **1 个主进程（Master Process）和多个工作进程（Worker Processes）** 组成。主进程只负责管理，不直接处理请求，避免了像店长亲自做披萨那样拖慢整体效率。

![workerprocess](../assets/Nginx2/workerprocess.jpeg)

### 1. Master Process（主进程）
Nginx 服务启动时，会先初始化主进程。它的核心职责是：
- 读取并校验配置文件
- 绑定监听端口、初始化日志
- 创建工作进程、缓存加载器、缓存管理器
- 管理工作进程（启动、停止、重启）

### 2. Worker Processes（工作进程）
真正干活的是工作进程！它们负责：
- 处理客户端的网络连接、读写磁盘内容
- 反向代理、负载均衡（将请求分发到上游服务器）
- 健康检查、处理客户端请求

每个工作进程都是**单线程**的，它们之间通过**共享内存**来同步缓存数据、会话信息等共享资源，避免重复工作。

### 3. Cache Loader（缓存加载器）
当 Nginx 重启（或首次启动）时，`Cache Loader` 会扫描磁盘上的缓存目录，将已有缓存文件的元数据加载到内存中。这样，Nginx 可以直接使用这些已缓存的文件，不用再重新请求或计算，就像披萨店提前备好的食材，随取随用。

### 4. Cache Manager（缓存管理器）
`Cache Manager` 会定期运行，检查磁盘上的缓存条目，负责缓存的维护与清理，确保缓存资源高效利用，避免磁盘空间被无用数据占满，就像主厨管理食材库存一样。

---

## 关键配置：工作进程数
在 `/etc/nginx/nginx.conf` 中，可以配置工作进程的数量：
```nginx
# 推荐设置为 auto，自动匹配服务器的 CPU 核心数
worker_processes auto;
```
通常设置为和 CPU 核心数一致，能最大化发挥多核性能，避免进程间的上下文切换开销。

# 译文 + 核心知识点：如何查看 Nginx 工作进程数
## 英文原文翻译
**如何知道你的 Nginx 环境中配置了多少个工作进程？**
打开你的 Linux 终端，输入以下命令：
```bash
grep -E 'processor|core id' /proc/cpuinfo
```
你会看到类似图 1.6 的输出，显示如下信息：

![cpuinfo](../assets/Nginx2/cpuinfo.png)

---

## 关键解释（超级重要）
1. **这个命令的作用**
   它不是**直接查 Nginx**，而是**查看你的服务器有多少个 CPU 核心**。
   → **Nginx 官方推荐：`worker_processes` 设置成和 CPU 核心数一样**。

2. **为什么要这么做？**
   Nginx 工作进程是**单线程、事件驱动**的，
   1 个工作进程 = 跑满 1 个 CPU 核心
   所以核心数 = 最佳工作进程数。

3. **真正查看 Nginx 当前运行的工作进程数**
   用这个 Linux 命令：
   ```bash
   ps aux | grep nginx
   ```
   - 显示 `master process`：**1 个（管理者）**
   - 显示 `worker process`：**实际运行的工作进程数量**

4. **查看 Nginx 配置文件里写的进程数**
   ```bash
   grep worker_processes /etc/nginx/nginx.conf
   ```
   常见配置：
   ```nginx
   worker_processes auto;  # 自动匹配CPU核心数（最佳实践）
   # 或 worker_processes 4;
   ```

---

## 快速总结
你要记住这 **3 条命令** 就够了：
```bash
# 1. 看服务器CPU核心数（决定最佳worker数）
grep -c processor /proc/cpuinfo

# 2. 看Nginx正在运行多少个worker进程
ps aux | grep nginx | grep worker

# 3. 看Nginx配置里写的进程数
grep worker_processes /etc/nginx/nginx.conf
```

# 译文：Nginx 事件驱动 / 异步 / 非阻塞架构解析
先给你翻译原文，再结合这张架构图，把三个核心概念彻底讲透👇

---

## 原文翻译
**事件驱动（Event-Driven）**  
在事件驱动架构中，Nginx 不会为每个连接分配独立的进程或线程，而是使用少量工作进程，并依靠**事件循环**来处理请求。事件循环通过系统级调用执行 I/O 事件，并在事件发生时立即处理它们。

这样一来，Nginx 无需等待，就能高效处理大量新的并发连接。这些事件包括：接收新连接、完成读写操作、重载配置文件等。事件驱动模型让 Nginx 无需为每个连接创建专用线程/进程，就能快速响应各类事件。

**异步处理（Asynchronous Processing）**  
异步处理就像披萨店的员工，同时处理多个订单，不用等一个订单完全做好再开始下一个。Nginx 也一样，它不会等一个操作完成再处理下一个。

比如，当 Nginx 执行文件读写这类 I/O 操作时，不会阻塞其他任务（如重载配置）的执行。相反，它会在等待 I/O 操作完成的同时，并行处理其他事件。因此，Nginx 能高效处理大量并发连接，更充分地利用系统资源。

**非阻塞 I/O（Non-Blocking I/O）**  
非阻塞 I/O 就像接订单时不会让整个披萨制作流程停摆，让披萨店在高峰期也能平稳运转。

Nginx 采用非阻塞 I/O 模型，意味着当执行读取客户端数据这类操作时，它不会等操作完成再去处理下一个任务，而是继续处理其他事件和请求。

总而言之，Nginx 事件驱动、异步、非阻塞的架构，让它能以极低的资源消耗，高效处理海量并发连接。这些特性在需要高性能和可扩展性的场景（如 Netflix）中尤为重要。

---

## 架构图深度拆解


这张图把 Nginx 的内部处理流程拆解得非常清楚：

1.  **左侧入口：WEB、EMAIL 和 TCP 流量**
    客户端的 HTTP/HTTPS、邮件（IMAP/POP3/SMTP）和通用 TCP 流量，都会先进入 Nginx。

2.  **核心：非阻塞事件驱动处理引擎**
    这是整个架构的心脏，里面包含三个状态机：
    - `STREAM STATE MACHINE`：处理通用 TCP/UDP 流量（比如数据库代理、自定义TCP服务）
    - `HTTP STATE MACHINE`：处理 HTTP/HTTPS 请求，是最核心的模块
    - `MAIL STATE MACHINE`：处理邮件协议代理
    它们都在同一个事件循环中运行，由少量工作进程统一调度。

3.  **辅助线程：处理阻塞操作**
    虽然 Nginx 主流程是非阻塞的，但磁盘 I/O（读写文件、日志）这类天生阻塞的操作，会交给独立的辅助线程处理，避免拖慢主事件循环。

4.  **对外接口**
    - 上层：反向代理接口，可转发 HTTP、邮件、TCP 流量
    - 中层：FastCGI/uWSGI/SCGI 网关，对接后端应用服务器（比如 PHP、Python 应用）
    - 下层：静态内容缓存、访问/错误日志，直接和磁盘交互

---

## 三个核心概念的通俗对比
| 概念 | 通俗类比 | Nginx 实际效果 |
| :--- | :--- | :--- |
| **事件驱动** | 店员不用每个订单单独派一个人，而是统一看“待办事件清单”，哪个好了就处理哪个 | 用少量工作进程，处理海量并发连接 |
| **异步处理** | 烤披萨的同时，可以接下一个订单、准备配料，不用等一个烤完再做下一个 | 等待 I/O 操作的间隙，继续处理其他请求 |
| **非阻塞 I/O** | 不会因为某个顾客没说完订单，就暂停接下一个顾客 | 执行读写操作时，不会卡住整个进程，继续处理其他事件 |

这三者共同作用，让 Nginx 实现了**单进程高并发、低资源占用**的优势，也是它能支撑 Netflix、Airbnb 这类超大规模流量的根本原因。

# 译文：Nginx 如何提升网站安全性
我们将网站部署在反向代理之后，是为了提升安全性。像亚马逊这样的电商平台会存储客户的个人信息、支付明细等敏感数据，因此 Web 服务器与反向代理的安全性至关重要。

Nginx 主要通过以下方式增强网站安全：

## 1. 禁止直接访问后端 Web 服务器
反向代理是客户端与 Web 服务器之间的**中间层**。使用 Nginx 后，后端服务器的真实 IP 不会暴露给客户端，大幅降低被针对性攻击（如 DDoS）的风险。
Nginx 还支持 **IP 封禁** 和 **限流** 等安全能力（本书后续会详细讲解）。

## 2. SSL 加密（SSL 终端/卸载）
反向代理可以统一对**入站请求解密**、**出站响应加密**。
加解密运算非常消耗性能，由 Nginx 承担后，可以显著减轻后端应用服务器的负担。

## 3. 速率限制（Rate Limiting）
Nginx 支持限流功能，可以控制客户端在指定时间内的请求次数（每秒/每分钟）。
能有效防御**暴力破解**、**DoS / DDoS 攻击**。

## 4. 访问控制（Access Controls）
Nginx 可以基于 **IP 地址、地理位置** 等条件设置访问规则，允许或拒绝特定资源的访问，防止敏感路径被未授权访问。

## 5. 安全响应头
Nginx 可以配置各类安全头部，例如：
- Content Security Policy (CSP)
- Strict-Transport-Security (HSTS)
- X-Content-Type-Options

这些头部提供额外防护，抵御多种攻击。
典型配置：`server_tokens off;`
作用：隐藏 Nginx 版本号，浏览器不会看到类似 `Server: nginx 1.21.1` 的信息，减少暴露攻击面。

## 6. 请求过滤
通过正则表达式、精准字符串匹配编写 location 规则，Nginx 可以过滤恶意请求，对异常路径直接返回 404。
能够拦截携带恶意 payload、攻击特征的非法请求。

# 译文：Apache 介绍 & Nginx 与 Apache 核心对比
Apache 是由 Apache 软件基金会开发维护的**免费开源 Web 服务器软件**，是 Nginx 的主要替代方案。它速度快、稳定可靠，是仅次于 Nginx、全球使用最广泛的 Web 服务器之一。

Nginx 采用**事件驱动架构**，**单线程即可同时处理大量请求**；
而 Apache 是**进程驱动架构**，**为每个请求创建单独线程**。

最新数据显示，Nginx 目前的市场份额高于 Apache：
- Nginx 约占 **33.9%**
- Apache 约占 **27%**

### 极简对比（一眼看懂区别）
| 特性 | Nginx | Apache |
|---|---|---|
| **架构** | 事件驱动（单线程 handle 大量请求） | 进程/线程驱动（一个请求一个线程） |
| **高并发能力** | 极强，低内存 | 较弱，高资源消耗 |
| **静态资源** | 极快 | 一般 |
| **动态内容** | 需配合后端 | 原生支持更好 |
| **市场份额** | 更高（第一） | 次之（第二） |
