# 核心一句话
**Base64 是一种「二进制转文本」的编码方式，把3个字节（24bit）的数据，变成4个可打印字符。**

---

# 核心原理（结合你给的图）
1.  **输入：3个8位字节（共24bit）**
    `01010101 11001100 01010101`
2.  **重新切分：4个6位块（每块对应一个Base64字符）**
    `010101 011100 110001 010101`
3.  **查表：每个6位块 → 一个字符**
    - `010101` → `V`
    - `011100` → `c`
    - `110001` → `x`
    - `010101` → `V`
    所以结果是：`VcxV`

---

# 关键规则
- 每个 Base64 字符代表 **6bit** 数据
- 每 **4个字符** 还原出 **3个字节** 数据
- 如果输入字节数不是3的倍数，就需要用 `=` 填充：
  - 剩 1 个字节（8bit）：填充 **2个 `=`**
  - 剩 2 个字节（16bit）：填充 **1个 `=`**
  - 刚好是3的倍数：不用填充

---

# 和 JWT 的关系
- JWT 里用的不是标准 Base64，而是 **Base64URL**（后面会讲）
- 标准 Base64 里的 `+` 和 `/` 会被换成 `-` 和 `_`，去掉填充的 `=`，方便放在 URL 里

---

# 你给的 Base64 表，只需要记住这 4 类
| 范围 | 字符 |
| :--- | :--- |
| 0-25 | 大写字母 A-Z |
| 26-51 | 小写字母 a-z |
| 52-61 | 数字 0-9 |
| 62-63 | `+` 和 `/` |
| 填充 | `=` |

---

# 一句话总结
**Base64 就是把二进制数据，用「A-Za-z0-9+/」这64个字符表示出来，方便传输，代价是体积变大1/3。**

# 7.1.1.1 Base64-URL（核心）
**Base64-URL = 专门给 URL 用的安全版 Base64**

标准 Base64 里有两个字符 **`+` 和 `/`**，放在 URL 里会被当成特殊符号，导致解析错误。
所以 **Base64-URL 只做了两件事**：
1. 把 `+` 换成 `-`
2. 把 `/` 换成 `_`
3. （通常还会去掉末尾填充的 `=`，JWT 就是这么做的）

# 一、SHA 是什么（一句话核心）
**SHA 是一套安全哈希算法，JWT 里用的是 SHA-2 家族（主要是 SHA-256）。**
作用：把任意长度的数据，算出一串**固定长度、不可逆、防篡改**的哈希值。

# 二、关键知识点（必须记住）
1. **SHA-1 已废弃**（不安全）
2. **JWT 只用 SHA-2**：SHA-256、SHA-512
3. **SHA-256 输出固定 256 位（32 字节）**
4. 特点：
   - 输入改一点点，输出完全不同
   - 无法从哈希反推原始数据
   - 极难找到两个不同数据算出相同哈希

# 三、SHA-256 怎么工作（极简流程）
1. **填充数据**
   把数据补到 **512 位的整数倍**
   - 末尾加 `1`
   - 补 `0`
   - 最后 64 位存原始数据长度
2. **分块处理**
   每次处理 **512 位（64 字节）**
3. **压缩运算**
   用 8 个初始哈希值 + 64 轮固定函数运算
   不断更新哈希
4. **输出结果**
   8 个 32 位值拼接 → **64 个十六进制字符**

# 四、代码里那些函数是干嘛的？
不用懂数学，只需要知道：
- **rotr**：循环右移（比特位旋转）
- **ch / maj**：选择、多数投票逻辑
- **bsig0 / bsig1 / ssig0 / ssig1**：混淆函数，让结果更乱更安全

这些都是**标准固定函数**，库会自动调用，你不用自己写。

# 五、JS 实现的核心
1. **padMessage**
   给数据做 SHA-256 要求的**填充**，让长度符合 512 位规则
2. **sha256 主函数**
   - 初始化 8 个哈希常量
   - 循环处理每块数据
   - 64 轮压缩运算
   - 输出最终哈希（十六进制字符串 或 字节数组）

# 六、SHA-256 和 SHA-512 的区别
- **SHA-256**：32 位运算，512 位块，256 位输出
- **SHA-512**：64 位运算，1024 位块，512 位输出
JS 只原生支持 32 位运算，所以 SHA-512 需要额外库。

---

# 终极精简总结（你只记这个）
1. **SHA-256 = JWT 最核心的哈希算法**
2. **作用：防篡改、验签名、生成指纹**
3. **流程：填充 → 分块 → 压缩 → 输出固定哈希**
4. **不可逆、极难碰撞、改一点全变样**
5. **JS 实现靠固定函数 + 填充 + 64 轮运算**

# 一、HMAC 是什么（一句话核心）
**HMAC 是一种带密钥的哈希校验算法，用「消息 + 密钥」算出一个防篡改的验证码。**
它能同时实现**身份认证（知道密钥的才对）**和**数据完整性（数据被改就会失效）**。

---

# 二、流程图拆解（你给的这张图）
```
      ┌────────────────────┐
      │  哈希算法（SHA）  │
      └─────────┬────────┘
                ↓
┌────────┐    ┌───────────┐    ┌────────┐
│ Message│───▶│ HMAC 算法 │───▶│ HMAC   │
└────────┘    └───────────┘    └────────┘
                ↑
      ┌─────────┴────────┐
      │ Shared Secret    │
      └─────────────────┘
```
**三个输入，一个输出**：
1.  **Message**：要保护的原始数据（比如 JWT 的头部+载荷）
2.  **Shared Secret**：双方共享的密钥（只有知道的人才能算出正确结果）
3.  **Hashing Algorithm**：基础哈希算法（比如 SHA-256）
4.  **输出**：HMAC 验证码（JWT 里的签名部分）

---

# 三、核心公式（你看到的那行）
```
HMAC(message) = H( (K' XOR opad) || H( (K' XOR ipad) || message ) )
```
拆解成大白话：
1.  **预处理密钥 K'**
    - 密钥太短：补 0 到块长度
    - 密钥太长：先哈希一次再用
2.  **内层哈希**：
    `H( (K' XOR ipad) || message )`
    密钥和 ipad 异或后，和消息拼接，做一次哈希
3.  **外层哈希**：
    `H( (K' XOR opad) || 内层哈希结果 )`
    密钥和 opad 异或后，和内层哈希结果拼接，再做一次哈希
4.  最终结果就是 HMAC 验证码

> ipad = `0x36` 重复 B 次
> opad = `0x5C` 重复 B 次
这两个是固定常量，用来给密钥做“两次不同的混淆”，防止直接用原始密钥计算。

---

# 四、JWT 里的应用（HS256 就是它）
- **HS256** = HMAC + SHA-256
- 用一个共享密钥，给头部+载荷算出签名
- 验证时：用同一个密钥重新算一遍，和收到的签名对比，一样就说明数据没被改，且发送方知道密钥

---

# 五、关键特点（必须记住）
1.  **对称算法**：加签和验签用的是同一个密钥
2.  **防篡改**：只要消息改了一个字节，HMAC 结果就会完全不同
3.  **身份认证**：没有密钥的人无法伪造正确的 HMAC
4.  **依赖哈希强度**：MD5 不安全，必须用 SHA-2 系列（SHA-256/SHA-512）

---

# 六、代码逻辑（JS 实现，看懂流程就行）
1.  生成 ipad/opad（`0x36` 和 `0x5C` 填充到块大小）
2.  处理密钥：补 0 或哈希，得到 `paddedSecret`
3.  计算 `ipadSecret` 和 `opadSecret`（密钥和 ipad/opad 异或）
4.  内层哈希：`hash(ipadSecret + message)`
5.  外层哈希：`hash(opadSecret + 内层哈希结果)`
6.  输出最终 HMAC

---

# 终极一句话总结
**HMAC 就是用「密钥+消息+哈希算法」算出一个验证码，JWT 里的 HS256 就是它。**

# 一、这段内容到底在讲什么？
**HS256 = JWT 最基础、最常用的签名算法**
= **HMAC + SHA256**

它做两件事：
1. **生成带签名的 JWT**（`jwtEncode`）
2. **验证 JWT 签名是否合法**（`jwtVerifyAndDecode`）

实现的安全能力：
- **防篡改**（数据改一点，签名立刻失效）
- **身份验证**（只有持有密钥的人才能生成合法签名）

---

# 二、核心原理（一句话）
把 JWT 的**头部 + 载荷**用 **Base64URL** 编码，
再用 **密钥 + HMAC-SHA256** 生成签名，
最后拼接成 **头部.载荷.签名** 格式。

验证时：
用**同样密钥**重新算一遍签名，**对比是否一致**。

---

# 三、`jwtEncode` 生成签名（逐段看懂）
```js
function jwtEncode(header, payload, secret) {
  // 1. 类型检查
  header.alg = 'HS256'; // 强制算法为 HS256

  // 2. 头部、载荷转 JSON → Base64URL 编码
  const encHeader = b64(JSON.stringify(header));
  const encPayload = b64(JSON.stringify(payload));

  // 3. 拼接前两段
  const jwtUnprotected = `${encHeader}.${encPayload}`;

  // 4. 用 HMAC-SHA256 生成签名
  const signature = b64(hmac(sha256, 512, secret, stringToUtf8(jwtUnprotected), true));

  // 5. 三段拼成最终 JWT
  return `${jwtUnprotected}.${signature}`;
}
```

## 真正做的 4 件事
1. 把头部、载荷变成 **Base64URL**
2. 拼接成 `头部.载荷`
3. 用 **密钥 + HMAC-SHA256** 签名
4. 签名也编码 → 最终 `头部.载荷.签名`

---

# 四、`jwtVerifyAndDecode` 验证签名
```js
function jwtVerifyAndDecode(jwt, secret) {
  // 1. 拆成三段
  const split = jwt.split('.');

  // 2. 解析头部，确认算法是 HS256
  const header = JSON.parse(unb64(split[0]));

  // 3. 重新计算签名
  const newSignature = b64(hmac(sha256, 512, secret, stringToUtf8(`${split[0]}.${split[1]}`), true));

  // 4. 对比签名是否一致
  return {
    valid: newSignature === split[2]
  };
}
```

## 验证逻辑（超级简单）
- 拆分 JWT
- 用**同样密钥 + 同样算法**重新生成签名
- **新签名 == 旧签名 → 合法、未被篡改**

# 一句话核心
**RSA 是目前最主流的非对称加密/签名算法。
安全来自：大质数乘积极难反向分解。
公钥加密/验签，私钥解密/签名。**

---

# 1. RSA 是什么？
- **非对称加密**：有**公钥**和**私钥**两把钥匙
- **公钥**：公开给所有人，用于**加密 / 验签**
- **私钥**：自己保密，用于**解密 / 签名**
- JWT 里的 **RS256** 就是 RSA + SHA256

---

# 2. 为什么 RSA 安全？（核心原理）
RSA 的安全来自一个数学难题：
### **大整数质因数分解极难**
- 选**两个很大的质数 p、q**
- 相乘得到 **n = p × q**
- n 公开，但**p、q 保密**
- 想从 n 反推 p、q，**以现在的计算机几乎不可能**

例子：
35 = 5×7 很简单
但**2048位的超大数**，人类算几百年都算不出来

---

# 3. 为什么必须用两个质数相乘？
- 两个质数相乘 → 结果**只有两个因子**
- 因子越少 → 越难被分解
- 非质数相乘 → 因子多，很容易被破解

所以 RSA 强制要求：**p、q 必须是大质数**

---

# 4. RSA 公式（看懂就行，不用背）
```
(m^e)^d ≡ m (mod n)
```
- **m**：原始消息
- **e、n**：**公钥**
- **d**：**私钥**
- 知道 e、n，**极难算出 d**
- 这就是 RSA 安全的本质

---

# 5. RSA 在 JWT 里怎么用？
- **私钥签名**（只有服务器能做）
- **公钥验签**（所有人都能验证）
- 任何人都能验，但**没人能伪造签名**

这比 HS256（对称）更安全，因为**密钥不用共享**。

---

# 6. 未来威胁：量子计算机
- 现在的 RSA 很安全
- 但未来**量子计算机**能用 **Shor 算法**快速分解大整数
- 一旦实用化，**现有 RSA 会全部失效**
- 目前还没实现，只是研究中

---

1. **RSA = 非对称加密/签名算法**
2. **公钥加密/验签，私钥解密/签名**
3. **安全来自：大质数乘积极难反向分解**
4. **JWT 里的 RS256 就是 RSA+SHA256**
5. **未来量子计算机可能破解 RSA**

# 7.2.2.1 生成 RSA 密钥对（e, d, n）
## 极简流程（5步）
1. **选两个大质数 p、q**
2. **n = p × q**
   n 是公钥+私钥都有的模数，长度就是密钥长度（2048位）
3. **算 φ(n) = (p-1)(q-1)**
4. **选公钥 e**
   1 < e < φ(n)，且和 φ(n) 互质
   常用 e = 65537
5. **算私钥 d**
   d 是 e 在模 φ(n) 下的逆元
   `e × d ≡ 1 mod φ(n)`

## 最终密钥
- **公钥 = (n, e)**
- **私钥 = (n, d)**
- p、q、φ(n) 必须销毁或保密

## 为什么安全？
知道 n、e，**无法算出 d**，因为分解 n 极难。

---

# 7.2.2.2 RSA 基础签名与验证
## 签名（私钥）
1. 消息 → SHA256 哈希
2. **签名 = 哈希^d mod n**

## 验证（公钥）
1. 解密签名：**哈希 = 签名^e mod n**
2. 重新哈希消息
3. 对比两个哈希 → 一致则合法

---

# 7.2.2.3 RS256 = RSA-SHA256-PKCS1-v1.5
**JWT 最常用的 RSA 签名**

## 签名流程
1. **EMSA-PKCS1-v1_5**
   消息哈希 + DER 包装 + 填充 `00 01 FF...FF 00 哈希`
2. **OS2IP**：字节 → 大整数
3. **RSASP1**：`m^d mod n` 签名
4. **I2OSP**：整数 → 签名字节

## 验证流程
1. 签名 → OS2IP → 整数
2. **RSAVP1**：`s^e mod n`
3. I2OSP → 编码消息
4. 重新编码消息
5. 对比 → 一致则合法

## 特点
- **确定签名**（同样消息每次签名一样）
- 简单、快、兼容广
- JWT 里 alg = **RS256**

---

# 7.2.2.4 PS256 = RSA-SHA256-PSS
**更安全的新一代 RSA 签名**

## 与 RS256 区别
- **随机签名**（每次签结果都不同）
- 用 **MGF1 掩码** + **随机盐 salt**
- 抗攻击更强
- 算法更现代

# 7.2.3 椭圆曲线（EC）**一句话核心**
**EC = 比 RSA 更现代、更安全、更小体积的非对称加密算法。
安全来自：椭圆曲线离散对数难题（极难破解）。
私钥是数字，公钥是曲线上的点。
JWT 里的 ES256 就是它。**

---

# 一、EC 为什么安全？
和 RSA 一样靠**数学难题**保护：
- **RSA**：大整数分解难
- **EC**：椭圆曲线离散对数难（知道 G 和 kG，求不出 k）

**优势：**
- **256位 EC ≈ 3072位 RSA 安全等级**
- 体积更小、计算更快
- 手机/物联网设备首选

---

# 二、曲线方程（不用背）
```
y² = x³ + ax + b
```
所有点都在这条曲线上。
密码学只使用**有限域（素数域）**上的点。

---

# 三、EC 三种核心运算（看懂作用就行）
1. **点加法 P+Q**：两个不同点相加
2. **点加倍 2P**：同一个点加自己
3. **标量乘法 k × G**：核心运算，生成公钥
   - 快速运算：二进制展开 + 加倍 + 加法

**公钥 = 私钥 × G**
G = 固定基点

---

# 四、ECDSA（椭圆曲线签名）是什么？
**ECDSA = 椭圆曲线数字签名算法**
JWT 里就是 **ES256**。

特点：
- **私钥签名**
- **公钥验证**
- 签名结果是 **(r, s)** 一对数字
- 最终拼成 64 字节（32+32）

---

# 五、ES256 签名流程（极简）
1. 消息 → SHA256 哈希 → e
2. 生成随机数 k
3. 计算 k×G → 取 x 坐标 → r
4. 计算 s = (e + d·r) / k mod n
5. 签名 = (r, s)

---

# 六、ES256 验证流程（极简）
1. 消息哈希 → e
2. 计算 c = 1/s mod n
3. u1 = e·c, u2 = r·c
4. 计算点 P = u1·G + u2·公钥
5. P 的 x 坐标 == r → 验证通过

---

# 七、密钥规则（超简单）
- **私钥**：1 ~ n 之间随机数
- **公钥**：点 Q = d × G
- P-256 是 JWT 最常用曲线

---

# 八、代码到底在干嘛？
- **ecMultiply**：标量乘法（核心）
- **ecAdd / ecDouble**：点加、点加倍
- **sign**：生成 (r,s)
- **verify**：验证 (r,s)
- **jwtEncode**：生成 ES256 格式 JWT
- **jwtVerifyAndDecode**：验证 JWT

---

# 九、最重要的安全点
1. **必须验证公钥是否在曲线上**（防攻击）
2. **随机数 k 必须安全**（k 泄露会导致私钥泄露）
3. **P-256/P-384/P-521 是 JWT 标准安全曲线**
