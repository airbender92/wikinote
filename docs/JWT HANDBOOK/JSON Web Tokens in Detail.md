# 全文完整中文解释

## 第一段
正如第1章所述，所有 JWT 都由**三个部分**组成：
1. **头部（Header）**
2. **载荷（Payload）**
3. **签名 / 加密数据（Signature/Encryption Data）**

前两个部分（头部、载荷）都是**固定结构的 JSON 对象**。
第三部分（签名）取决于使用的签名/加密算法。
**如果是没有签名、没有加密的 JWT，第三部分就直接省略。**

JWT 可以被编码成一种紧凑格式，叫做：
**JWS / JWE 紧凑序列化（Compact Serialization）**

---

## 第二段
JWS（签名）和 JWE（加密）规范还定义了**另一种格式**，叫做：
**JSON 序列化（JSON Serialization）**

它不是紧凑格式，而是**展开的 JSON 格式**，
允许一个 JWT 里**包含多个签名**，或者**发给多个接收者**。
这部分会在第4、5章详细讲解。

---

## 第三段（最核心）
**紧凑序列化格式（就是我们平时用的 JWT）规则：**

1. 头部（Header）和载荷（Payload）先转成 **UTF-8 字节**
2. 再用 **URL 安全的 Base64** 编码
3. 签名/加密数据也用同样方式编码
4. 三部分之间用 **点号 `.`** 连接

最终格式就是：
**头部.载荷.签名**

---

## 第四段（关键知识点）
JWT 使用的是 **Base64 的变种**，专门为 URL 安全设计，叫做 **base64url**。

规则：
- 把 `+` 换成 `-`
- 把 `/` 换成 `_`
- 去掉末尾填充的 `=`

文档里所有提到 **Base64** 的地方，**全部指的是这个 URL 安全变种**。

---

## 第五段（示例）
最终生成的 JWT 是一长串可打印字符串，例如：

```

```

三部分用点分隔，顺序是：
**头部 → 载荷 → 签名**

# 3.1 JWT 头部（Header / JOSE Header）完整中文逐段解释
完全忠于原文、不省略、通俗直白，对应你前面学的结构。

---

## 第一段原文解释
每个 JWT 都包含一个**头部（也叫 JOSE 头部）**，里面是**描述 JWT 自身属性的声明（claims）**。
这些声明用来定义：
- 使用什么加密算法
- 当前 JWT 是**签名的**还是**加密的**
- 以及整体应该如何解析后面的内容

简单说：**头部 = 告诉程序：这个JWT是什么类型、用什么算法处理、怎么解码**。

---

## 第二段原文解释
根据 JWT 类型不同，头部必须包含的字段也不一样。
举个例子：
**加密型 JWT（JWE）** 头部，必须包含：
- 密钥加密算法
- 内容加密算法

而**普通签名/明文 JWT** 不需要这些字段。

---

## 第三段：明文/签名 JWT 头部唯一必填字段：alg
对于**非加密 JWT**（我们平时用的绝大多数）：
头部**唯一强制必填字段只有一个：alg**

> **alg**：当前 JWT 用来签名 / 解密的主算法。

重点原文：
> 如果是**无签名明文 JWT**，alg 必须设置为 `none`

---

## 第四段：两个可选字段 typ、cty
### 1. typ：JWT 的媒体类型
作用：当 JWT 和其他带 JOSE 头部的数据混在一起时，用来区分哪个是 JWT。
实际开发几乎用不到。
如果存在，值固定为：`JWT`

### 2. cty：内容类型
- 普通 JWT（载荷就是普通 JSON）：**不能设置 cty**
- **嵌套 JWT（载荷本身又是另一个 JWT）**：必须设置 cty: `JWT`
  告诉程序：载荷里还有一层 JWT，需要继续解析。
嵌套 JWT 非常少见，所以头部几乎都不带 cty。

---

## 第五段：最简单无签名 JWT 头部示例
原文给出最简头部：
```json
{
"alg": "none"
}
```
Base64url 编码后就是：
```
eyJhbGciOiJub25lIn0
```
这就是**完全无签名、明文 JWT** 的标准头部。

---

## 最后一段解释
你可以**自定义额外字段加到头部**。
但一般用处不大，
只有加密 JWE 在解密之前需要一些自定义元数据时才会用到。

---

# 超精简总结（考试记忆版）

2. 非加密 JWT **必填只有 alg**
3. alg=none → 无签名明文 JWT
4. typ：类型（几乎不用）
5. cty：是否是嵌套 JWT（几乎不用）
6. 头部可以自定义字段，但不常用

# 第一段：Payload 整体说明（原文完整翻译+解释）
载荷（Payload）就是**存放所有业务用户数据**的部分。
同时里面也可以包含规范定义好的标准字段声明（claims）。

和头部一样，Payload 也是一个 JSON 对象。
规范里**没有任何字段是强制必填**的，但是部分字段有标准、固定的含义。

JWT 规范规定：
> 程序遇到**不认识的字段，直接忽略即可**，不需要报错。

有固定标准含义的字段，统称为：**注册声明（Registered Claims）**

---

# 3.2.1 注册声明 Registered Claims（标准内置字段，逐个解释）
都是规范定义好、全世界通用的短英文缩写，目的是**让 JWT 尽可能短小**。

## iss (Issuer 签发者)
大小写敏感的字符串 / URI，**唯一标识是谁签发了这个 JWT**。
怎么解读完全由业务自己决定，没有全球统一管理机构。

## sub (Subject 主题/用户主体)
大小写敏感字符串 / URI，**唯一标识这个 JWT 是给谁的（用户本人）**。
意思：当前令牌里所有信息，都是描述这个用户的。
规范要求：在同一个签发者 iss 下，sub 必须唯一；做不到就必须全球唯一。

> 你示例里 `"sub":"1234567890"` 就是标准注册声明，表示用户ID。

## aud (Audience 受众/接收方)
单个字符串/URI，或者数组。
**唯一标识这个令牌是发给谁用的（你的后端API）**。
规则：
验证方必须发现**自己在 aud 列表里**，才承认这个令牌有效；
不在里面直接废弃令牌。

## exp (Expiration Time 过期时间)
数字时间戳（Unix 秒，从1970年纪元开始的秒数）。
设置令牌**从这一刻开始失效、不可用**。
大部分实现会允许一点点时间误差（时钟偏移，宽限几分钟）。

## nbf (Not Before 生效时间)
exp 的反义词。
令牌**必须到达/超过这个时间才生效**，之前一律无效。
同样允许时钟偏移。

## iat (Issued At 签发时间)
令牌**是什么时候签发创建的**，同样是Unix秒时间戳。

## jti (JWT ID JWT唯一编号)
当前这个令牌的全局唯一ID。
用途：防重放攻击、区分内容相同的不同令牌、拉黑作废令牌。
唯一性由业务自己保证。

---

# 3.2.2 公共声明 & 私有声明（Public & Private Claims）
除了上面标准注册声明以外，剩下所有字段分成两类：

## Private Claims 私有声明（自定义私有字段）
由**令牌发行方和使用方自己约定定义**的字段。
比如你示例里的：
`name`、`admin`
都属于私有自定义声明。

注意：自己随便定义容易**字段名冲突撞名**，需要自己规避。

## Public Claims 公共声明
两种：
1. 在 IANA 官方 JWT 注册表注册过的公开标准字段（全球统一，不会冲突）
2. 自己命名时加上命名空间前缀（比如 `myapp_role`），保证不会和别人撞名

---

# 最后一段：重复字段规则
JSON 里如果出现**重复同名key**：
标准处理规则：**只保留最后一个出现的为有效值**。
同时规范也允许实现直接把重复字段的JWT判定为无效。
开发建议：**永远不要写重复字段**。

---

# 终极精简总结（好记版）
1. **Payload = 数据区**，放用户信息+标准字段
2. **Registered Claims 标准内置字段**：iss签发者、sub用户、aud接收方、exp过期、nbf生效、iat签发时间、jti编号
3. **Private Claims**：自己自定义字段（name、admin）
4. **Public Claims**：官方注册/防重命名的公开字段
5. 不认识的字段直接忽略；重复字段以最后一个为准
6. 全部用短缩写，为了让JWT尽可能小

# 3.3 无安全保护JWT（Unsecured JWTs）完整中文逐段解释
完全忠于原文、不省略、通俗讲透原理+风险+用途

---
## 第一段：什么是无保护JWT？
结合前面学的头部+载荷知识，我们可以构造**无签名、无加密的JWT**。
这是最简单的一种JWT：

头部固定极简写法：
```json
{"alg": "none"}
```
含义：**不使用任何签名算法**

再加上用户自定义的载荷，例如：
```json
{
"sub": "user123",
"session": "ch72gsb320000udocl363eofy",
"name": "Pretty Name",
"lastpage": "/views/settings"
}
```

---
## 第二段：编码格式（重点！末尾的点）
因为**没有签名、没有加密**，标准JWT结构是：
`头部.载荷.签名`
现在签名部分是空字符串。

所以编码后长这样：
```
eyJhbGciOiJub25lIn0.eyJzdWIiOiJ1c2VyMTIzIiwic2Vzc2lvbiI6ImNoNzJnc2IzMjAwMDB1ZG9jbDM2M2VvZnkiLCJuYW1lIjoiUHJldHR5IE5hbWUifQ.
```

注意末尾**最后还有一个小数点 .**
原文重点说明：
> 因为签名部分为空，但格式要求必须保留点号 `.`
> 结构永远是：`头部.载荷.` （第三部分为空）

很多初学者第一次看到末尾单独一个点，都会疑惑，这就是原因。

---
## 第三段：这种JWT适合什么场景？（原文核心用途）
这种无签名JWT**只适合纯前端客户端本地使用**。

适用条件：
1. sessionId 是**极难猜测、随机安全**的字符串
2. 里面其他数据**只用来前端展示页面**，后端完全不依赖这些数据做权限校验、登录判断
3. 就算攻击者篡改里面内容，也**得不到任何权限、得不到任何好处**

例子场景：
SPA单页应用页面跳转时：
前端用它保存用户名、上次访问页面地址 `/views/settings`
跳转时**不用请求后端接口**，直接本地渲染好看的昵称页面。

即使恶意用户修改 name、lastpage 字段：
也只是改自己页面显示效果，**无法提升权限、无法冒充别人、无法绕过登录**。
这种场景下，再加签名就是多余浪费性能。

---
## 最后一段：现实情况
但是在**实际工程开发中，无保护JWT非常少见**。

原因：
绝大多数后端接口，都会**信任JWT里的权限数据**
一旦用 alg=none 无签名JWT，攻击者可以任意篡改 sub、权限、用户ID，直接冒充管理员登录。
极度危险。

---
# 精简总结（超好记）
1. 无保护JWT = `alg:none` + 载荷 + **空签名**
2. 格式：`头部.载荷.`（末尾必须带点）
3. 特点：**可任意篡改、完全不防伪造**
4. 仅适合：纯前端展示、后端不校验里面数据的场景
5. 实际开发几乎不用，安全风险极大
6. 你之前学的**签名篡改攻击（alg:none攻击）**，就是利用这种JWT漏洞

# 3.4 制作无保护JWT（Unsecured JWT）
**逐句完整中文解释 + 步骤拆解 + 重点注意事项，完全忠于原文、不省略**

---

## 原文整体意思
本节告诉你：
怎么把 JSON 格式的**头部 + 载荷**，一步步变成标准紧凑格式的**无签名 JWT 字符串**（alg=none 那种）。

---

# 原文5个步骤逐句翻译+通俗解释
## 步骤1
把头部 JSON 转为 **UTF-8 字节数组**。
规范**不要求**编码前把 JSON 压缩、去掉空格换行等多余字符。
（带空格的格式化 JSON 也能直接编码，不是必须压缩成一行）

## 步骤2
对上面的字节数组，使用 **Base64-URL 算法编码**，并且**去掉末尾所有等号 =**。

## 步骤3
把载荷 JSON 转为 **UTF-8 字节数组**。
同样：编码前**不需要压缩、不需要去掉空格换行**。

## 步骤4
对载荷字节数组，使用 **Base64-URL 算法编码**，去掉末尾等号 =。

## 步骤5
按顺序拼接：
`编码后的头部` + `.` + `编码后的载荷`

因为是**无保护/无签名 JWT**，**第三部分签名直接为空**
所以最终格式是：
```
头部Base64URL.载荷Base64URL.
```
末尾**必须带一个点**，和你上一节学到的一致。

---

# 最后一段重点原文解释（非常重要）
> Validation of both the header and the payload (with respect to the presence of required claims and
> the correct use of each claim) must be performed before encoding

**在编码生成JWT之前，必须先校验头部和载荷：**
1. 是否包含了必要的声明字段
2. 每个字段格式、用法是否正确合规

也就是说：
**不能随便乱填JSON就直接编码，必须先校验合法，再生成JWT。**

---

# 极简总结（一步看懂）
制作无签名JWT流程：
1. 头部JSON → UTF8字节 → Base64URL（去=）
2. 载荷JSON → UTF8字节 → Base64URL（去=）
3. 拼接：`头部.载荷.`（末尾带点，签名为空）
4. **编码前必须先校验头部&载荷合法性**
5. 编码不要求压缩JSON，带空格也可以

# 完整代码 + 逐行中文解释

```javascript
// URL-safe variant of Base64
// 注释意思：这是 Base64 的【URL 安全版本】（就是 JWT 要求的 base64url）
function b64(str) {
  return new Buffer(str).toString('base64')  // 1. 先做普通 base64 编码
    .replace(/=/g, '')   // 2. 去掉所有末尾填充的 =
    .replace(/\+/g, '-') // 3. 把 + 换成 -
    .replace(/\//g, '_');// 4. 把 / 换成 _
}
```

### 这个 `b64` 函数干什么？
它实现了 **JWT 标准要求的 Base64-URL 编码规则**：
1. 普通 Base64
2. **去掉 =**
3. **+ → -**
4. **/ → _**

这就是书上说的 **URL-safe Base64**。

---

```javascript
function encode(h, p) {
  const headerEnc = b64(JSON.stringify(h));  // 头部 JSON → 字符串 → base64url
  const payloadEnc = b64(JSON.stringify(p));// 载荷 JSON → 字符串 → base64url
  return `${headerEnc}.${payloadEnc}`;       // 拼接：头部.载荷
}
```

### 这个 `encode` 函数干什么？
1. 把**头部 JSON** 变成字符串，再用 `b64()` 编码
2. 把**载荷 JSON** 变成字符串，再用 `b64()` 编码
3. 拼接成：
   ```
   头部编码.载荷编码
   ```
   （因为是无签名 JWT，所以**没有第三部分**）

# 3.5 解析无保护的 JWT（完整逐句解释）
## 核心意思
这一节教你：**把一段紧凑格式的无签名 JWT，还原成可读的 JSON 头部 + JSON 载荷**。

---

# 一、解析步骤（原文 4 步，100% 忠实翻译）
1. 找到 JWT 里**第一个点 `.`**，把点**前面**的字符串取出来（不包含点）。
2. 用 **Base64-URL** 算法解码这个字符串 → 得到 **JWT 头部（Header）**。
3. 取出第一步中，点**后面**的字符串。
4. 用 **Base64-URL** 算法解码 → 得到 **JWT 载荷（Payload）**。

最后一步补充：
解析出来的 JSON 字符串可以**格式化、加空格、换行**，让它更好看。

---

# 二、3.5.1 示例代码逐行解释
```javascript
// 解码无保护 JWT 的函数
function decode(jwt) {
  // 第一步：按点 . 分割 JWT，拿到头部编码、载荷编码
  const [headerB64, payloadB64] = jwt.split('.');

  // 注释：支持解析 URL 安全的 Base64 编码
  // 第二步：解码头部
  const headerStr = new Buffer(headerB64, 'base64').toString();
  // 第四步：解码载荷
  const payloadStr = new Buffer(payloadB64, 'base64').toString();

  // 转成 JSON 对象并返回
  return {
    header: JSON.parse(headerStr),
    payload: JSON.parse(payloadStr)
  };
}
```

## 代码每一步干了什么？
1. **`jwt.split('.')`**
   把 JWT 按点切开，直接拿到：
   - 第一段：编码后的头部
   - 第二段：编码后的载荷
   （无保护 JWT 没有第三段）

2. **`new Buffer(..., 'base64')`**
   Node.js 自带的方法，**自动支持 URL 安全的 Base64 解码**
   不用自己处理 `-` `_` `=`，系统直接搞定。

3. **`.toString()`**
   把解码后的二进制转成字符串 → 得到原始 JSON 字符串。

4. **`JSON.parse()`**
   把 JSON 字符串转成真正的 JSON 对象。
