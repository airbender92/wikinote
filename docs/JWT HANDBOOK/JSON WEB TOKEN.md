# JWT 到底是什么、怎么工作、为什么要用
JWT（JSON Web Token）是一种**轻量、安全、跨语言**的标准，用来在两个系统之间**安全传递身份和权限信息**。
它不像传统 Session 那样存在服务器，而是把数据**直接存在客户端**，靠**签名**保证没被篡改。

---

# JWT 的结构（三段式）
一个 JWT 长得像一串乱码，其实由三部分组成，用 `.` 分开：

**Header（头部）**
告诉系统：我用什么算法签名（比如 HS256、RS256），我是 JWT 类型。

**Payload（载荷）**
放真正要传的数据，比如用户ID、角色、过期时间、签发方等。
**注意：Payload 只是编码，不是加密！任何人都能解开看。**

**Signature（签名）**
服务器用密钥把前两段算一遍签名，用来证明：
- 数据没被改
- 确实是合法服务器发的

所以完整格式：
**Header.Payload.Signature**

---

# JWT 最核心的两个能力
## 1. JWS（签名）——防篡改
- 只用签名，不加密内容
- 别人能看，但**不能改**，一改签名就失效
- 适合：登录凭证、API 鉴权、用户身份传递

## 2. JWE（加密）——防偷看
- 内容完全加密，只有持有密钥的人能解开
- 适合：存敏感信息（隐私数据、密钥等）

---

# JWT 用来干嘛（最常见 4 个场景）
1. **用户登录认证**
   登录成功 → 服务器发 JWT → 前端每次请求带上 → 服务器验证 JWT 就知道你是谁。

2. **无状态会话**
   不用服务器存 Session，所有信息存在客户端，减轻服务器压力。

3. **单点登录（SSO）**
   一个账号登录，所有关联系统都能登录，OpenID Connect 就是用 JWT 实现的。

4. **API 授权**
   微服务、第三方接口之间互相信任，用 JWT 传递权限。

---

# JWT 为什么流行
- **超级小**：适合放在 URL、Header、Cookie 里
- **跨语言**：Java/Go/PHP/JS 全支持
- **无状态**：服务器不用存数据，水平扩展轻松
- **自己带数据**：不需要每次查库验身份

---

# 最危险的 5 个坑（手册重点强调）
1. **alg:none 攻击**
   把算法改成 none，去掉签名，弱校验系统会直接认。

2. **把 RSA 公钥当 HMAC 密钥用**
   算法被篡改，导致伪造令牌通过校验。

3. **密钥太弱**
   HS256 密钥太短会被暴力破解，必须 ≥32 位随机字符串。

4. **不验证过期、签发方、受众**
   别人偷到令牌就能随便用。

5. **XSS 窃取令牌**
   存在 localStorage 容易被偷，推荐存在 HttpOnly Cookie。

---

# 正确使用 JWT 的 7 条黄金规则
1. **永远强制指定算法**，不要让令牌自己决定用什么算法
2. **能⽤ RS256/ES256 就不用 HS256**（非对称更安全）
3. **必须验证 exp/iss/aud**（过期、签发者、接收方）
4. **令牌有效期设短**（几分钟～几小时）
5. **Cookie 开启 HttpOnly + Secure**
6. **不同业务用不同密钥**
7. **Payload 不要放敏感信息**（它不加密）

---

# 一句话总结
JWT 是一个**带签名的轻量 JSON 数据格式**，能安全地在系统间传递身份与权限，实现登录、鉴权、单点登录；
它简单、通用、无状态，但**安全取决于校验是否严格**，不做好校验会出现严重漏洞。

# 1.1 什么是 JSON Web Token（JWT）？完整解释
JSON Web Token（简称 JWT，读作 “jot”）是一种**紧凑、可打印、URL 安全**的标准格式，用来把一组**声明（claims）**打包成一串字符串，并且可以带上**签名**来确保内容没被篡改、来源可信。

它看起来像是一串无意义的乱码，比如：
```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiYWRtaW4iOnRydWV9.
TJVA95OrM7E2cBab30RMHrHDcEfxjoYZgeFONFh7HgQ
```
但这串字符串其实**结构非常清晰**，由三部分用 `.` 连接而成：
1. 头部（Header）
2. 载荷（Payload）
3. 签名（Signature）

把它解码后，会变成两段清晰的 JSON + 一段签名：

**头部（Header）**
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```
说明用什么算法签名、这是一个 JWT。

**载荷（Payload）**
```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "admin": true
}
```
这里放的就是**声明（claims）**——也就是关于某个用户、某个实体的描述、身份、权限等信息。

---

## 什么是“声明（claims）”？
claims 就是对某个人或某个数据的**陈述与定义**，比如：
- 你是谁（sub = 用户ID）
- 叫什么名字
- 是不是管理员
- 什么时候签发、什么时候过期

一部分 claims 是 JWT 标准规定的（比如 sub、iss、exp），另一部分可以由开发者自定义。
JWT 的价值就在于：**把这些信息标准化，让不同系统都能看懂、信任**。

---

## JWT 最关键的能力：签名 + 加密
JWT 不只是把信息打包，它还能做两件极其重要的事：

1. **签名（JWS）**
   用密钥给头部+载荷签名，确保：
   - 内容没被篡改
   - 确实是合法服务器发出来的

2. **加密（JWE）**
   把内容完全加密，只有持有密钥的人才能解开，保证隐私。

依靠 JWS（签名）和 JWE（加密），JWT 成为了一种**通用、安全、跨语言**的信息传递方案。

# 1.2 JWT 到底解决了什么问题？（完整解释）

JWT 的**主要目的**，是在**两个系统/两方之间安全传递“声明信息”**（比如你是谁、你有什么权限、什么时候过期）。

但它**真正最厉害、最核心的贡献**，是：
**它把这件事做成了一套简单、通用、标准化的格式。**

以前大家也需要传递身份信息，但问题很多：
- 每个公司自己写一套格式（很乱）
- 有的方案太复杂
- 有的不安全
- 不同系统之间不兼容

而 JWT 提供了一个**通用盒子**：
- 简单好用
- 可以选择加签名（验证没被改）
- 可以选择加密（不让别人看）
- 所有语言、所有系统都能看懂

这就是 JWT 解决的根本问题：
**把“安全传身份/权限信息”这件事，从各家自己乱做，变成统一、标准、简单的方案。**

后面第2章会讲具体用法，这里先列它能解决哪些实际问题：

1. **认证（Authentication）**
   你登录后，服务端给你发 JWT，以后带着它就证明“是你本人”。

2. **授权（Authorization）**
   JWT 里可以写你是普通用户还是管理员，服务端一看就知道你能访问什么。

3. **联合身份（Federated identity）**
   比如用微信、Google、GitHub 登录别的网站，JWT 让不同平台能互相信任你的身份。

4. **客户端会话（无状态会话）**
   不用服务器存 session，所有信息存在用户端，服务端压力更小、更容易扩容。

5. **客户端机密信息**
   可以安全地在客户端存一些需要保密的小数据。

# 2.1 客户端会话 / 无状态会话（完整解释）

所谓**无状态会话（stateless sessions）**，本质就是：
**把原本存在服务器里的用户会话数据，直接存在用户端（浏览器/APP）。**

它的关键在于：
**必须用签名（或加密）来保护这些数据，不让别人乱改。**

因为数据存在用户自己那边，**任何人都能改**，所以后端必须非常小心，不能直接信任客户端传来的数据。

---

## JWT 怎么实现无状态会话？
JWT 靠两个能力保护客户端数据：
1. **签名（JWS）**
   保证数据**没被篡改**，谁改谁失效。
2. **加密（JWE）**
   保证数据**别人看不见**。

大多数场景只需要**签名**就够了，比如：
- 用户 ID
- 角色
- 登录状态

这些内容就算被别人看到也没关系，只要**不能改**就行。

但有些数据**不能公开**，比如：
- 购物车内容
- 隐私信息
- 敏感配置

这种就需要**加密**，不然别人能偷看、爬取数据。

# 2.1.1 安全注意事项

## 2.1.1.1 签名剥离攻击（Signature Stripping）
这是最常见、最简单的 JWT 攻击方式：

攻击者把一个**正常的、带签名的 JWT**
1. 删掉最后一段签名
2. 把头部的算法改成 `"alg":"none"`

变成一个**没有签名、但能被错误校验通过**的令牌。

如果后端的 JWT 库**没有强制校验必须有签名**，就会直接信任这个被篡改的数据。

### 怎么防御？
非常简单：
**后端永远不接受无签名（alg:none）的 JWT。**

## 一、整段核心完整复述（非翻译，原意完整概括）
JWT 签名机制虽然能保证数据不可篡改，但本身存在安全隐患，**签名剥离攻击是针对带签名JWT最常见、最简单的攻击方式**。

标准JWT是三段式结构：头部Header.载荷Payload.签名Signature，三段各自独立编码、用点分隔。
攻击者可以直接删掉最后一段签名，再修改头部声明「这个JWT是无签名的」，把三段JWT改成两段无签名JWT。
如果开发者使用的JWT校验库写得很粗心、校验逻辑不严谨，就会**把这个被篡改过的无签名令牌当成合法令牌**。
此时攻击者可以随心所欲修改载荷里的任意信息（比如把自己改成管理员、修改用户ID、延长过期时间），完全绕过服务端的身份校验。
这个漏洞的防御方案非常简单：**校验服务端必须强制规定，无签名的JWT一律不合法、直接拒绝**。

---
## 二、攻击分步拆解（完全对应你上一张原理图）
### 正常流程（原理图逻辑）
服务端：原始数据Data + 密钥 → 生成完整三段签名JWT（Header.Payload.Signature）
公网传输 → 客户端收到完整令牌
客户端回传令牌 → 服务端校验签名合法 → 信任数据未被篡改

### 签名剥离攻击完整步骤
1. 攻击者截获一个**合法、完整、带签名的正常JWT**
2. 直接删除最后一段`Signature`签名，令牌变成两段：Header.Payload
3. 修改头部Header内容，把加密算法字段`alg`改成`none`（声明：本令牌无签名）
4. 随意篡改Payload载荷里的所有数据（比如把普通用户改成admin管理员）
5. 把这个**无签名、已篡改**的令牌发给服务端

### 漏洞触发条件
服务端JWT校验库逻辑粗心，**没有强制校验必须存在签名、没有禁止alg:none算法**，直接认为这个无签名令牌是合法有效的。

### 攻击最终危害
攻击者可以任意伪造身份、权限、过期时间，完全接管账号权限，服务端完全信任篡改后的数据。

---
## 三、结合原理图理解攻击本质
你上一张图的核心承诺是：
> 只要签名校验通过，服务端就信任数据未被修改

而签名剥离攻击，就是**攻击者直接删掉签名、破坏整个信任前提**：
- 原本原理图里攻击者`Can't modify the Data`（无法篡改数据），是因为有签名约束
- 攻击后签名消失，约束不存在，攻击者就可以随便改数据
- 校验库粗心放行，服务端就错误信任了被篡改的数据

---
## 四、官方给出的解决方案（原文重点）
防御极其简单，一行配置就能解决：
**JWT校验应用必须强制配置规则：任何无签名（alg:none）的JWT令牌，一律判定为无效，直接拒绝处理**。

额外补充最佳实践：
1. 校验时**强制指定允许的签名算法白名单**（比如只允许RS256/HS256），禁止none算法
2. 禁止信任令牌Header里声明的算法，服务端本地硬编码指定校验算法
3. 必须校验签名完整性，缺失签名直接拒绝

---
## 五、一句话精简总结
签名剥离攻击：删掉JWT签名→改算法为none→篡改数据→粗心校验库放行；
本质是破坏JWT「签名=数据可信」的底层逻辑；
防御：服务端永远不接受无签名JWT即可。

### 二、 攻击全流程解析
1. **攻击准备**
   攻击者先获取一个合法的、带完整签名的 JWT 令牌。
2. **核心攻击动作**
   -  **删签名**：直接移除令牌末尾的签名段（图中蓝色标记的完整签名部分被丢弃）。
   -  **改数据**：解析并修改 Payload 载荷，将普通管理员 `admin` 篡改为超级权限 `root`，用户 ID 也改为 `ROOT`。
   -  **改算法**（关键）：将头部 Header 中的算法字段 `alg` 改为 `none`（图中未直接显示头部，但操作逻辑一致），声明该令牌无签名。
3. **攻击结果**
   恶意用户将篡改后的**无签名 JWT** 发送给服务端。
4. **服务端漏洞**
   服务端配置错误，**未强制校验签名**，甚至没有拦截 `alg:none` 类型的令牌，最终错误地信任了篡改后的数据。

### 三、 攻击最终危害
服务端误判攻击者身份为 `user-id-ROOT`，并授予最高级别的 `root` 管理员权限，导致整个系统权限沦陷。

### 四、 核心防御方案
针对此攻击，必须在服务端实现以下 3 个强制校验：
1.  **强制校验签名**：绝不接收无签名（`alg:none`）的 JWT。
2.  **固定校验算法**：不信任令牌头部声明的算法，而是服务端硬编码指定算法（如 RS256）。
3.  **完整验签**：必须校验完整的 Header + Payload 签名，签名不匹配直接拒绝。

# 2.1.1.2 CSRF 跨站请求伪造攻击（完整复述+大白话深度解释）
我**不逐句翻译**，完整还原原文原意、讲透原理、结合JWT场景、攻击流程、危害、防御方案，完全对应你正在看的《JWT Handbook》章节。

---
## 一、原文完整核心概括（非逐句翻译，100%保留原意）
CSRF跨站请求伪造攻击的核心原理：
攻击者诱导**已经登录目标网站的用户浏览器**，在攻击者自己的恶意网站里，**偷偷向目标正规网站发送非自愿的恶意请求**。

浏览器有一个原生规则：只要请求目标是正规网站，浏览器会**自动带上该网站下所有有效的登录Cookie**，完全不受恶意网站控制。
如果正规网站没有做CSRF防护，就会把这个恶意请求，当成**登录用户本人主动发起的合法请求**执行，从而完成越权操作。

---
## 二、原文例子逐句拆解（就是最经典CSRF攻击模型）
攻击者在自己的恶意页面里，嵌入这么一行代码：
```html
<!-- 存在攻击者恶意网站里，不是目标网站！ -->
<img src="http://target.site.com/add-user?user=name&grant=admin">
```
攻击分步逻辑：
1.  用户**已经登录过正规网站 `target.site.com`**，浏览器保存了登录会话Cookie
2.  用户点开攻击者的恶意网页
3.  浏览器加载页面时，会自动解析`<img>`标签，**自动向正规网站地址发起请求**
4.  根据浏览器同源Cookie规则：请求发给`target.site.com`时，浏览器会**自动带上用户登录该网站的全部Cookie**
5.  正规网站没做CSRF防护 → 识别不到这是伪造请求 → 直接以用户身份执行「新增管理员账号」的高危操作

攻击者全程不需要偷用户Cookie、不需要知道用户账号密码，就能借用用户身份执行操作。

---
## 三、和JWT强相关的关键原文知识点
原文重点：
> JWTs, like any other client-side data, can be stored as cookies.
> JWT和所有客户端数据一样，**完全可以存在Cookie里**。

这句话是整个章节的核心：
只要你把**JWT令牌存在Cookie中**，你的网站就**天生存在CSRF攻击风险**！
原因和上面完全一致：
攻击者诱导浏览器发请求 → 浏览器自动带上存JWT的Cookie → 服务端拿到JWT、鉴权通过 → 执行恶意操作。

---
## 四、CSRF攻击核心3个特征（必记）
1.  攻击者**无法窃取、查看用户Cookie/JWT**，只是利用浏览器自动带Cookie的原生特性
2.  请求是**用户浏览器被动发起**，不是攻击者服务器直接发起
3.  危害：借用登录用户身份，执行增删改高危操作（改密码、加管理员、转账），不能读取用户数据

---
## 五、CSRF vs 你之前学的XSS 简单区分
| 攻击 | 核心原理 | 能不能偷令牌 | 风险 |
|------|----------|--------------|------|
| CSRF | 诱导浏览器代发请求，自动带Cookie | ❌ 不能偷看/窃取令牌 | 越权执行操作 |
| XSS | 注入JS脚本，直接读取客户端令牌 | ✅ 完全偷走令牌 | 账号彻底被盗 |

---
## 六、JWT场景下CSRF标准防御方案
1.  **同步令牌Token校验（最标准）**：页面返回CSRF Token，请求时手动放在请求头/表单里，Cookie自动带JWT，服务端校验两者匹配才放行
2.  **SameSite Cookie属性**：给存JWT的Cookie设置`SameSite=Strict/Lax`，禁止跨站请求自动携带Cookie
3.  **Referer/Origin请求头校验**：服务端校验请求来源域名是否合法
4.  不把JWT存在Cookie里，存在LocalStorage：可以**彻底规避CSRF风险**（但会新增XSS风险）

---
## 七、一句话精简总结
CSRF就是：攻击者骗登录用户浏览器，偷偷给正规网站发恶意请求；浏览器自动带上登录Cookie/JWT；网站没防护就当成用户本人操作执行；
而**JWT存在Cookie里就会受CSRF威胁**，这也是JWT存储方案必须权衡的核心安全问题。

# 完整解读：XSS跨站脚本攻击（结合图片+原文，不逐句翻译，完整讲透原理、流程、和JWT的关系）
## 一、原文核心完整概括
XSS跨站脚本攻击，是**攻击者向可信正规网站注入恶意JavaScript代码**；
注入的JS会在**正常用户的浏览器里执行**，直接读取浏览器本地存储的登录凭证（Cookie、LocalStorage里的JWT令牌）；
只要令牌没过期，攻击者就能拿到被盗的JWT，完全冒充用户身份访问接口、操作账号；
漏洞根源是：后端对用户输入的数据**没有做过滤、转义、消毒处理**（和SQL注入原理类似）。

最典型场景：网站评论区。用户发布评论，后端直接存入数据库、展示给所有人；不对评论做安全处理，攻击者就能在评论里插入`<script>`恶意脚本，其他任何人打开页面，脚本都会自动执行。

---
## 二、上图2种XSS攻击逐图拆解（完全对应手册内容）
### 上图：持久型XSS（Stored/Persistent XSS，存储型XSS，手册原文讲的就是这种）
标题：Figure 2.4 Persistent Cross Site Scripting 持久型跨站脚本攻击
1.  **Malicious User 恶意攻击者**：向存在漏洞的`Vulnerable Server 脆弱正规服务器`，提交恶意JS脚本；服务器直接保存脚本、不做过滤消毒。
2.  后续正常`User 用户`访问正规网站页面时，**漏洞服务器会把攻击者的恶意JS下发给用户浏览器**执行。
3.  浏览器里执行的恶意JS：从用户浏览器`LocalStorage / Cookie`中，窃取JWT令牌、登录凭证。
4.  窃取到的凭证，通过JS发送回**恶意攻击者**手里。
5.  攻击者拿到未过期JWT，直接冒充用户登录操作。

特点：脚本**永久存在服务器数据库**，所有访问页面的用户都会中招，危害最大，就是手册原文举例的评论区漏洞类型。

---
### 下图：反射型XSS（Reflected XSS）
1.  **Malicious Server 恶意服务器**：给正常用户发送一个精心构造的恶意链接。
2.  用户点击链接，请求发送给**漏洞正规服务器**。
3.  漏洞服务器不对参数处理，直接把链接里的恶意JS，原样返回给用户浏览器执行。
4.  执行的JS窃取本地JWT凭证，发送给攻击者。

特点：脚本**不存入服务器**，一次性、依附链接存在，需要诱导用户点击链接才会触发。

---
## 三、XSS 和 JWT 存储方案的致命关系（本章核心考点）
你之前学过JWT两种存储方式，在这里完全对应风险：
| JWT存储位置 | XSS攻击影响 | 同时受不受CSRF影响 |
|------------|------------|--------------------|
| LocalStorage / SessionStorage | ❌ **完全致命**：JS可以直接读取窃取，XSS一打就丢令牌 | ✅ 完全不受CSRF攻击影响 |
| HttpOnly Cookie | ✅ **安全防护**：JS无法读取Cookie内容，XSS无法窃取令牌 | ❌ 天生受CSRF跨站请求伪造影响 |

> 手册核心结论：JWT存在客户端，就必然面临XSS窃取风险；这是JWT无状态客户端存储，天生自带的安全权衡。

---
## 四、XSS vs 之前学的CSRF 核心对比（必记）
| 攻击类型 | 能不能偷JWT令牌 | 攻击原理 | 危害 |
|----------|----------------|----------|------|
| XSS跨站脚本 | ✅ 可以直接偷走完整令牌 | 注入JS在用户浏览器执行，读取本地数据 | 账号彻底被盗，攻击者完全接管账号 |
| CSRF跨站请求伪造 | ❌ 完全偷不到令牌 | 诱导浏览器代发请求，自动携带Cookie | 只能冒用身份执行操作，看不到令牌本身 |

---
## 五、防御方案（手册原文+最佳实践）
1.  后端对所有用户输入**强制HTML转义、消毒过滤**，禁止`<script>`等脚本标签渲染到页面（根源修复）
2.  JWT存入Cookie时，开启**HttpOnly属性**，禁止JS读取Cookie，抵御XSS窃取
3.  开启Content-Security-Policy(CSP)安全策略，禁止页面执行任意第三方JS
4.  不要把敏感令牌存在LocalStorage，降低XSS被盗风险

---
## 一句话精简总结
XSS就是攻击者给正规网站塞恶意JS，在用户浏览器里执行、偷走浏览器里的JWT登录令牌；
分为存储型（持久化）和反射型，是JWT客户端存储最大安全威胁；同时也是你之前纠结「JWT存Cookie还是LocalStorage」的核心安全原因。

# 2.1.2 客户端会话真的有用吗？（完整复述+深度解析）
这一段是全书的**核心决策章节**，它不讲攻击，而是讲**架构取舍**。我完全不逐句翻译，而是把这段文字的核心逻辑、权衡利弊、以及如何做决策，一次性讲透。

---

## 一、原文核心含义（100%还原原意）
任何技术方案都有优缺点，**客户端会话（Client-side Sessions，如 JWT）也不例外**。
它的核心争议在于：**当客户端存储的数据量过大时，传输成本可能会抵消掉“服务端无状态”带来的性能优势。**

文章给出了一个关键公式：**必须在「客户端数据体积」和「服务端数据库查询」之间找到平衡**。
这完全取决于你的应用数据模型。有些应用适配客户端会话，有些则完全不适合。最终的决策没有标准答案，必须通过**性能测试（Benchmarks）**来验证。

---

## 二、核心矛盾拆解：为什么会有“有用”和“没用”的争议？
这段文字本质上在讨论一个核心矛盾：**带宽（Bandwidth） vs 延迟（Latency）** 的交换。

### 1. ✅ 客户端会话的优势（Trade-off 1）
*   **优势**：**后端低延迟、高并发**。
    服务端不存 Session，不用查数据库、不用做分布式同步，响应极快。
*   **适用场景**：会话数据小（如只存 userID、role）、请求频率高的系统。

### 2. ❌ 客户端会话的劣势（Trade-off 2）
*   **劣势**：**带宽占用、请求体臃肿**。
    如果你的应用需要在 Session 里存大量数据（比如完整的用户配置、权限列表、购物车详情等），JWT 会变得非常大。
    每一次 HTTP 请求都要携带这一大坨数据，**会消耗大量网络带宽**。
*   **风险**：**带宽成本的增加，可能彻底抵消**掉服务端免查库带来的性能收益。
    比如：请求节省了 1ms 查库时间，但传输 JWT 多消耗了 10ms 网络时间，得不偿失。

---

## 三、关键决策问题（The Decision Framework）
作者给出了一套**非常实操的决策清单**，必须逐一回答，才能确定是否使用客户端会话：

### 1. 数据体量问题
*   **JWT 太大了吗？**
    如果 JWT 体积过大（比如超过 1KB/5KB），每次请求传输的开销巨大。
*   **带宽影响大吗？**
    在高并发场景下，巨大的 Header 或 Body 会造成显著网络开销。

### 2. 请求聚合问题
*   **小请求能合并吗？**
    如果前端发的是一堆微小请求，最好合并成一个大请求，减少往返传输。

### 3. 数据库依赖问题
*   **去掉数据库查询真的有必要吗？**
    如果即便把数据存在客户端，某些核心接口依然需要查库验证权限或数据，那客户端存储就失去了意义。

---

## 四、最终结论与最佳实践
作者没有给出绝对答案，而是给出了**工程化建议**：

1. **不要迷信“无状态”**：
    并不是所有数据都要存在 JWT 里。如果数据不常变、体积巨大，反而应该存在服务端，只存一个 ID 在 JWT 里。
2. **做基准测试（Run benchmarks）**：
    不要凭空想象，通过压力测试实测：
    *   用 JWT 模式 vs 用 Session 模式
    *   大 JWT 模式 vs 小 JWT + 查库模式
    的性能差异（QPS、响应时间、带宽占用）。

3. **动态权衡**：
    *   **轻量会话** ➡️ 首选 **客户端会话（JWT）**。
    *   **超大会话** ➡️ 回归 **服务端会话（Session + Cookie）**。

---

## 五、一句话总结
这一节在告诉你：**客户端会话（JWT）不是万能药。**
它能让服务端变快、好扩容，但**代价是消耗网络带宽**。
你需要根据你的**数据大小**和**请求频率**，做一次**带宽与延迟的博弈测算**，最终选择最合适的方案。

# 2.1.3 案例：用 JWT 实现客户端购物车（完整讲解）

## 一、案例整体介绍（完整复述+解释）
我们要做一个**简单的购物网站**。
用户的**购物车数据不存服务器，而是存在客户端浏览器里**。

在这个案例里，**一共用到 3 种 JWT**，各司其职：
1. **ID Token（身份令牌）**
   存放用户头像、昵称等资料，专门给前端页面展示用。
2. **Access Token（访问令牌）**
   用来调用后端 API，证明你有权限访问接口。
3. **Cart Token（购物车令牌）**
   **专门存购物车数据**，这是本案例的核心。

---

## 二、购物车 JWT 长什么样？
解码后的购物车数据：
```json
{
  "items": [0, 2, 4],
  "iat": 1493139659,
  "exp": 1493143259
}
```
- `items`：购物车里的商品 ID 列表
- `iat`：签发时间
- `exp`：过期时间

编码签名后的 JWT 就是一长串字符串，存在 Cookie 里。

---

## 三、前端怎么展示购物车？（前端代码解释）
```js
function populateCart() {
  const cartElem = $('#cart');
  cartElem.empty();
  const cartToken = Cookies.get('cart'); // 从 Cookie 拿购物车 JWT
  if(!cartToken) return;

  const cart = jwt_decode(cartToken).items; // 只解码，不验签名
  cart.forEach(itemId => {
    const name = items.find(i => i.id == itemId).name;
    cartElem.append(`<li>${name}</li>`);
  });
}
```

### 重点：
**前端只解码 JWT，不校验签名！**
展示数据而已，不需要验证是否被篡改。
**真正的校验全部交给后端做。**

---

## 四、后端怎么校验购物车是否被篡改？
```js
function cartValidator(req, res, next) {
  if(!req.cookies.cart) {
    req.cart = { items: [] };
  } else {
    try {
      // 后端真正校验签名！
      req.cart = jwt.verify(
        req.cookies.cart,
        process.env.AUTH0_CART_SECRET
      ).items;
    } catch(e) {
      req.cart = { items: [] }; // 验签失败 → 清空购物车
    }
  }
  next();
}
```

### 关键点：
- 前端不验签
- **后端必须严格验签**
- 验不过 → 直接清空购物车

---

## 五、添加商品时，后端重新生成购物车 JWT
```js
app.get('/protected/add_item', idValidator, cartValidator, (req, res) => {
  req.cart.items.push(parseInt(req.query.id));

  // 重新签名生成新 JWT
  const newCart = jwt.sign(req.cart, SECRET);

  // 写回 Cookie
  res.cookie('cart', newCart, { maxAge: 1000*60*60 });
  res.end();
});
```

每次加商品，后端都会：
1. 校验旧购物车 JWT
2. 添加新商品
3. **重新签名生成新 JWT**
4. 存回客户端 Cookie

---

## 六、最重要的安全设计：双令牌校验
这个案例最牛的地方在这里：

**/protected 开头的接口，必须经过两层校验：**
1. **Access Token**：校验你是否有权限调用接口（鉴权）
2. **Cart Token**：校验购物车数据没被篡改（数据完整性）

### 一句话总结：
- **Access Token = 你能不能进这个门**
- **Cart Token = 你手里的购物车有没有被动手脚**

---

## 七、登录流程（Auth0 第三方登录）
前端点击登录 → 跳转到 Auth0 登录页
登录成功后，Auth0 会返回两种令牌：
- access_token
- id_token

后端接收后，把它们存成 Cookie：
```js
app.post('/auth', (req, res) => {
  res.cookie('access_token', req.body.access_token, {
    httpOnly: true,  // 防 XSS
  });
  res.cookie('id_token', req.body.id_token);
  res.redirect('/');
});
```

# 2.2 联邦身份（完整中文解释）

## 第一段（什么是联邦身份）
**联邦身份系统**，让多个**彼此独立、没有关系**的系统之间，可以**共享认证和授权服务**。
换句话说：**用户的身份是中心化的**，不用在每个系统都注册一遍。

常见的联邦身份方案有两种：
- **SAML**
- **OpenID Connect**

有些公司专门提供中心化的登录与授权产品，
它们要么用上面这些标准，要么用自己的方案，
**其中很多都使用 JWT 来实现**。

---

## 第二段（通用授权流程，6 步）
不管用什么技术，联邦身份的**核心流程都是这 6 步**：

1. 用户想要访问某个服务器上的资源。
2. 用户没有凭证，于是服务器把用户**重定向到授权服务器**。
   授权服务器会引导用户去**身份提供商**那里登录。
3. 用户被跳转到**身份提供商的登录页面**（比如微信、GitHub、Facebook）。
4. 用户登录成功，跳转回授权服务器。
   授权服务器根据身份提供商给的信息，生成资源服务器需要的凭证。
5. 授权服务器把用户**跳回资源服务器**，并带上合法凭证。
6. 用户成功访问资源。

---

## 第三段（数据如何传递）
所有服务器之间传递的数据，**都是通过用户的浏览器跳转来传递**的（通常放在 URL 里）。
所以 **传输安全（TLS/HTTPS）** 和 **数据安全（签名/加密）** 非常重要。

授权服务器返回给用户的**凭证**，就可以用 **JWT** 来表示。

---

## 第四段（JWT 的作用）
如果授权服务器支持用第三方登录（微信、Facebook、Twitter 等），
那么它就是在给用户提供**统一的接口 + 统一的数据格式**，
这个统一格式就是 **JWT**。

---

## 最后一句（本节示例）
本节后面的例子，会用 **Auth0 作为授权服务器**，
支持三种登录方式：
- Twitter 登录
- Facebook 登录
- 普通用户名密码登录

# 2.2.1 访问令牌与刷新令牌（完整解释）

## 第一段（开门见山）
在各种联邦身份（第三方登录）方案里，你会大量见到两种令牌：
**访问令牌（Access Token）**
**刷新令牌（Refresh Token）**

这里简单讲它们是什么、在登录与授权里起什么作用。

---

## 第二段（背景：基于 OAuth2）
这两个概念都来自 **OAuth2 规范**。

OAuth2 的核心思想是：
**把“资源的所有权”和“访问权”分开**。
比如：你是资源主人，但你可以授权别人临时访问你的资源。

OAuth2 只规定**流程和用途**，不规定具体实现。
比如：
- 令牌长什么样、用什么格式，OAuth2 没强制规定
- 所以不同系统的 OAuth2 不一定能互相通用
- 但**用途和功能是统一的**

---

## 第三段：Access Token（访问令牌）
**访问令牌 = 给你开门的临时钥匙**

作用：
谁拿到它，就能**访问受保护的资源/接口**。

特点：
1. **生命周期很短**（几分钟、几小时）
2. 里面可以带过期时间
3. 可以附带额外信息（比如允许哪些IP访问）
4. 具体带什么信息，由各系统自己决定

---

## 第四段：Refresh Token（刷新令牌）
**刷新令牌 = 用来换新钥匙的“授权卡”**

作用：
当**访问令牌过期**后，
客户端可以拿着刷新令牌，去授权服务器**申请新的访问令牌**。

特点：
1. **生命周期很长**（几天、几周、几个月）
2. 只用来换新访问令牌，不能直接访问资源
3. 必须严格保管，不能泄露

---

# 最精简一句话总结
**Access Token 是短寿命的临时钥匙，用来访问资源；
Refresh Token 是长寿命的授权凭证，用来在旧钥匙过期后换新钥匙。**

# 这段文字的完整中文解释（原汁原味、不删减）

## 第一句
**把访问令牌和刷新令牌分开的最关键好处，是能让「访问令牌」变得非常容易校验。**

一个自带签名的访问令牌（比如签过名的 JWT），
**资源服务器自己就能直接验证真假，不需要再去询问授权服务器。**

---

## 第二句
**而刷新令牌则不一样，必须去授权服务器才能验证。**

通过把「验证访问令牌」和「查询授权服务器」分开，
系统可以获得：
- **更低的延迟**
- **更简单的访问逻辑**

---

## 第三句
**在令牌泄露时的安全性，靠两个手段保证：**
1. 让**访问令牌的有效期尽可能短**
2. 在访问令牌里加入额外检查（比如客户端校验）

---

## 第四句
**刷新令牌因为有效期很长，所以必须严防泄露。**

一旦刷新令牌泄露，服务器可能需要把它加入**黑名单**。
但因为访问令牌很短命，
只要刷新令牌被拉黑，**旧的访问令牌一过期，攻击者就再也无法续期**，资源就安全了。

---

## 最后一句（注意）
**访问令牌、刷新令牌这两个概念是 OAuth2 提出来的。**
OAuth 1.0 和 1.0a 里的 token 含义完全不一样，不要混淆。

---

# 最精简总结（一句吃透）
**访问令牌短寿命、自带签名、资源服务器自己就能验，速度快；
刷新令牌长寿命、必须找授权服务器验，更安全；
两者分开，既保证性能，又提高泄露后的安全性。**

# 2.2.2 JWTs and OAuth2 完整中文解释（**完全忠于原文、不省略、逐句翻译+通俗讲解**）
## 原文逐段对照解释
### 第一段
> Although OAuth2 makes no mention of the format of its tokens, JWTs are a good match for its requirements.
虽然 OAuth2 规范**完全没有规定令牌必须是什么格式**，但 JWT 格式非常契合 OAuth2 的设计需求。

> Signed JWTs make good access tokens, as they can encode all the necessary data to differentiate access levels to a resource, can carry an expiration date, and are signed to avoid validation queries against the authorization server.
带签名的 JWT 非常适合做**访问令牌（Access Token）**：
1. 可以编码权限等级、身份等所有必要信息
2. 自带过期时间字段
3. 自带数字签名
资源服务器**不需要再远程请求授权服务器验证令牌**，本地验签即可，速度更快。

> Several federated identity providers issue access tokens in JWT format.
现在绝大多数联邦身份服务商（比如 Auth0、微信登录、谷歌登录），下发的访问令牌**默认就是 JWT 格式**。

---

### 第二段
> JWTs may also be used for refresh tokens. There is less reason to use them for this purpose, though.
JWT 也可以用来做**刷新令牌（Refresh Token）**，但**完全没必要这么做**。

> As refresh tokens require access to the authorization server, most of the time a simple UUID will suffice, as there is no need for the token to carry a payload (it may be signed, though)
原因：
刷新令牌本身**必须去授权服务器后台查询验证**，本地验签没有任何意义；
它也不需要携带权限、过期时间等载荷数据；
所以用一个**单纯的随机 UUID 字符串**就足够了，完全不用做成 JWT。

---

# 核心总结（一句话吃透本节）
1. **Access Token 非常适合用 JWT**：自带权限、过期时间、签名，本地可验，不用查授权服务器，性能好。
2. **Refresh Token 不适合用 JWT**：它本来就要后台查询验证，不需要携带数据，用简单随机 ID 就行。

---
顺便帮你串联前面所有知识点：
OAuth2 分两种令牌 + JWT 特性完美匹配：
- Access Token → JWT（有数据、有签名、短时效、本地验签）
- Refresh Token → 纯随机字符串 UUID（无数据、长时效、必须后台验签）

# 2.2.3 JWT 与 OpenID Connect
**完整原文逐段通俗中文解释｜不省略、不删减、完全忠于原文逻辑**

---

## 第一段完整解释
OpenID Connect（简称 OIDC）是一套标准化规范。
它把 OAuth2 那些混乱、不统一、留给开发者自己随意实现的常用场景，全部整理成**一套统一、定义清晰、标准严谨的规范**。

OAuth2 本身有大量细节没有规定死，各家实现都不一样；
OpenID Connect 就是把这些缺失的空白全部补全、标准化。

具体来说：
1. 它定义了一套标准 API 和数据格式，来统一实现 OAuth2 的授权流程
2. 它在 OAuth2 授权流程之上，额外加了一层**身份认证（登录认证）能力**
3. 它内部多处数据格式直接选用 **JWT**
4. 其中最重要的：**ID Token（身份令牌）**，就是一种专门携带**已登录用户信息**的特殊 JWT

> 一句话理解：
OAuth2 只管「授权」（能不能访问资源）
OIDC 在 OAuth2 之上，补上了「登录认证」（你是谁），并且**ID Token 强制使用 JWT 格式**。

---

# 2.2.3.1 OpenID Connect 流程与 JWT
OpenID Connect 定义了三种授权登录流程，不同流程返回数据的方式不一样，
其中**ID Token 一律使用 JWT 格式**。

下面逐点完整解释原文三个流程：

## 1. 授权码模式 Authorization flow（最安全、最常用）
客户端先向授权接口 `/authorize` 请求**授权码 code**。
拿到授权码之后，再拿着 code 去令牌接口 `/token` 换令牌。
可以换到：
- ID Token（JWT 格式，用户身份信息）
- Access Token（访问资源令牌）
- Refresh Token（刷新令牌）

特点：令牌不在浏览器暴露，最安全，现在官方主推。

## 2. 隐式模式 Implicit flow
客户端直接从授权接口 `/authorize` **一次性直接拿到所有令牌**，不需要 code 换令牌。
请求里声明要 ID Token，就会直接返回 **JWT 格式的 ID Token**。

特点：简单但不安全，令牌直接暴露在浏览器地址栏，现在已经废弃不推荐。

## 3. 混合模式 Hybrid flow
客户端从 `/authorize` 接口**同时拿到授权码 + 一部分令牌**。
如果请求了 ID Token，会直接返回 JWT 格式 ID Token；
如果这一步没要 ID Token，后面也可以单独去 `/token` 接口再获取。

结合了授权码模式和隐式模式的特点。

1. OIDC = **标准化 OAuth2 + 登录认证能力**
2. OIDC 强制使用 **JWT 作为 ID Token 格式**（ID Token = 用户身份凭证）
3. OIDC 有三种登录流程：授权码、隐式、混合模式
4. 三种流程里，只要返回 ID Token，**一定是 JWT**

# 2.2.4.1 为 Node.js 应用配置 Auth0 Lock（完整中文解释）

## 一、配置 Auth0 库（原文代码 + 逐行解释）
下面是初始化 Auth0 认证库的代码：
```javascript
const auth0 = new window.auth0.WebAuth({
  domain: domain,           // 你的 Auth0 租户域名
  clientID: clientId,       // 你的应用在 Auth0 上的客户端 ID
  audience: 'app1.com/protected',  // 要访问的 API 受众
  scope: 'openid profile purchase',// 要获取的用户信息和权限
  responseType: 'id_token token',   // 要返回：ID令牌 + 访问令牌
  redirectUri: 'http://app1.com:3000/auth/', // 登录成功后跳转地址
  responseMode: 'form_post'         // 用 POST 表单把令牌发给后端
});
```

### 每一项含义（非常简单）
- **domain**：Auth0 给你的登录地址
- **clientID**：你的应用唯一标识
- **audience**：你要访问的受保护 API
- **scope**：你想获取用户的哪些信息（基本资料、权限等）
- **responseType**：要拿到 **ID 令牌** 和 **访问令牌**
- **redirectUri**：登录完跳回哪里
- **responseMode: form_post**：安全地把令牌传给后端
