# 一句话核心
**`alg: none` 攻击 = 攻击者把 JWT 头部算法改成 `none`，删掉签名，伪造身份/权限，让服务器直接跳过签名验证。**

---

# 完整攻击流程（超简单）
## 1. 正常合法 JWT
- 头部：`alg: HS256`
- 载荷：`role: user`
- 有**签名**
- 服务器验证签名 → 合法

## 2. 攻击者动手脚
1. 把头部算法改成 **`alg: "none"`**
2. 把载荷改成 **`role: "admin"`**
3. **删掉最后一段签名**
4. 拼接成：
   `头部.载荷.`（后面空的）

## 3. 旧版/劣质 JWT 库的漏洞
服务器拿到这个伪造 token：
- 读取头部 `alg: none`
- **以为“不需要验证签名”**
- 直接跳过验证，信任载荷内容
- **攻击者成功变成管理员！**

---

# 为什么这个漏洞能生效？（核心原因）
**早期很多 JWT 库完全信任头部里的 `alg` 字段**
你说用什么算法，它就用什么算法
你说 `none`，它就**不验证**

这不是 JWT 标准的漏洞，
是**库实现太蠢**。

---

# 现在怎么防御？（标准答案）
1. **直接禁止 `alg: none`**，遇到直接拒绝
2. **不在头部读取算法**，而是**代码里写死预期算法**（比如只允许 HS256）
3. 使用**现代、安全的 JWT 库**（已默认修复）

---

# 终极精简总结（你只记这个）
- **`alg: none` = 不需要签名**
- 攻击者利用这点**伪造权限、跳过验证**
- 漏洞根源：**库信任头部的算法**
- 防御：**禁用 none、写死算法、用新版库**

# 一句话核心（最关键）
**攻击者把 RS256 算法的 JWT 头部改成 HS256，
然后用公开的 RSA 公钥当 HS256 密钥重新签名，
服务器因为信任头部算法，把公钥当对称密钥验证，
结果**伪造的 token 被判定为合法**。**

# 完整攻击流程（超简单）
## 1. 正常情况
- 服务器用 **RSA 非对称** 签名
- **私钥签名** → 生成 alg=**RS256** 的 JWT
- **公钥验证** → 所有人都能拿到公钥
- 公钥公开是**正常且安全**的

## 2. 攻击者动手
1. 拿到**公开的 RSA 公钥**
2. 伪造一个 JWT：
   - 头部算法改成 **HS256**
   - 载荷改成 `role: admin`
3. **用 RSA 公钥当 HS256 的密钥**，给伪造的 token 签名

## 3. 服务器验证（漏洞触发）
服务器代码：
```js
jwtDecode(token, 公钥);
```

服务器做了什么蠢事？
1. 读取头部：`alg: HS256`
2. **把传入的“公钥”当成 HS256 对称密钥**
3. 用 HS256 验证签名
4. **验证通过！信任伪造的管理员权限**

---

# 为什么这个攻击能成功？（核心原因）
## 1. 库的 API 设计烂
```
jwtDecode(token, secretOrPublicKey)
```
一个参数**既可以当对称密钥，也可以当公钥**。

## 2. 库**完全信任头部的 alg**
你写 HS256 → 它就用对称密钥验证
你写 RS256 → 它就用公钥验证
**攻击者说什么，它信什么**。

## 3. RSA 公钥是公开的
攻击者可以**随便拿到**，并用它当 HS256 密钥签名。

---

# 攻击总结（3 行秒懂）
1. 正常：**RS256 + 私钥签名 + 公钥验证**
2. 伪造：**HS256 + 公钥当密钥签名**
3. 漏洞：服务器**用公钥当 HS256 密钥验证** → 伪造通过

---

# 怎么防御？（标准答案）
1. **验证 JWT 时，强制写死预期算法**
   不准用头部的 alg！
   ```js
   // 正确写法
   jwt.verify(token, key, { algorithms: ['RS256'] })
   ```
2. **不要使用混用密钥类型的模糊 API**
   对称、非对称分开处理
3. **严格校验 alg 字段**
   不允许从 RS256 变成 HS256

---

# 终极精简总结（你只记这个）
- **RS256 公钥是公开的**
- 攻击者把 alg 改成 **HS256**
- 用**公钥当 HS256 密钥**签名
- 服务器信任头部算法 → **伪造成功**
- 防御：**强制指定算法，不相信头部 alg**

# 8.1.3 Weak HMAC Keys（弱密钥问题）
**HS256 这类 HMAC 算法非常快，暴力破解也很快，
所以密钥不能像普通密码那样短，必须足够长、足够强。**

---

# 核心内容总结
1. **HMAC 和密码不一样**
   - 密码可以短一点，因为存密码时会用慢哈希（带盐），暴力破解很难
   - 但 JWT 里的 HMAC 是**高速算法**，验签/签名都很快
   → 这也意味着**暴力猜密钥也非常快**

2. **规范强制要求密钥长度**
   RFC 7518 明确规定：
   - HS256 的密钥**至少 256 位（32 字节）**
   - HS384 → 至少 384 位
   - HS512 → 至少 512 位

   简单说：
   **密钥长度 ≥ 哈希输出长度**

3. **人类可读密码基本都不合格**
   日常密码十几位、二十几位，远远达不到 32 字节的安全要求。
   用弱密钥，攻击者很容易通过暴力破解找到密钥，然后**伪造任意 JWT**。

4. **建议**
   - 要么用**足够长、随机的密钥**
   - 要么直接换成 **RS256 / ES256 这类非对称算法**，更安全、不用纠结密钥长度

# 核心一句话
**只加密不签名，或者嵌套 JWT 只验证外层，是完全错误的。加密≠防篡改，必须同时验证完整性。**

---

# 一、签名 vs 加密（先分清）
- **签名（JWS）**：防篡改，能读但不能改，改了就验证失败
- **加密（JWE）**：防泄露，别人读不懂，但不代表不能乱改
- 很多人误以为“加密了就安全了，别人改不了”，这是大错特错

---

# 二、为什么只加密不签名会有问题？
1. 很多加密算法**即使数据被篡改，解密也不会报错**，只是输出垃圾数据
2. 攻击者可以：
   - 不断修改加密后的 token
   - 解密后得到不同的垃圾结果
   - 利用这些结果触发系统漏洞
   - 甚至在某些场景下直接翻转布尔值（比如把 `admin: false` 改成 `true`）
3. 只要系统在解密后不做完整性校验，就可能被利用

---

# 三、嵌套 JWT（先签后加密）的正确用法
1. 正确流程：
   - 内层：**先做 JWS 签名**（保证防篡改）
   - 外层：**再做 JWE 加密**（保证内容不泄露）
2. 验证时：
   - 先解密外层 JWE
   - 再**验证内层 JWS 的签名**（必须！）
3. 很多错误实现：
   - 只解密外层，拿到数据就直接用
   - 完全不验证内层的签名 → 用了未受信任的数据，等于没安全措施

---

# 四、你这张图在说什么？


1. 左边输入一个 token，分两种情况：
   - **签名验证失败 → 直接判定为无效 token**
   - 签名验证成功，继续往下走
2. 中间：如果 token 里套了另一个 JWT（比如 `{"jwt": "...", "aud": "..."}`），必须再对里面的 token 做验证：
   - 内层签名验证成功 → 有效 token
   - 内层签名验证失败 → 无效 token

核心意思：**嵌套 JWT 必须层层验证，不能只看外层。**

---

# 五、JWA 规范的解决方式
- JWT 标准里的加密算法（比如 `AES-GCM`）都自带**认证加密**，解密时会自动校验完整性
- 所以用标准 JWE 算法，不用额外签名也能防篡改
- 但如果你用非标准加密算法，就必须：
  - 要么用**嵌套 JWT（先签后加密）**
  - 要么确保加密算法自带完整性校验

---

# 终极精简总结（你只记这3点）
1. **加密≠防篡改**，不要以为加密了就不会被改
2. **嵌套 JWT 必须层层验证**，外层解密后，内层签名也要验证
3. 用标准 JWE 算法自带完整性校验，否则必须先签名再加密

# 一句话核心
**无效曲线攻击，就是攻击者给你传一个“假的、不在标准曲线上”的公钥点，
如果你的库不校验这个点是否合法，就可能被攻击者利用，算出你的私钥。**

---

# 1. 椭圆曲线密码学的前提
EC 算法（比如 ES256）的安全，完全建立在**“我们只在一条固定、标准的曲线上做运算”**这个前提上。
- 所有公钥，都必须是**标准曲线上的有效点**
- 所有运算（点加、点乘），也必须在这条曲线上进行
- 这样才能保证离散对数问题的难度，私钥不会被算出来

---

# 2. 攻击的核心：传一个“假点”
攻击者构造一个**不在标准曲线上的点**，发给你：
- 这个点看起来像个公钥
- 但它不在你用的那条曲线上
- 如果你直接用这个点参与运算（比如验签），又不做校验
- 算法库在处理时，就会跑到**另一条弱曲线**上做计算
- 而弱曲线上的离散对数问题，**非常容易被破解**
- 攻击者就能通过多次这样的操作，算出你的私钥

---

# 3. 为什么会被攻破？
- 标准曲线（比如 P-256）的参数是精心设计的，安全强度极高
- 但攻击者构造的假点，会让运算跑到一条参数有缺陷的“弱曲线”上
- 在弱曲线上，私钥很容易被反推出来
- 问题根源不是 EC 算法本身不安全，而是**库实现没做输入校验**

---

# 4. 防御方法（规范里写得很清楚）
1. **必须校验公钥点的有效性**
   - 确认点的坐标满足标准曲线方程
   - 确认点的阶是正确的
   - 确认点不是无穷远点
2. **必须校验私钥的有效性**
   - 确保私钥在正确的数值范围内
3. 现代、高质量的 EC 库（比如 OpenSSL、Node.js 内置的 crypto 模块）**都会自动做这些校验**，不会直接使用未验证的点

---

# 终极精简总结（你只记这3点）
1. **EC 算法安全的前提是：所有运算都在标准曲线上进行**
2. 无效曲线攻击 = 用不在标准曲线上的假点，诱导运算跑到弱曲线上，从而泄露私钥
3. 防御 = 用高质量库 + 强制校验公钥点是否在标准曲线上

# 8.1.6 替换攻击（Substitution Attacks）
## 一句话核心
**攻击者把本来给别的系统、别的用途、别的用户的合法 JWT，
偷拿来用在你的系统上，骗过校验，拿到权限。**

---

## 1. 不同接收者攻击（Different Recipient）
- 场景：
  - 系统 A 给用户发了一个合法 JWT
  - 系统 B 也用同一套密钥/算法
- 攻击：
  - 攻击者把给 A 的 token 直接发给 B
  - B 只验签名合法，**不看这个 token 是不是发给自己的**
  - 结果：攻击者在 B 系统冒充合法用户

## 2. 跨 JWT 攻击（Cross-JWT）
- 同一个系统里，有多种 token：
  - 登录 token
  - 刷新 token
  - 邮箱验证 token
- 攻击：
  - 攻击者拿**刷新 token**去当**登录 token**用
  - 或者拿**验证邮箱 token**去改密码
  - 系统只验签名有效，**不校验 token 用途**
  - 直接越权

---

# 怎么防御（书里的重点）
1. **必须校验 `aud`（受众）**
   token 是发给谁的，只能谁用
2. **必须校验 `iss`（签发者）**
   只信任指定签发方
3. **必须校验 `azp` / 用途字段**
   刷新 token 不能当登录 token 用
4. 不同用途的 token，**最好用不同密钥 / 不同算法**

---

# 终极精简总结
- **替换攻击 = 偷别人的合法 JWT，在别的地方用**
- 关键漏洞：只验签名，不验**给谁用、用来干嘛**
- 防御：校验 `iss` + `aud` + token 用途

---

# 一句话核心
**这是「同接收方、跨服务」的 JWT 替换攻击：攻击者拿着 A 服务的合法 Token，去骗同一个公司里校验逻辑有漏洞的 B 服务，从而越权操作。**

---

# 先看懂这张图在讲什么


- **Mail Server**：用私钥签名，给用户 Joe 发一个 `role: write` 的合法 Token，只能用在邮件服务上。
- **Finance Ops**：财务系统，用的是同一个公钥验签。
- **漏洞**：财务系统只验了签名合法，没校验这个 Token 是给哪个服务的（比如没校验 `aud` 受众字段）。
- **攻击**：Joe 把邮件服务的 Token 发给财务系统，财务系统因为验签通过，就把他当成了有写权限的用户，允许他操作财务数据。

---

# 再看文字里的例子（和图是同一个逻辑）
- Token 是发给 `cool-company/user-database` 的，写着 `aud: "cool-company/user-database"`
- 但另一个服务 `cool-company/item-database`，校验 `aud` 时犯了个错误：
  它不是**完全匹配**，而是只要包含 `cool-company` 字符串就通过。
- 结果：`aud: "cool-company/user-database"` 也通过了这个“包含校验”
- 攻击者拿着给用户数据库的 Token，就能在物品数据库里拿到 `write` 权限，越权操作。

---

# 攻击的关键漏洞点
1.  **跨服务用同一套密钥**：不同服务用同一个私钥/公钥签名验签，导致 Token 可以跨服务复用。
2.  **`aud` 校验逻辑不严谨**：不是精确匹配，而是用了模糊匹配（包含、前缀、后缀），导致其他服务的 Token 也能通过。
3.  **没校验 Token 的用途/权限范围**：只看了 `role: write`，没看这个 `write` 是哪个服务的。

---

# 防御方法（标准写法）
1.  **`aud` 必须做精确匹配**：Token 的 `aud` 必须和当前服务的 ID 完全一致，不能用模糊匹配。
2.  **不同服务用不同密钥**：邮件、财务、用户数据库，各用各的密钥，避免跨服务复用。
3.  **校验 `scope`/权限字段**：Token 里的权限要明确指定适用范围，比如 `scope: "mail:write"`，而不是简单的 `write`。
4.  **限制 Token 的生命周期**：短时效 Token 即使泄露了，也没法长期被利用。

---

# 终极精简总结
- **同接收方跨服务攻击**：同一公司内，用 A 服务的合法 Token 去骗 B 服务。
- 漏洞根源：跨服务共享密钥 + `aud` 校验不严谨。
- 防御：精确匹配 `aud`、服务间密钥隔离、细化权限字段。

# 一句话核心
**JWT 的安全不是靠算法，而是靠严格的校验和正确的配置。所有最佳实践，本质上都是为了堵住前面提到的各种攻击漏洞。**

---

# 逐条拆解（对应每一条实践）

## 8.2.1 永远做算法校验（核心中的核心）
- 目的：防止 `alg: none` 攻击、RS256 公钥当 HS256 密钥的攻击
- 做法：
  - 不要信任头部的 `alg` 字段！
  - 代码里**写死允许的算法**（比如只允许 `RS256`）
  - 用库的 `algorithms` 参数强制指定，不要用“自动选择”模式
- 示例（Node.js）：
  ```js
  // 正确：强制只接受 RS256
  jwt.verify(token, publicKey, { algorithms: ['RS256'] });

  // 错误：用头部 alg 自动选择，会被攻击利用
  jwt.verify(token, publicKey);
  ```

## 8.2.2 选择合适的算法
- 不是越复杂越好，而是要匹配你的场景：
  - 单服务、前后端分离：HS256 够用，但密钥要够强
  - 多服务、联邦身份、对外提供接口：用 RS256/ES256（非对称）
- 原则：即使验签通过，如果算法不是你预期的，也要拒绝

## 8.2.3 嵌套 Token 必须层层验证
- 场景：JWT 里再套一个 JWT（比如先签后加密）
- 错误做法：只解密外层，不验证内层的签名
- 正确做法：
  1.  解密外层 JWE
  2.  拿到内层 JWS
  3.  再验内层的签名和所有声明
- 否则等于用了未受信任的数据，完全白搭

## 8.2.4 永远校验密码学输入
- 目的：防止无效椭圆曲线这类攻击
- 做法：
  - 用成熟的密码学库（不要自己实现）
  - 确保库会自动校验公钥/私钥的有效性
  - 比如 EC 公钥必须是标准曲线上的有效点
- 不做校验，可能直接泄露私钥

## 8.2.5 使用强密钥
- 对 HMAC（HS256/HS384/HS512）：
  - 密钥长度 ≥ 哈希输出长度（HS256 至少 32 字节）
  - 必须用加密安全的随机数生成（`crypto/rand` 这类）
  - 不能用人类可读的密码，很容易被暴力破解
- 对非对称算法（RSA/EC）：
  - 密钥长度要够（RSA 至少 2048 位，EC 用 P-256 及以上）

## 8.2.6 校验所有关键声明
- 只验签名不够，必须同时校验内容：
  - `iss`（签发者）：只信任你预期的签发方
  - `aud`（受众）：Token 是发给谁的，只能谁用（**必须精确匹配**，不能模糊）
  - `exp`（过期时间）：过期的 Token 直接拒绝
  - `nbf`（生效时间）：没到时间的 Token 也拒绝
  - `sub`（主体）：确认用户身份
- 很多替换攻击，就是因为没校验这些声明才成功的

## 8.2.7 用 `typ` 声明区分 Token 类型
- `typ` 字段通常是 `JWT`，但也可以自定义
- 场景：系统里有多种 Token（登录、刷新、验证）
- 做法：
  - 给不同用途的 Token 设置不同的 `typ`，比如 `access`、`refresh`
  - 校验时只接受预期的类型，防止拿刷新 Token 当登录 Token 用

## 8.2.8 为不同 Token 用不同的校验规则
- 核心思想：**Token 之间要隔离，不能互相替代**
- 具体做法：
  - 不同子系统用不同的签名密钥（避免跨服务复用）
  - 细化声明：比如 `iss` 用服务的 URL，而不是公司名
  - 给不同用途的 Token 加自定义声明，明确适用范围

# 8.3 结论（一句话）
**JWT 安全的关键，是用成熟的库、严格的校验、合理的配置，而不是靠算法本身。永远不要自己实现密码学相关的逻辑。**
