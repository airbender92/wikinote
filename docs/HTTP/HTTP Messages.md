## 1. HTTP 消息的流向

消息在客户端、服务器和代理之间流动时，有几种方向性的术语：

- **inbound / outbound**（入站 / 出站）  
  消息从客户端**发往源服务器**的方向称为 **入站**；从源服务器**返回客户端**的方向称为 **出站**。

- **upstream / downstream**（上游 / 下游）  
  无论请求还是响应，所有消息都沿着 **downstream**（下游）方向流动。  
  - 发送消息的一方是上游（upstream）。  
  - 接收消息的一方是下游（downstream）。  
  例如：客户端 → 代理1 → 代理2 → 服务器，对于请求来说，代理1 是代理2 的上游；对于响应来说，代理2 是代理1 的上游。

---

## 2. HTTP 消息的三个组成部分

每个 HTTP 消息（无论是请求还是响应）都由三个部分构成：

| 部分 | 作用 | 示例 |
|------|------|------|
| **起始行**（start line） | 描述要做什么（请求）或发生了什么（响应） | `GET /index.html HTTP/1.1` |
| **首部字段**（headers） | 一组 key-value 元数据，以空行结束 | `Host: www.example.com`<br>`Content-Type: text/html` |
| **主体**（body / entity body） | 可选数据，可以是文本或二进制（如图片、视频） | HTML 内容、图片二进制数据 |

- 起始行和首部都是纯文本（ASCII），每行以 **CRLF**（`\r\n`）结尾。  
- 首部结束后有一个**空行**（CRLF），之后是主体内容。

---

## 3. 请求消息 vs. 响应消息

| 对比项 | 请求消息（Request） | 响应消息（Response） |
|--------|----------------|----------------|
| **起始行格式** | `方法 请求URI 版本`<br>例如：`GET /index.html HTTP/1.1` | `版本 状态码 原因短语`<br>例如：`HTTP/1.1 200 OK` |
| **示例** | `POST /login HTTP/1.1` | `HTTP/1.1 404 Not Found` |
| **主体** | 可选，如表单数据、上传文件 | 可选，如网页内容、图片 |

---

## 4. 请求消息支持的方法（Methods）

HTTP/1.1 定义了一些常用的请求方法：

| 方法 | 含义 | 是否有主体 |
|------|------|------------|
| `GET` | 获取资源 | 否 |
| `HEAD` | 只获取首部（不返回主体） | 否 |
| `POST` | 向服务器提交数据（如表单） | 是 |
| `PUT` | 将主体存储到指定 URI | 是 |
| `DELETE` | 删除指定资源 | 否 |
| `TRACE` | 追踪请求经过的代理链 | 否 |
| `OPTIONS` | 查询服务器支持的方法 | 否 |

> 扩展方法（如 `LOCK`、`COPY`）用于 WebDAV 等扩展。

---

## 5. 响应消息中的状态码（Status Codes）

状态码是三位数字，分为五类：

| 范围 | 类别 | 示例 | 含义 |
|------|------|------|------|
| 100–199 | 信息类 | `100 Continue` | 初步收到请求，客户端可继续 |
| 200–299 | 成功类 | `200 OK` | 请求成功 |
| 300–399 | 重定向类 | `301 Moved Permanently` | 资源已永久移动 |
| 400–499 | 客户端错误 | `404 Not Found` | 资源不存在 |
| 500–599 | 服务器错误 | `500 Internal Server Error` | 服务器内部错误 |

> 状态码常附带一个“原因短语”（reason phrase），如 `200 OK` 中的 `OK`，但程序只依赖数字码。

---

## 6. HTTP 首部（Headers）的作用

首部是消息的“元数据”，按功能可分为几类：

| 类别 | 作用 | 示例 |
|------|------|------|
| **通用首部** | 请求和响应都可出现 | `Date`、`Connection`、`Cache-Control` |
| **请求首部** | 客户端告知服务器偏好 | `Accept`、`Accept-Language`、`If-Modified-Since` |
| **响应首部** | 服务器告知客户端信息 | `Server`、`Age`、`WWW-Authenticate` |
| **实体首部** | 描述主体内容 | `Content-Type`、`Content-Length`、`Content-Encoding` |
| **扩展首部** | 非标准但常用的首部 | `Cookie`、`Set-Cookie`、`X-Forwarded-For` |

通过首部，HTTP 能够支持内容协商、缓存控制、连接管理、身份认证等高级功能。

## 二、请求消息（Request Message）的格式

### 起始行格式
```

```
例如：`GET /specials/saw-blade.gif HTTP/1.0`

| 组成部分 | 含义 | 示例 |
|----------|------|------|
| `method` | 客户端希望服务器执行的动作 | `GET`, `POST`, `HEAD` 等 |
| `request-URL` | 请求的资源路径（可以是完整 URL 或路径部分） | `/specials/saw-blade.gif` 或 `http://www.example.com/index.html` |
| `version` | 使用的 HTTP 版本 | `HTTP/1.0`, `HTTP/1.1` |

> 如果直接与服务器通信，通常只写路径（如 `/specials/saw-blade.gif`），服务器会根据自己的主机名补全。

### 示例（图 3‑5 上部）
```

Accept: text/*
Host: www.joes-hardware.com

```
- 第一行是起始行：`GET` 方法、请求 URL、版本 `HTTP/1.1`。
- 之后两行是请求首部：`Accept` 和 `Host`。
- 此请求没有实体主体（没有空行后的内容）。

## 三、响应消息（Response Message）的格式

### 起始行格式
```
<version> <status-code> <reason-phrase>
```
例如：`HTTP/1.0 200 OK`

| 组成部分 | 含义 | 示例 |
|----------|------|------|
| `version` | 服务器使用的 HTTP 版本 | `HTTP/1.0` |
| `status-code` | 三位数字，表示请求结果（成功、重定向、错误等） | `200`, `404`, `500` |
| `reason-phrase` | 给人读的状态描述，不用于程序判断 | `OK`, `Not Found` |

> **注意**：`reason-phrase` 只是可读文本，不同服务器可能返回不同的短语（如 `200 OK` 或 `200 Document attached`），程序应只依赖数字状态码。

```
Content-length: 19

Hi! I'm a message!
```
- 第一行：状态行（200 OK）。
- 两行响应首部：说明主体是纯文本，长度 19 字节。

| **空行** | 必须存在（即使没有首部） | 必须存在 | 表示首部结束，主体开始 |
| **主体** | 可选，如表单数据、上传文件 | 可选，如 HTML、图片、JSON | 可以是任意二进制数据 |

- 规范要求：首部结束后必须有一个空行（CRLF）。

## 一、起始行（Start Line）

所有 HTTP 消息都以起始行开头：
- **请求消息的起始行**（请求行）说明**要做什么**。
- **响应消息的起始行**（状态行）说明**发生了什么**。

### 1. 请求行格式
```
<method> <request-URL> <version>
```
- `method`：客户端希望服务器执行的动作（如 `GET`、`POST`）。
- `request-URL`：请求的资源路径（可以是完整 URL 或路径部分）。
- `version`：客户端使用的 HTTP 版本（如 `HTTP/1.1`）。

> 在 HTTP/1.0 之前，请求行可以不包含版本号。

### 2. 状态行格式
```
<version> <status-code> <reason-phrase>
```
- `version`：服务器使用的 HTTP 版本。
- `status-code`：三位数字的结果代码（如 `200`、`404`）。
- `reason-phrase`：给人读的文本描述（如 `OK`、`Not Found`）。

> 在 HTTP/1.0 之前，响应可以不包含状态行。

---

## 二、方法（Methods）

方法告诉服务器要执行什么操作。HTTP/1.1 定义了一些常见方法：

| 方法 | 作用 | 是否有请求主体 |
|------|------|----------------|
| `GET` | 获取资源 | 否 |
| `HEAD` | 只获取首部（不返回主体） | 否 |
| `POST` | 向服务器提交数据（如表单） | 是 |
| `PUT` | 将主体存储到指定 URI | 是 |
| `DELETE` | 删除指定资源 | 否 |
| `TRACE` | 追踪请求经过的代理链 | 否 |
| `OPTIONS` | 查询服务器支持的方法 | 否 |

> 不是所有服务器都实现全部方法。HTTP 允许扩展，额外的方法称为**扩展方法**（如 `LOCK`、`COPY`）。

---

## 三、状态码（Status Codes）

状态码是三位数字，告诉客户端请求的结果。它们按首位数字分为五类：

| 范围 | 类别 | 含义 |
|------|------|------|
| 100–199 | 信息类 | 请求已接收，继续处理 |
| 200–299 | 成功类 | 请求成功完成 |
| 300–399 | 重定向类 | 资源已移动，需要进一步操作 |
| 400–499 | 客户端错误 | 请求有误（如 `404 Not Found`） |
| 500–599 | 服务器错误 | 服务器内部出错（如 `500 Internal Server Error`） |

> 常见的状态码示例：
- `200 OK` – 成功
- `401 Unauthorized` – 需要认证
- `404 Not Found` – 资源不存在

> 如果收到未定义的状态码（如 `515`），应将其视为所属大类（`5xx`）处理，即服务器错误。

---

## 四、原因短语（Reason Phrase）

- 状态码的数字部分供程序使用，**原因短语仅供人类阅读**。
- 不同服务器可能对同一状态码使用不同的短语（如 `200 OK` 与 `200 Document attached`），程序应**只依赖数字状态码**。

---

## 五、版本号（Version Numbers）

- 格式：`HTTP/<major>.<minor>`，如 `HTTP/1.1`。
- 版本号告知对方自己支持的最高 HTTP 版本。
- **版本号不是小数**，比较时要逐段比较数字：
  - `HTTP/2.22` > `HTTP/2.3`，因为 `22` > `3`。
- 注意：HTTP/1.0 应用看到 `HTTP/1.1` 响应时，可能误解为“这是一个 1.1 的响应”，实际上那只是响应方使用的版本号。

---

## 总结表：请求行 vs 状态行

| 对比项 | 请求行 | 状态行 |
|--------|--------|--------|
| **位置** | 请求消息的第一行 | 响应消息的第一行 |
| **格式** | `<方法> <URL> <版本>` | `<版本> <状态码> <原因短语>` |
| **示例** | `GET /index.html HTTP/1.1` | `HTTP/1.0 200 OK` |
| **目的** | 告诉服务器做什么 | 告诉客户端发生了什么 |

## 一、HTTP 首部（Headers）

首部位于起始行之后，可以出现**零个、一个或多个**。它们以“名: 值”对的形式提供额外信息，每行以 CRLF（`\r\n`）结尾。

### 首部的分类

| 类别 | 含义 | 出现位置 |
|------|------|----------|
| **通用首部（General headers）** | 请求和响应中都可以出现 | 双向 |
| **请求首部（Request headers）** | 提供关于请求的更多信息 | 仅请求 |
| **响应首部（Response headers）** | 提供关于响应的更多信息 | 仅响应 |
| **实体首部（Entity headers）** | 描述主体的大小、内容类型或资源本身 | 双向（取决于消息） |
| **扩展首部（Extension headers）** | 未在规范中定义的新首部，由应用自行定义 | 依情况而定 |

### 首部续行（Header continuation lines）

如果一个首部值很长，可以拆成多行，每个续行前面至少加一个空格或制表符（Tab）。

示例：
```
HTTP/1.0 200 OK
Content-Type: image/gif
Content-Length: 8572
Server: Test Server
  Version 1.0
```
- `Server` 首部的完整值是 `Test Server Version 1.0`。
- 第二行开头的空格表示它是上一行的续行。

---

## 二、实体主体（Entity Body）

- 是 HTTP 消息的**第三部分**（可选）。
- **有效载荷**：HTTP 真正要传输的数据。
- 可以是任意类型：文本、HTML、图片、视频、软件、信用卡交易信息、电子邮件等。

> 如果消息没有主体，则首部结束后直接跟一个空行（CRLF），没有后续数据。

---

## 三、HTTP/0.9 消息（早期版本）

HTTP/0.9 是一个非常简单的早期协议（1991 年左右），现今仍有极少部分旧应用在使用。

### HTTP/0.9 的特点

| 特征 | HTTP/0.9 | 现代 HTTP（≥1.0） |
|------|----------|-------------------|
| **请求格式** | `方法 请求URL`（无版本号） | `方法 请求URL 版本` |
| **响应格式** | 只有实体主体（无状态行、无首部） | 状态行 + 首部 + 可选主体 |
| **支持首部** | ❌ 不支持 | ✅ 支持 |
| **状态码** | ❌ 无 | ✅ 有（如 200 OK） |
| **版本号** | ❌ 无 | ✅ 有（如 HTTP/1.0、HTTP/1.1） |

### 示例（图 3-6 描述）

请求（客户端 → 服务器）：
```
GET /specials/saw-blade.gif
```
响应（服务器 → 客户端）：
```
[图片的二进制数据]
```
- 没有版本号、没有状态码、没有首部。

### 局限性

- 无法支持内容协商、缓存控制、持久连接等现代 HTTP 特性。
- **应用开发者仍需了解它**，以便与极少数遗留系统兼容。

---

## 总结表：HTTP 消息的三个部分

| 部分 | 内容 | 是否必需 |
|------|------|----------|
| **起始行** | 请求行或状态行 | 是 |
| **首部** | 零个或多个 `名: 值` 行，以空行结束 | 是（即使为空，也要有空行） |
| **主体** | 任意数据（文本或二进制） | 否 |

## 一、Safe Methods（安全方法）

HTTP 定义了一组**安全方法**：`GET` 和 `HEAD`。

- **含义**：安全方法不会在服务器上引起动作（即不会产生副作用）。
- **例子**：在线购物时点击“提交购买”按钮发送的是 `POST` 请求，它会触发扣款等操作，因此 `POST` 是**不安全**的方法。
- **注意**：安全方法不保证绝对没有动作（由开发者负责）。安全方法的定义主要是为了让浏览器能在用户即将发送不安全请求时弹出警告（例如“您即将提交订单，是否继续？”）。

---

## 二、常见 HTTP 方法详解

| 方法 | 作用 | 是否有请求主体 | 是否安全 | 说明 |
|------|------|----------------|----------|------|
| `GET` | 获取资源 | 否 | ✅ 安全 | HTTP/1.1 强制要求实现 |
| `HEAD` | 只获取首部，不返回实体主体 | 否 | ✅ 安全 | 可用于检查资源是否存在、是否被修改 |
| `PUT` | 将请求主体存储到指定 URI（创建或替换） | 是 | ❌ 不安全 | 通常需要密码认证 |
| `POST` | 向服务器提交数据（如表单） | 是 | ❌ 不安全 | 数据通常交给网关程序处理 |
| `TRACE` | 追踪请求经过的代理链，回显收到的请求 | 否 | 安全（只读） | 用于诊断，响应主体包含原始请求 |
| `OPTIONS` | 查询服务器支持的 HTTP 方法 | 否 | 安全（只读） | 可用于整个服务器或特定资源 |
| `DELETE` | 删除指定 URI 的资源 | 否 | ❌ 不安全 | 服务器可以不执行删除（由实现决定） |

---

## 三、各方法的补充说明

### 1. `GET`
- 最常用的方法。
- HTTP/1.1 要求所有服务器必须实现。

### 2. `HEAD`
- 与 `GET` 完全相同，但服务器**不返回实体主体**。
- 用途：
  - 获取资源类型（`Content-Type`）
  - 检查资源是否存在（看状态码）
  - 检查资源是否被修改（看 `Last-Modified` / `ETag`）
- HTTP/1.1 要求服务器必须实现。

### 3. `PUT`
- 语义：将请求主体存储到指定 URI。
  - 如果 URI 不存在 → 创建新资源
  - 如果 URI 已存在 → 替换整个资源
- 通常需要认证（防止随意修改/删除）。

### 4. `POST`
- 设计用于向服务器发送输入数据。
- 常用于 HTML 表单提交。
- 与 `PUT` 的区别：
  - `POST` 用于提交数据给服务器处理（可能产生多个资源或触发动作）
  - `PUT` 用于在指定 URI 上存放/替换资源（幂等）

### 5. `TRACE`
- 用于诊断。服务器收到 `TRACE` 请求后，会在响应主体中原样返回收到的请求消息。
- 用途：查看请求在经过代理、网关等中间节点后是否被修改。
- 限制：请求不能包含主体。
- 缺点：假设所有方法被中间节点同等对待，但实际并不总是如此（例如代理可能只缓存 `GET`）。

### 6. `OPTIONS`
- 查询服务器支持哪些方法。
- 可以针对整个服务器（`OPTIONS * HTTP/1.1`），也可以针对特定资源（`OPTIONS /index.html HTTP/1.1`）。
- 响应中包含 `Allow` 首部，列出支持的方法。

### 7. `DELETE`
- 请求服务器删除指定资源。
- 服务器可以忽略该请求（HTTP 规范允许），因此客户端不保证删除一定会执行。

---

## 四、Extension Methods（扩展方法）

HTTP 设计为可扩展，允许定义新的方法。

常见扩展方法（来自 WebDAV，见第 19 章）：

| 方法 | 作用 |
|------|------|
| `LOCK` | 锁定资源（防止多人同时编辑） |
| `MKCOL` | 创建集合（如目录） |
| `COPY` | 复制资源 |
| `MOVE` | 移动资源 |

### 处理未知扩展方法的原则
- 代理应尽量转发未知方法的请求，避免破坏端到端行为。
- 如果无法处理，应返回 `501 Not Implemented`。
- 黄金法则：**保守发送，宽容接收**（be conservative in what you send, be liberal in what you accept）。

---

## 总结表：方法与常见用途

| 方法 | 典型场景 |
|------|----------|
| `GET` | 浏览网页、下载资源 |
| `HEAD` | 检查资源是否存在或是否更新 |
| `POST` | 提交表单、上传文件、创建子资源 |
| `PUT` | 上传文件到指定位置（完全替换） |
| `DELETE` | 删除文件或资源 |
| `TRACE` | 调试代理链 |
| `OPTIONS` | 发现服务器能力 |
| `LOCK`/`UNLOCK` | 协同编辑（WebDAV） |

## 一、状态码（Status Codes）概览

HTTP 状态码分为五大类，每类的首位数字标识其类别：

| 范围 | 类别 | 含义 |
|------|------|------|
| 100–199 | 信息类 | 请求已接收，继续处理 |
| 200–299 | 成功类 | 请求成功完成 |
| 300–399 | 重定向类 | 需要进一步操作（如资源已移动） |
| 400–499 | 客户端错误 | 请求有误（如语法错误、资源不存在） |
| 500–599 | 服务器错误 | 服务器内部出错或无法响应 |

下面分别说明各类中重要的状态码。

---

### 1. 100–199：信息类状态码

| 状态码 | 原因短语 | 含义 |
|--------|----------|------|
| 100 | Continue | 服务器已收到请求头，客户端应继续发送实体主体 |
| 101 | Switching Protocols | 服务器同意切换协议（通过 `Upgrade` 头） |

#### 重点：`100 Continue` 的交互细节

- **目的**：避免客户端发送一个服务器无法处理的超大实体。
- **客户端行为**：
  - 发送 `Expect: 100-continue` 请求头。
  - 等待 100 Continue 响应后，再发送实体主体。
  - 如果超时未收到 100 Continue，应直接发送主体。
- **服务器行为**：
  - 收到 `Expect: 100-continue` 后，回复 100 Continue 或错误码。
  - 如果已收到部分主体，可跳过 100 Continue。
  - 若决定拒绝请求，不能只发响应就关闭连接（会丢失响应）。
- **代理行为**：
  - 若下一跳服务器支持 HTTP/1.1，转发 `Expect` 头。
  - 若下一跳是 HTTP/1.0 或更早，返回 `417 Expectation Failed`。
  - 不应将 100 Continue 转发给 HTTP/1.0 客户端。

---

### 2. 200–299：成功类状态码

| 状态码 | 原因短语 | 含义 |
|--------|----------|------|
| 200 | OK | 请求成功，主体包含所请求的资源 |
| 201 | Created | 资源创建成功（如 PUT 请求），Location 头指向新资源 |
| 202 | Accepted | 请求已接受但尚未处理（异步） |
| 203 | Non-Authoritative Info | 实体头来自副本，非源服务器 |
| 204 | No Content | 响应头正确，但没有实体主体（常用于表单刷新） |
| 205 | Reset Content | 清空表单内容 |
| 206 | Partial Content | 范围请求成功，需包含 Content-Range 等头 |

---

### 3. 300–399：重定向类状态码

| 状态码 | 原因短语 | 含义 |
|--------|----------|------|
| 300 | Multiple Choices | 资源有多种表示（如语言），客户端需选择 |
| 301 | Moved Permanently | 资源永久移动，新 URL 在 Location 头 |
| 302 | Found | 临时移动，但 HTTP/1.0 客户端会将 POST 改为 GET 重定向 |
| 303 | See Other | 应使用 GET 获取新 URL（用于 POST 后重定向） |
| 304 | Not Modified | 条件请求满足，资源未修改（无主体） |
| 305 | Use Proxy | 必须通过代理访问（Location 头指定代理） |
| 307 | Temporary Redirect | 临时移动，且**不允许改变方法**（POST 仍为 POST） |

#### 重点：302、303、307 的区别

- **HTTP/1.0 时代**：`302` 既表示临时移动，又要求客户端将 POST 改为 GET 重定向。
- **HTTP/1.1**：
  - `303 See Other`：明确要求客户端使用 GET 重定向（通常用于 POST 提交后的结果页面）。
  - `307 Temporary Redirect`：临时移动，但**保持原请求方法不变**（POST 仍为 POST）。
- **服务器实践**：
  - 对 HTTP/1.0 客户端，仍可发送 `302` 来获得 POST→GET 的转换。
  - 对 HTTP/1.1 客户端，应使用 `303`（若希望改为 GET）或 `307`（若保持方法）。

---

### 4. 400–499：客户端错误状态码

常见的有：

| 状态码 | 原因短语 | 含义 |
|--------|----------|------|
| 400 | Bad Request | 请求语法错误 |
| 401 | Unauthorized | 需要认证（WWW-Authenticate 头） |
| 403 | Forbidden | 拒绝访问（通常不解释原因） |
| 404 | Not Found | 资源不存在 |
| 405 | Method Not Allowed | 方法不允许（Allow 头列出允许的方法） |
| 406 | Not Acceptable | 无法满足客户端的 Accept 头要求 |
| 407 | Proxy Authentication Required | 需要代理认证 |
| 408 | Request Timeout | 客户端发送请求超时 |
| 409 | Conflict | 请求与资源当前状态冲突 |
| 410 | Gone | 资源已被永久删除 |
| 413 | Request Entity Too Large | 实体太大 |
| 414 | Request-URI Too Long | URI 太长 |
| 415 | Unsupported Media Type | 不支持的媒体类型 |
| 417 | Expectation Failed | 无法满足 Expect 头的要求 |

---

### 5. 500–599：服务器错误状态码

| 状态码 | 原因短语 | 含义 |
|--------|----------|------|
| 500 | Internal Server Error | 服务器内部错误 |
| 501 | Not Implemented | 方法不被支持 |
| 502 | Bad Gateway | 网关或代理收到无效响应 |
| 503 | Service Unavailable | 服务器暂时不可用（可带 Retry-After 头） |
| 504 | Gateway Timeout | 网关超时 |
| 505 | HTTP Version Not Supported | 不支持的 HTTP 版本 |

## 二、首部（Headers）的五种类别

| 类别 | 出现位置 | 作用 | 示例 |
|------|----------|------|------|
| **通用首部** | 请求和响应皆可 | 提供消息基本元信息 | `Date`、`Connection`、`Cache-Control` |
| **请求首部** | 仅请求 | 补充请求信息（如客户端能力） | `Accept`、`Host`、`If-Modified-Since` |
| **响应首部** | 仅响应 | 补充响应信息（如服务器软件） | `Server`、`Age`、`WWW-Authenticate` |
| **实体首部** | 请求或响应 | 描述主体内容 | `Content-Type`、`Content-Length`、`Content-Encoding` |
| **扩展首部** | 任意 | 非标准但由应用定义 | `X-Forwarded-For`、`Set-Cookie` |

### 通用首部中的缓存控制首部

| 首部 | 作用 |
|------|------|
| `Cache-Control` | 传递缓存指令（如 `no-cache`、`max-age`） |
| `Pragma` | 旧式缓存控制（`Pragma: no-cache`），已被 `Cache-Control` 取代 |

> 注意：`Pragma` 在规范中是请求首部，但常被错误地用于响应；建议使用 `Cache-Control`。

---

## 三、总结

- **状态码**是 HTTP 响应中最重要的反馈机制，程序应依赖三位数字码，而非原因短语。
- 重点掌握：
  - `100 Continue` 用于大实体发送前的握手。
  - `302`、`303`、`307` 的重定向语义差异（尤其是 POST 请求的重定向）。
  - `304 Not Modified` 用于条件请求验证缓存。
  - 4xx 和 5xx 区分客户端责任与服务器责任。
- **首部**按功能分为五类，理解其分类有助于快速定位和设计 HTTP 消息。

---

## 一、请求首部的作用

请求首部**只出现在请求消息中**，用于向服务器提供关于客户端的信息，例如：
- 谁在发送请求（客户端软件、用户邮箱）
- 请求从何而来（Referer）
- 客户端的偏好和能力（接受什么格式、语言、编码等）

服务器可以利用这些信息来提供更合适的响应。

---

## 二、请求首部的分类

### 1. 请求信息首部（Table 3-13）

| 首部 | 描述 |
|------|------|
| `Client-IP` | 客户端的 IP 地址（非标准，但许多客户端实现） |
| `From` | 用户的电子邮件地址（RFC 822 格式） |
| `Host` | 请求的目标服务器主机名和端口（HTTP/1.1 必须包含） |
| `Referer` | 当前请求 URI 所在页面的 URL（即从哪个链接点过来的） |
| `UA-Color` | 客户端显示器的颜色能力 |
| `UA-CPU` | 客户端 CPU 类型或制造商 |
| `UA-Disp` | 客户端屏幕显示能力 |
| `UA-OS` | 客户端操作系统名称和版本 |
| `UA-Pixels` | 客户端屏幕像素信息 |
| `User-Agent` | 发起请求的应用程序名称（如浏览器标识） |

> ⚠️ 注意：`Client-IP` 和所有 `UA-*` 首部**不是 RFC 2616 标准**，且 `UA-*` 可能带来不良影响（不应针对特定客户端配置定制内容）。

---

### 2. Accept 首部（Table 3-14）

客户端通过 Accept 首部告知服务器自己**愿意接收**的内容类型，以及**偏好程度**（通过质量值 `q`）。这有助于服务器进行内容协商。

| 首部 | 描述 |
|------|------|
| `Accept` | 可接受的媒体类型（如 `text/html`, `image/*`） |
| `Accept-Charset` | 可接受的字符集（如 `utf-8`, `iso-8859-1`） |
| `Accept-Encoding` | 可接受的内容编码（如 `gzip`, `deflate`） |
| `Accept-Language` | 可接受的自然语言（如 `en`, `zh-CN`） |
| `TE` | 可接受的传输编码扩展（如 `trailers`, `chunked`） |

> `TE` 首部的详细说明见第 15 章。

---

### 3. 条件请求首部（Table 3-15）

客户端可以在请求中附加**条件**，服务器**仅当条件满足时才执行请求**。常用于缓存验证、断点续传等场景。

| 首部 | 描述 |
|------|------|
| `Expect` | 列出客户端期望服务器支持的行为（如 `100-continue`） |
| `If-Match` | 仅当实体标签（ETag）匹配时才执行 |
| `If-Modified-Since` | 仅当资源在指定日期之后被修改过才返回 |
| `If-None-Match` | 仅当 ETag 不匹配时才执行（常用于缓存验证） |
| `If-Range` | 用于条件性的范围请求 |
| `If-Unmodified-Since` | 仅当资源在指定日期之后未修改过才执行 |
| `Range` | 请求资源的部分内容（字节范围） |

> 实体标签（ETag）是资源的版本标识符，详见第 7 章。

---

### 4. 请求安全首部（Table 3-16）

HTTP 原生支持质询/响应（challenge/response）认证框架，这些首部用于传递认证信息。

| 首部 | 描述 |
|------|------|
| `Authorization` | 客户端提供给服务器的认证凭证（如用户名/密码的 base64 编码） |
| `Cookie` | 客户端向服务器传递的会话状态令牌（非严格安全头，但有安全影响） |
| `Cookie2` | 指示客户端支持的 Cookie 版本（RFC 2965） |

> 认证机制详见第 12–14 章，Cookie 详见第 11 章。

---

### 5. 代理请求首部（Table 3-17）

用于帮助代理服务器更好地处理请求。

| 首部 | 描述 |
|------|------|
| `Max-Forwards` | 请求最多可被转发的次数（用于 TRACE 方法，防止无限循环） |
| `Proxy-Authorization` | 客户端向代理服务器提供认证凭证（与 `Authorization` 类似） |
| `Proxy-Connection` | 代理使用的连接控制（类似 `Connection` 头） |

> `Max-Forwards` 的详细说明见第 6 章。

## 三、总结：请求首部速查表

| 类别 | 常见首部示例 | 主要用途 |
|------|--------------|----------|
| 信息首部 | `Host`, `User-Agent`, `Referer` | 标识客户端和目标服务器 |
| Accept 首部 | `Accept`, `Accept-Language` | 内容协商，告诉服务器客户端能处理什么 |
| 条件首部 | `If-Modified-Since`, `If-None-Match` | 缓存验证、避免重复传输 |
| 安全首部 | `Authorization`, `Cookie` | 认证与会话管理 |
| 代理首部 | `Max-Forwards`, `Proxy-Authorization` | 控制代理转发行为 |

---

## 一、响应首部的作用

响应首部**只出现在响应消息中**，为客户端提供关于响应的补充信息，例如：
- 响应的“年龄”（经过了多少时间）
- 服务器的软件信息
- 资源不可用时的重试建议
- 协商相关的能力（支持哪些范围请求、内容是否随某些请求头变化）
- 安全挑战（要求客户端认证）

这些信息帮助客户端正确处理当前响应，并优化后续请求。

---

## 二、响应首部的分类

### 1. 响应信息首部（Table 3-18）

| 首部 | 描述 |
|------|------|
| `Age` | 响应已经经过的秒数（暗示经过了代理或缓存） |
| `Public` | 服务器对资源支持的方法列表（定义于 RFC 2068，已不在 RFC 2616 中） |
| `Retry-After` | 资源不可用时，建议客户端重试的日期或延迟秒数 |
| `Server` | 服务器应用软件的名称和版本 |
| `Title` | HTML 文档的标题（非标准，来自早期 HTTP/1.0 草案） |
| `Warning` | 比原因短语更详细的警告信息（如缓存相关警告） |

> `Public` 和 `Title` 在现代 HTTP 规范中已不定义，但可能在某些实现中出现。

---

### 2. 协商首部（Table 3-19）

当服务器对一个 URI 存在多种表示（例如同一文档的法语版和德语版）时，通过协商首部告知客户端可协商的能力。

| 首部 | 描述 |
|------|------|
| `Accept-Ranges` | 服务器接受的范围请求类型（如 `bytes`） |
| `Vary` | 列出哪些请求头会影响响应内容的选择（例如 `Vary: Accept-Language` 表示响应取决于 `Accept-Language` 头的值） |

> 内容协商的详细说明见第 17 章。

---

### 3. 响应安全首部（Table 3-20）

这些首部用于 HTTP 的质询/响应认证机制，服务器向客户端发出认证要求。

| 首部 | 描述 |
|------|------|
| `Proxy-Authenticate` | 代理服务器向客户端发出的认证质询（如 `Basic realm="..."`） |
| `Set-Cookie` | 服务器在客户端设置 Cookie（虽然不是严格的安全头，但与安全密切相关） |
| `Set-Cookie2` | RFC 2965 版本的 Cookie 设置头 |
| `WWW-Authenticate` | 源服务器向客户端发出的认证质询 |

> 认证机制详见第 12–14 章，Cookie 详见第 11 章。

---

## 三、总结：响应首部速查表

| 类别 | 常见首部示例 | 主要用途 |
|------|--------------|----------|
| 信息首部 | `Server`, `Age`, `Retry-After` | 描述响应来源、年龄、重试策略 |
| 协商首部 | `Accept-Ranges`, `Vary` | 支持内容协商和缓存多样性 |
| 安全首部 | `WWW-Authenticate`, `Set-Cookie` | 认证挑战、状态管理 |

响应首部帮助客户端理解响应的上下文（是否来自缓存、是否需要认证、资源是否可部分获取等），是实现高级 HTTP 功能（缓存验证、内容协商、安全认证）的关键。

您发来的这段内容是《HTTP权威指南》第3章中关于**实体首部（Entity Headers）** 的讲解。实体首部用于描述 HTTP 消息的**有效载荷（payload）**，即实体主体（entity body）的相关信息。下面我用中文为您系统性地总结。

---

## 一、实体首部的作用

实体首部可以出现在**请求消息**或**响应消息**中（只要消息包含实体主体）。它们告诉接收方：
- 实体的类型（文本、图片、视频等）
- 实体的大小、编码、语言
- 实体的位置、校验和
- 缓存验证信息（如最后修改时间、过期时间、实体标签）

---

## 二、实体首部的分类

### 1. 实体信息首部（Table 3-21）

| 首部 | 描述 |
|------|------|
| `Allow` | 列出该资源支持哪些请求方法（如 `GET`, `HEAD`, `PUT`） |
| `Location` | 告知客户端实体实际所在的 URL（常用于重定向，指向新位置） |

> `Location` 通常在 3xx 重定向响应中使用，也可用于 201 Created 响应中指示新资源的位置。

---

### 2. 内容首部（Table 3-22）

提供关于实体主体内容的详细信息。

| 首部 | 描述 |
|------|------|
| `Content-Base` | 解析主体中相对 URL 的基 URL（**不在 RFC 2616 中定义**） |
| `Content-Encoding` | 对主体应用的编码（如 `gzip`, `deflate`） |
| `Content-Language` | 实体的自然语言（如 `en`, `zh-CN`） |
| `Content-Length` | 主体的大小（字节数） |
| `Content-Location` | 资源的实际位置（可能与请求 URL 不同） |
| `Content-MD5` | 主体的 MD5 校验和（用于检查数据完整性） |
| `Content-Range` | 当前实体代表完整资源的哪个字节范围（与 206 Partial Content 一起使用） |
| `Content-Type` | 实体的媒体类型（如 `text/html`, `image/jpeg`），可带字符集参数 |

---

### 3. 实体缓存首部（Table 3-23）

提供缓存验证所需的信息，用于判断缓存副本是否仍然有效。

| 首部 | 描述 |
|------|------|
| `ETag` | 实体标签（entity tag），是资源特定版本的唯一标识符 |
| `Expires` | 实体过期的绝对日期时间（过期后必须重新从源服务器获取） |
| `Last-Modified` | 实体最后被修改的日期和时间 |

> 缓存相关的详细内容见第 7 章。

---

## 三、总结：实体首部速查表

| 类别 | 常见首部示例 | 主要用途 |
|------|--------------|----------|
| 信息首部 | `Allow`, `Location` | 说明支持的方法或资源位置 |
| 内容首部 | `Content-Type`, `Content-Length`, `Content-Encoding` | 描述主体的格式、大小、编码、语言、校验和等 |
| 缓存首部 | `ETag`, `Last-Modified`, `Expires` | 缓存验证与过期控制 |

实体首部是 HTTP 消息中**最贴近实际传输数据**的一类首部，它们确保客户端能够正确解析、显示、缓存和验证接收到的内容。
