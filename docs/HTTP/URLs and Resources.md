| 现实世界 | 互联网世界 |

## URL 与 URN 的对比

书中本章会详细区分：

- **URL**：强调 **位置**（Location），例如 `http://www.example.com/index.html`  
- **URN**：强调 **名称**（Name），例如 `urn:isbn:0451450523`（一本书的 ISBN），不关心它当前存在哪个服务器上

> 日常我们说的“网址”绝大多数是 URL。URN 尚未普及，但作者会在本章末尾讨论其前景。

## 核心概念回顾

- **URI（Uniform Resource Identifier）**：统一资源标识符，一个通用的概念，用于唯一标识一个资源。
- **URL（Uniform Resource Locator）**：统一资源定位符，URI 的子集，通过**描述资源的位置**来标识它。
- **URN（Uniform Resource Name）**：统一资源名称，URI 的另一子集，通过**给资源一个唯一的名字**来标识它，与当前位置无关。

> 在实际的 HTTP 应用中，几乎只使用 URL。所以本书后续虽然可能混用 URI 和 URL 术语，但本质上讨论的都是 URL。

---

## URL 的三部分结构（以 `http://www.joes-hardware.com/seasonal/index-fall.html` 为例）

| 组成部分 | 示例 | 作用 |
|----------|------|------|
| **方案（scheme）** | `http` | 告诉客户端用什么协议访问资源（如 HTTP、FTP、RTSP） |
| **服务器地址（server location）** | `www.joes-hardware.com` | 资源所在的主机名（通过 DNS 解析为 IP） |
| **资源路径（resource path）** | `/seasonal/index-fall.html` | 服务器上该资源的本地路径 |

> 其他协议的例子：
> - `mailto:president@whitehouse.gov`（电子邮件地址）
> - `ftp://ftp.lots-o-books.com/pub/complete-price-list.xls`（FTP 文件）
> - `rtsp://www.joes-hardware.com:554/interview/cto_video`（实时流媒体）

## URL 的通用格式

大多数 URL 遵循相同的模式：

```
<scheme>://<host>/<path>
```

这种统一的结构让任何资源（无论类型、无论访问方式）都可以用一个字符串方便地表示和分享。

### 核心三要素（最重要）

绝大多数 URL 都包含这三个部分，是定位资源的基础：

1.  **scheme (协议)**：告诉浏览器应该用什么规则去访问资源。最常见的是 `http` 和 `https`。**没有默认值**，必须指定。
2.  **host (主机)**：资源所在服务器的地址，可以是 `www.example.com` 这样的域名，也可以是 ``192.168.1.1`` 这样的 IP 地址。**没有默认值**，必须指定。
3.  **path (路径)**：资源在服务器上的具体位置，就像你电脑里的文件夹路径。例如 `/images/logo.png`。**没有默认值**。

### 其他常见组件

这些组件为资源访问提供了更多细节，但并非必需：

4.  **port (端口)**：服务器上的一个“入口编号”。就像一栋大楼有不同的门，不同的服务走不同的门。HTTP 的默认端口是 `80`，HTTPS 默认是 `443`。如果你不写，浏览器会自动使用默认端口。
5.  **query (查询字符串)**：向服务器传递参数，常用于搜索或动态页面。它跟在 `?` 后面，通常以 `key=value` 的形式出现，多个参数之间用 `&` 连接。
    -   **例子**：`/search?q=URL语法&page=1`
6.  **frag (片段)**：指向页面内部的特定位置，如一个小节或标题。它跟在 `#` 后面，并且**不会发送给服务器**，仅供浏览器在加载完页面后定位使用。
    -   **例子**：`/chapter2.html#url-syntax`

### 较少见或用于特殊场景的组件

这些组件在现代网页浏览中不常见，但在特定协议或旧系统中会出现：

7.  **user (用户名) & password (密码)**：用于向服务器提供身份验证信息，例如在 FTP 协议中。通常格式为 `username:password@`。出于安全考虑，现代 Web 浏览器正逐渐减少对在 URL 中明文传递密码的支持。
8.  **params (参数)**：为特定协议（如 FTP）提供额外的输入参数。它跟在路径后，用 `;` 与路径和其他参数分隔。

你提供的这段文字解释了 URL 中 **scheme（协议）** 组件的作用和规则。

### 什么是 Scheme？

**Scheme** 告诉浏览器或其他应用程序：**你应该用什么“语言”（协议）去和服务器沟通？** 它是 URL 中最关键的部分，决定了后续所有信息的解读方式。

-   例如，`http://` 告诉浏览器使用 HTTP 协议；`https://` 使用加密的 HTTP 协议；`ftp://` 使用文件传输协议。

### Scheme 的语法规则

根据你给出的文字，Scheme 组件遵循以下三个简单规则：

1.  **必须以字母开头**：`http`、`https`、`ftp` 都是合法的，而 `1http` 或 `!ftp` 不合法。
2.  **由第一个冒号 `:` 与后面的内容分隔**：冒号紧跟在 scheme 后面，标志着 scheme 的结束和 URL 其余部分的开始。
3.  **不区分大小写**：`http://`、`HTTP://` 和 `Http://` 在浏览器看来是完全一样的。

### 示例

你引用的例子非常清楚：

-   `http://www.joes-hardware.com` 中的 scheme 是 `http`
-   `HTTP://www.joes-hardware.com` 中的 scheme 是 `HTTP`

尽管第二个用了大写字母，但根据规则它是**大小写不敏感**的，所以这两个 URL **完全等价**，浏览器都会以 HTTP 协议去访问 `www.joes-hardware.com`。

---

### 一句话总结

**Scheme 就是 URL 开头的那个“协议名”，它必须以字母开头、跟一个冒号，并且不区分大小写。**

常见的 scheme 还包括 `https`、`ftp`、`mailto`、`file` 等，每个都对应不同的网络协议或访问方式。

这段文字解释了 URL 中 **host（主机）** 和 **port（端口）** 组件的作用。

简单来说，要找到互联网上的一个资源，你需要两样信息：

1.  **它在哪台机器上？** → 这就是 **host**
2.  **在这台机器的哪里？**（因为一台机器上可能运行着多个网络服务） → 这就是 **port**

### Host（主机）

**host** 指明了资源所在的具体机器。它有两种写法：

-   **域名**：例如 `www.joes-hardware.com`，方便人类记忆。
-   **IP 地址**：例如 `161.58.228.45`，这是网络设备真正用来寻址的数字标识。

就像你给朋友寄信：**host** 相当于**城市和街道名**（`北京市朝阳区XX路`），定位到具体的一栋楼（服务器）。

### Port（端口）

**port** 指明了服务器上正在监听的特定网络服务入口。一台物理服务器可以同时运行多个服务（网页、邮件、FTP 等），**端口号**用来区分这些不同的服务。

-   例如，HTTP 服务的默认端口是 **80**，HTTPS 是 **443**，邮件服务可能是 25。
-   在 URL 中，端口号跟在主机名后面，用冒号 `:` 分隔。

就像你到了那栋楼（服务器），还得知道**具体去几楼几室**（端口），才能找到真正办事的地方。

### 示例解析

你提供的两个 URL 指向**完全相同的资源**：

```
http://www.joes-hardware.com:80/index.html
http://161.58.228.45:80/index.html
```

-   **scheme** 都是 `http`。
-   **host** 一个是 `www.joes-hardware.com`，另一个是其对应的 IP 地址 `161.58.228.45`。
-   **port** 都是 `80`。
-   **path** 都是 `/index.html`。

### 默认端口

-   大多数协议都有**默认端口**。对于 HTTP 协议，默认端口就是 **80**。
-   这意味着如果你在 URL 中**省略了端口号**，浏览器会自动使用该协议的默认端口。
-   所以，`http://www.joes-hardware.com/index.html` 和 `http://www.joes-hardware.com:80/index.html` 是完全等价的。只有当你需要访问非默认端口（例如内部测试服务器 `http://localhost:8080`）时，才需要显式写出端口号。

### 一句话总结

-   **host** = 哪台机器（域名或 IP）
-   **port** = 哪个服务（数字，HTTP 默认为 80）

这段文字解释了 URL 中的 **user（用户名）** 和 **password（密码）** 组件。这两个组件通常用于需要身份验证的协议，最典型的就是 **FTP**。

### 语法格式

用户名和密码放在主机之前，格式为：
```
<scheme>://<user>:<password>@<host>:<port>/<path>
```

-   用户名和密码之间用冒号 `:` 分隔。
-   整个用户/密码部分与后面的主机之间用 `@` 符号分隔。

### 示例解析

文字中给出了四个例子：

1.  `ftp://ftp.prep.ai.mit.edu/pub/gnu`
    -   没有用户名和密码。
    -   对于 FTP 协议，浏览器会自动补上默认的匿名用户名和密码。

2.  `ftp://anonymous@ftp.prep.ai.mit.edu/pub/gnu`
    -   只提供了用户名 `anonymous`，没有提供密码。
    -   浏览器会使用默认密码（如 IE 用 `IEUser`，Netscape 用 `mozilla`）。

3.  `ftp://anonymous:my_passwd@ftp.prep.ai.mit.edu/pub/gnu`
    -   同时提供了用户名 `anonymous` 和密码 `my_passwd`。

4.  `http://joe:joespasswd@www.joes-hardware.com/sales_info.txt`
    -   虽然是 HTTP 协议，但理论上也支持在 URL 中嵌入用户名和密码（尽管现代浏览器出于安全考虑已逐渐限制这种做法）。

### 默认行为（匿名登录）

许多公开的 FTP 服务器允许 **匿名登录**。如果你在 FTP URL 中没有提供用户名和密码，浏览器会自动为你填写：

-   **用户名**：`anonymous`
-   **密码**：不同浏览器不同，例如 IE 发送 `IEUser`，Netscape Navigator 发送 `mozilla`。

这样，用户就不需要手动输入，浏览器自动完成匿名身份验证。

### 安全提醒（重要）

虽然 URL 语法支持直接写入用户名和密码，但在现代 Web 环境中这种做法**强烈不推荐**，原因如下：

-   **明文传输风险**：整个 URL（包括密码）可能会被网络中的中间节点（如代理服务器、路由器）记录下来。
-   **浏览器历史记录**：带有密码的 URL 会被保存在浏览器的历史记录或书签中，容易被他人看到。
-   **逐渐废弃**：现代浏览器和 HTTP 协议正在逐步移除或限制对 URL 中明文密码的支持，推荐使用标准的 `Authorization` 头或 OAuth 等更安全的认证方式。

### 一句话总结

-   **user** 和 **password** 组件用于在 URL 中直接提供登录凭据，最常见于 FTP 协议。
-   如果不提供，许多协议（尤其是 FTP）会自动使用 `anonymous` 作为默认用户名。
-   在 HTTP 中使用这种方式已逐渐被淘汰，因为存在安全风险。

### Path 的作用

**path** 指明了资源在服务器上的具体位置。它通常是类似文件系统路径的层级结构，服务器根据这个路径去定位并返回相应的资源。

### 语法格式

-   路径以斜杠 `/` 开头，后面可以跟多级目录和文件名。
-   不同层级之间用 `/` 分隔。
-   例如：`/seasonal/index-fall.html` 表示在服务器根目录下的 `seasonal` 文件夹中，找到 `index-fall.html` 文件。

### 示例解析

原文中的 URL：
```
http://www.joes-hardware.com:80/seasonal/index-fall.html
```

-   **scheme**: `http`
-   **host**: `www.joes-hardware.com`
-   **port**: `80`
-   **path**: `/seasonal/index-fall.html`

这行路径看起来就像 Unix/Linux 系统上的文件路径：根目录下的 `seasonal` 目录，里面有一个 `index-fall.html` 文件。

### 路径段（Path Segments）与参数（Params）

-   路径可以被 `/` 分隔成多个**段（segments）**。例如 `/seasonal/index-fall.html` 有两个段：`seasonal` 和 `index-fall.html`。
-   每个段都可以拥有自己的**参数（params）**。参数紧跟在段后面，以分号 `;` 开头，格式为 `;param=value`。这样设计是因为不同的文件格式或服务可能需要传递特定的参数给某个中间目录或文件。

书中后面会有例子展示带参数的路径，比如：
```
http://www.joes-hardware.com/hammers;sale=false/index.html;graphics=true
```
这里 `hammers` 段有参数 `sale=false`，`index.html` 段有参数 `graphics=true`。

### 总结

-   **path** 告诉服务器：资源在哪个文件夹、叫什么名字。
-   它通常是层次化的，用 `/` 分隔不同层级。
-   每个路径段还可以附上自己的参数（`;key=value`），提供更精细的控制。

在实际的 HTTP URL 中，路径部分是最直观、最常用的组件之一。如果你需要访问网站上的某个特定页面或文件，基本上就是在构造一个正确的路径。

这段文字解释了 URL 中 **params（参数）** 组件的作用和使用场景。

### 为什么需要 Params？

除了主机、端口、路径、用户名/密码之外，许多协议还需要**额外的参数**才能正确工作。例如：

-   **FTP 协议**有二进制和文本两种传输模式。如果你传输一个图片文件（二进制），但错误地用了文本模式，图片数据会被破坏（scrambled）。
-   这些额外信息不能放在路径里，于是 URL 提供了 **params** 组件。

### Params 的语法格式

-   params 是一组 **键值对（name=value）**。
-   多个 params 之间用分号 `;` 分隔。
-   params 可以附加在路径的某个段（segment）后面，也可以附加在整个路径之后。
-   它位于路径段和查询字符串（`?`）之前。

一个简单的例子（FTP）：
```
ftp://prep.ai.mit.edu/pub/gnu;type=d
```
-   路径是 `/pub/gnu`
-   params 是 `type=d`，其中键是 `type`，值是 `d`（表示二进制模式）。

### Params 与路径段（Path Segments）的关系

URL 的路径可以被 `/` 分成多个段（segments）。**每个段都可以拥有自己独立的 params**。params 紧跟在它所属的段后面，用分号 `;` 开头。

示例：
```
http://www.joes-hardware.com/hammers;sale=false/index.html;graphics=true
```
-   第一个路径段：`hammers`，后面跟着 `;sale=false`，表示该段有一个参数 `sale=false`。
-   第二个路径段：`index.html`，后面跟着 `;graphics=true`，表示该段有一个参数 `graphics=true`。

服务器在处理请求时，会分别解析每个段及其参数，从而影响资源的定位或处理方式。

### 现代使用情况

需要说明的是，**params 组件在实际的 HTTP URL 中已经非常少见**。HTTP/1.1 规范虽然允许这种语法，但现代 Web 应用通常使用：
-   **查询字符串（query string）**：`?key=value&key2=value2` 传递参数
-   **请求头（headers）** 或 **请求体（body）** 传递更复杂的参数

params 更多地出现在一些老协议（如 FTP）或者某些特定框架的内部路由设计中。但作为 HTTP 知识体系的一部分，了解它有助于理解 URL 设计的完整性。

### 一句话总结

-   **params** 是附加在 URL 路径段上的键值对参数，用分号 `;` 分隔。
-   每个路径段都可以有自己的 params，用于向服务器传递该段特有的额外信息。
-   在现代 Web 开发中已不常用，但在某些协议（如 FTP）中至关重要。

### 什么是 Query String？

**query** 用于向服务器传递额外的参数，常见于数据库查询、搜索功能或动态页面。它允许客户端缩小请求范围或提供输入数据。

### 语法格式

-   query 位于 URL 末尾，以问号 `?` 开头。
-   通常（但非强制）格式为 **多个 `name=value` 键值对**。
-   多个键值对之间用 **`&`** 分隔。

### 示例解析

最基础的例子：
```
http://www.joes-hardware.com/inventory-check.cgi?item=12731
```
-   **path** 是 `/inventory-check.cgi`，标识这是一个网关程序（gateway）。
-   **query** 是 `item=12731`，表示查询商品编号为 12731 的库存。

更复杂的例子（包含多个参数）：
```
http://www.joes-hardware.com/inventory-check.cgi?item=12731&color=blue
```
-   有两个键值对：`item=12731` 和 `color=blue`。

用户上传的 **Figure 2-2** 展示了更完整的例子：
```
http://www.joes-hardware.com/inventory-check.cg1?item=12731&color=blue&size=large
```
-   注意路径中的文件是 `inventory-check.cg1`（可能是笔误，应该是 `.cgi`）。
-   三个参数：`item=12731`、`color=blue`、`size=large`。

图中箭头表示：客户端发送这个 URL 到服务器，服务器将 query 部分（`?` 后面的内容）传递给 “Inventory-check” 网关应用程序。网关负责解析这些参数，查询数据库，并返回结果。

### Query 与 Gateway 的关系

-   **path** 标识了一个**网关资源**（通常是一个 CGI 脚本、Servlet 或其他后端程序）。
-   **query** 是传给该网关的**输入参数**。
-   网关根据这些参数执行操作（如查询数据库、调用其他服务），然后生成动态响应返回给客户端。

### 格式的自由度

-   协议**没有强制要求** query 必须是 `name=value` 格式，唯一限制是某些字符（如空格、`#` 等）需要编码。
-   但 **按惯例**，绝大多数 Web 应用都采用 `key=value` 并用 `&` 分隔，因为这是 CGI 规范所支持的解析方式。

### 与 Params 的区别

| 特性 | params | query |
|------|--------|-------|
| 分隔符 | 分号 `;` | 问号 `?` |
| 位置 | 路径段内 | 路径之后 |
| 作用域 | 每个路径段独立 | 整个 URL 只有一个 query |
| 常见用途 | 旧协议（如 FTP） | 现代 Web 动态请求（搜索、筛选、API 调用） |

### 总结

-   **query** 是 URL 中 `?` 后面的部分，用于向服务器传递参数。
-   常见格式：`?key1=value1&key2=value2&key3=value3`。
-   通常与 **网关程序**（如 `.cgi`、`.php`、`.asp`）配合使用，网关读取参数后动态生成内容。
-   是现代 Web 开发中最常用的参数传递方式之一。

这段文字解释了 URL 中的 **frag（片段）** 组件。

### 什么是 Fragment？

**fragment** 用于指向资源内部的某个**特定部分**，例如一个长 HTML 文档中的某个章节、一个页面中的某个锚点，或者一幅图像的某个区域。它让 URL 能够在文档内部进行“定位”，而不只是指向整个文档。

### 语法格式

-   fragment 以井号 `#` 开头，紧跟在 URL 的末尾。
-   它的值是一个字符串（通常是 HTML 元素的 `id` 或 `name` 属性值），标识资源内的一个命名位置。

示例：
```
http://www.joes-hardware.com/tools.html#drills
```
-   完整资源是 `/tools.html`。
-   fragment 是 `drills`，表示这个网页中名为 `drills` 的锚点位置（可能是一个章节标题，如 `<h2 id="drills">Drills</h2>`）。

### Fragment 的处理流程

这是最关键的一点：**fragment 不会被发送给服务器**。

-   HTTP 协议中，客户端请求的 URL 只包含 scheme、host、port、path、query，以及必要的认证信息。**`#` 及其后面的内容会被浏览器截留，不包含在发送给服务器的 HTTP 请求中**。
-   浏览器首先请求完整的资源（服务器返回整个 HTML 文件）。
-   浏览器收到整个文件后，再根据 fragment 的值滚动到页面对应的位置。

所以书中 Figure 2-3（未在内容中显示）的标题是“The URL fragment is used only by the client, because the server deals with entire objects”，意思就是：**fragment 纯属客户端行为，服务器不参与处理**。

### 实际应用

-   HTML 页面中，通过 `<a name="drills">` 或任意元素的 `id="drills"` 可以创建锚点。
-   用户点击带有 `#drills` 的链接时，浏览器会打开对应页面并自动滚动到该锚点处。
-   单页应用（SPA）也常使用 fragment（例如 `#/home`）来实现前端路由，这部分同样不会发送到服务器，由前端 JavaScript 处理。

### 总结

| 组件 | 分隔符 | 发送给服务器？ | 用途 |
|------|--------|---------------|------|
| query | `?` | ✅ 是 | 传递参数给后端程序 |
| frag | `#` | ❌ 否 | 在客户端定位资源内部位置 |

-   **frag** 是 URL 中 `#` 后面的部分，用于指向资源内部的片段。
-   它**不会出现在 HTTP 请求中**，仅供浏览器使用。
-   常用于 HTML 文档的内链导航或前端路由。

- **书写简洁**：在 HTML 文档中，不需要重复写 `http://www.joes-hardware.com` 这一长串。
- **便于迁移**：如果把整套 HTML 文件挪到另一个服务器（或另一个目录），只要相对路径关系不变，所有链接依然有效。

你提供的这段文字解释了 **基 URL（Base URL）** 的概念和作用，以及如何找到它。

### 什么是基 URL？

**基 URL** 是用于**解析相对 URL** 的参照基准。因为相对 URL 缺少部分信息（如协议、主机等），必须借助基 URL 才能补全为一个完整的绝对 URL。

### 基 URL 的来源（按优先级）

文字中列出了三种情况：

1. **资源内显式指定**（优先级最高）  
   某些资源（如 HTML 文档）可以通过 `<BASE>` 标签明确指定基 URL。  
   例如：`<base href="http://www.joes-hardware.com/">`  
   一旦指定，文档中所有相对 URL 都会基于这个地址去解析。

2. **封装资源的 URL**（没有显式 `<BASE>` 时）  
   如果资源中没有指定基 URL，则使用**该资源本身所在的 URL** 作为基 URL。  
   比如你打开 `http://www.joes-hardware.com/tools.html`，这个页面里的相对 URL `./hammers.html` 就会基于这个地址去解析。

3. **没有基 URL**（异常情况）  
   这通常意味着你得到的是一个绝对 URL（不需要基 URL），但也可能出现 URL 不完整或格式错误的情况。

### Figure 2-4 示例回顾

- **基 URL**：`http://www.joes-hardware.com/tools.html`
- **相对 URL**：`./hammers.html`
- **解析后的绝对 URL**：`http://www.joes-hardware.com/hammers.html`

图中直观地演示了将相对 URL “拼接” 到基 URL 的目录部分，形成完整地址的过程。

### 为什么需要显式指定 `<BASE>`？

默认情况下，浏览器使用当前页面的 URL 作为基 URL。但如果你希望所有相对 URL 都相对于另一个地址（比如一个虚拟根目录），可以通过 `<base>` 标签覆盖。这在内容聚合、门户页面或某些 CMS 中很有用。

### 一句话总结

**基 URL 是相对 URL 的“参考点”**，它要么来自当前页面的地址，要么由 `<base>` 标签显式指定；没有基 URL 时，相对 URL 无法正常解析。

## 一、算法总览（流程图解读）

整个算法从解析后的相对 URL 组件开始（即 scheme、user、password、host、port、path、params、query、frag 等）。流程图的判断顺序大致如下：

1. **检查 scheme 是否为空**  
   - 如果不为空 → 该 URL 已经是绝对 URL（直接使用）  
   - 如果为空 → 需要从基 URL 继承 scheme

2. **检查 user、password、host、port 是否全部为空**  
   - 如果全部为空 → 从基 URL 继承这些组件  
   - 如果有任何一个非空 → 这些组件已经明确（无需继承），然后直接去看 path

3. **检查 path 组件**  
   - 如果路径为空 → 从基 URL 继承路径  
   - 如果路径非空：
     - 若路径以 `/` 开头 → 已是**绝对路径**，保留  
     - 若路径不以 `/` 开头 → 它是**相对路径**，需要与基 URL 的路径合并（处理 `./`、`../` 等）

4. **检查 params 组件**  
   - 如果 params 为空 → 从基 URL 继承 params  
   - 如果非空 → 使用相对 URL 自己的 params

5. **检查 query 组件**  
   - 如果 query 为空 → 从基 URL 继承 query  
   - 如果非空 → 使用相对 URL 自己的 query

6. **最后组合**：将继承来的和相对 URL 中非空的组件合并，形成新的绝对 URL。

> 注意：图中还有处理 `..` 和 `.` 的步骤（“Remove ‘..’ and ‘<seq>../.’ from path”），这是为了简化路径中的父目录引用。

## 二、文字中的例子逐步转换

**已知：**
- 基 URL：`http://www.joes-hardware.com/tools.html`
- 相对 URL：`./hammers.html`

**步骤按照书中描述：**

1. 解析相对 URL 的组件  
   - `scheme`: 空  
   - `host`, `port` 等：空  
   - `path`: `./hammers.html`  
   - 其他组件：空

2. **scheme 为空** → 继承基 URL 的 scheme → `http`

3. **host、port 等组件全部为空** → 继承基 URL 的 host 和 port  
   - host → `www.joes-hardware.com`  
   - port → `80`（因为默认端口，可以显式继承）

4. **path 非空**：`./hammers.html` 不是以 `/` 开头，需要与基 URL 的路径合并。  
   基 URL 的路径是 `/tools.html`（位于根目录），`./` 表示“当前目录”，所以合并后得到 `/hammers.html`（去掉了 `./`，且 `hammers.html` 与 `tools.html` 在同一目录下）

5. **params、query 均为空** → 继承基 URL 的（基 URL 也没有 params 和 query，所以为空）

6. **组合**：`http` + `://` + `www.joes-hardware.com` + `:80` + `/hammers.html`  
   → 得到 `http://www.joes-hardware.com/hammers.html`

## 三、关键点说明

| 概念 | 解释 |
|------|------|
| **继承** | 当相对 URL 缺少某个组件时，从基 URL 中取对应组件的值 |
| **绝对路径**（以 `/` 开头） | 不依赖基 URL 的路径，直接替换整个路径 |
| **相对路径**（不以 `/` 开头） | 需要与基 URL 的路径进行“目录合并”，例如 `./` 表示当前目录，`../` 表示上级目录 |
| **`#fragment`** | 始终不参与服务器端转换，完全由客户端处理（图中未画出，但通常 fragment 不参与继承决策，因为它在转换后直接附加到绝对 URL 末尾） |

## 一、主机名扩展（Hostname Expansion）

浏览器在你输入地址时，会根据一些简单的规则**自动补全主机名**，省去你输入 `www.` 和 `.com` 的麻烦。

### 示例
- 你输入：`yahoo`
- 浏览器自动扩展为：`www.yahoo.com`

### 工作原理
- 当你输入一个不完整的主机名（例如 `yahoo`），浏览器会尝试：
  1. 添加 `www.` 前缀
  2. 添加 `.com` 后缀
  3. 也可能尝试 `.org`、`.net` 等常见后缀

- 如果 DNS 解析失败，浏览器会依次尝试这些扩展组合，直到找到一个能成功解析的域名，或者放弃。

### 潜在问题
- 这种自动扩展在**直接访问服务器**时很方便，但**可能与代理服务器产生冲突**。
- 原因：当你配置了显式代理时，浏览器不会进行主机名扩展（如后面章节会讲到的），代理可能收到类似 `http://yahoo/` 这样的不完整 URL，导致请求失败。
- 书中提到将在 **第 6 章（Proxies）** 中详细讨论这个问题。

---

## 二、历史扩展（History Expansion）

浏览器会记录你**曾经访问过的 URL 历史**，当你输入地址时，它会根据你已输入的前缀，自动匹配历史记录中相似的 URL，并以下拉列表的形式提供补全建议。

### 示例
- 你开始输入：`http://www.joes-`
- 浏览器从历史中找到匹配的 URL：`http://www.joes-hardware.com`
- 你可以直接选择这个建议，而不必继续输入完整地址。

### 优点
- 对于经常访问的网站，可以快速打开，减少重复输入。

---

## 三、与代理一起使用时的注意事项

- 当你**显式配置了代理服务器**时，浏览器的**主机名扩展行为可能会改变**（通常被禁用或行为不同），这可能导致某些简写 URL 无法正确解析。
- 书中预告：第 6 章的“URI Client Auto-Expansion and Hostname Resolution”一节会深入讨论这个问题。

---

## 总结

| 扩展类型 | 行为 | 示例 | 潜在问题 |
|---------|------|------|----------|
| 主机名扩展 | 自动添加 `www.` 和 `.com` 等常见前后缀 | `yahoo` → `www.yahoo.com` | 与代理配合时可能产生不完整 URL |
| 历史扩展 | 基于访问历史自动补全已访问过的完整 URL | 输入 `http://www.joes-` 提示 `http://www.joes-hardware.com` | 无重大缺陷，仅依赖本地历史 |

这些功能是浏览器为提升用户体验而设计的便利特性，但开发者和高级用户需要了解其在特殊网络环境（如使用代理时的行为差异），以免导致意外的请求错误。

下面我来为你梳理这一节的核心内容。

---

## 一、为什么 URL 要限制字符？

URL 需要通过各种网络协议（如 HTTP、SMTP、FTP 等）传输。不同协议对数据的处理方式不同，有些协议（如早期 SMTP）只使用 7 位编码，可能会**剥离**某些字符（例如 8 位的高位）。为了保证 URL 在任何协议下都不会“被破坏”或丢失信息，设计者规定 URL **只能包含一个“普遍安全”的小字符集**。

同时，URL 还要**便于人类阅读**，所以**不可见的非打印字符**（如空格、制表符、控制字符等）也被禁止。

---

## 二、但这样不够“完整”

限制字符集带来了一个问题：用户可能需要 URL 中包含**二进制数据**或**非 ASCII 字符**（如中文、欧洲变音字母）。比如上传一个二进制密钥，或者文件名包含 €、é 等。

为了兼顾**安全性**（不违法字符集规则）和**完整性**（能表示任意数据），URL 引入了**转义机制**。

---

## 三、URL 字符集

早期系统多使用 **US-ASCII**（7 位，128 个字符）。它只包含英文字母、数字、少量标点和控制字符，**不支持**：

- 欧洲语言的变音字母（如 é、ü、ç）
- 非罗马字母语言（如中文、阿拉伯文）
- 任意二进制数据

因此，仅靠 ASCII 无法满足全球化和多样化的需求。

---

## 四、转义机制（Escape Mechanism）

URL 使用 **百分号编码（percent-encoding）** 来表示不安全字符：

- 格式：`%` 后跟两个十六进制数字（代表该字符的 ASCII 码或字节值）
- 例如：
  - 空格（ASCII 32） → `%20`
  - 波浪号 `~`（ASCII 126） → `%7E`
  - 中文“中”的 UTF-8 编码（`E4 B8 AD`） → `%E4%B8%AD`（如果 URL 使用 UTF-8）

这样，原本不允许出现的字符就被“安全”地转换成只包含 `%` 和字母数字的字符串，可以安全地通过任何协议传输。

---

## 五、一句话总结

| 问题 | 解决方案 |
|------|----------|
| 不同协议可能破坏某些字符 | 限制 URL 只使用安全字符集（US-ASCII 子集） |
| 安全字符集不能表示所有数据（如二进制、非英文字符） | 引入 **% 转义编码**，将不安全字符编码为 `%XX` |

## 一、编码规则

- **格式**：`%` + 两个十六进制数字（不区分大小写）
- 这两个十六进制数字表示该字符的 **ASCII 码**（对于二进制数据可能表示字节值）
- 这样，原本“不安全”的字符就被转换成由 `%`、数字、字母 `A-F` 组成的“安全”字符串

---

## 二、Table 2-2 示例解读

| 字符 | ASCII 码（十进制/十六进制） | 示例 URL | 说明 |
|------|--------------------------|----------|------|
| `~` | 126 (0x7E) | `http://www.joes-hardware.com/%7Ejoe` | 波浪号 `~` 在某些系统中有特殊含义（如 Unix 用户主目录），所以编码为 `%7E` |
| ` `（空格） | 32 (0x20) | `http://www.joes-hardware.com/more%20tools.html` | 空格在 URL 中是不允许的（会破坏解析），编码为 `%20` |
| `%` | 37 (0x25) | `http://www.joes-hardware.com/100%25satisfaction.html` | 百分号本身是转义前缀，如果要表示字面意义的 `%`，需要编码为 `%25` |

---

## 三、为什么要对这些字符编码？

- **波浪号 `~`**：虽然许多 Web 服务器会直接接受 `~`，但根据规范它属于“不安全字符”，在某些网络环境下可能被修改或解释错误，编码可保证最大兼容性。
- **空格**：URL 中不允许有空格（它会分隔 URL 与其他文本），必须编码为 `%20`。
- **百分号 `%`**：因为 `%` 是转义标识符，要表示字面百分号（如“100%满意”），就必须写成 `%25`，否则解析器会把 `%sa` 等后续字符误认为是编码。

---

## 四、现代 Web 中的实际应用

- **表单提交**：GET 请求中，浏览器会自动对 query 字符串中的特殊字符进行百分号编码。
- **前端开发**：JavaScript 提供 `encodeURIComponent()` 和 `encodeURI()` 函数来进行 URL 编码。
- **文件路径**：如果 URL 包含中文、空格、特殊符号，必须编码后才能作为合法 URL 发送。

---

## 五、常见错误

- 手动拼接 URL 时**没有对参数值进行编码**，例如直接写 `?name=张三&city=北京` → 非 ASCII 字符不应该原样出现，应编码为 `%E5%BC%A0%E4%B8%89` 等。
- **过度编码**：对已经编码过的字符串再次编码，例如 `%20` 变成 `%2520`（`%25` 是 `%` 的编码）。这会导致服务器解码后得到错误的值。

---

## 一、保留字符（Reserved）

这些字符在 URL 中有**特定的语法含义**，如果作为普通数据（例如文件名中的 `/` 或参数值中的 `&`）出现，必须被编码。

| 字符 | 保留用途 |
|------|----------|
| `%` | 作为转义编码的标记（`%` 后跟两位十六进制数） |
| `/` | 分隔路径中的段（segments） |
| `.` | 路径中的单点（当前目录）或双点（上级目录） |
| `..` | 路径中的上级目录引用 |
| `#` | 分隔 fragment（片段标识符） |
| `?` | 分隔 query（查询字符串） |
| `;` | 分隔 params（参数） |
| `:` | 分隔 scheme/host/port 或 user:password |
| `$` , `+` | 保留（某些协议有特殊含义） |
| `@` , `&` , `=` | 在某些协议的上下文中具有特殊含义（例如 `@` 分隔用户名和主机，`&` 分隔 query 中的键值对，`=` 连接键和值） |

### 示例

- 想表示一个**实际包含 `/` 的文件名**（如 `"a/b.html"`），需要将其中的 `/` 编码为 `%2F`：`a%2Fb.html`
- 想表示 query 参数值中的 `&`（例如 `name=AT&T`），需要将其编码为 `%26`：`name=AT%26T`

---

## 二、受限字符（Restricted）

这些字符本身**不是 US-ASCII 可打印字符**，或者可能被某些传输网关（如邮件网关）错误处理，因此**应避免直接使用**，必须编码。

| 类别 | 说明 |
|------|------|
| `{ } \| \ ^ ~ [ ] '` | 这些字符不安全，因为某些传输代理（如网关）会对它们进行特殊处理，可能导致数据损坏 |
| 双引号 `"`、尖括号 `<>`（表中未单独列出，但属于常见不安全字符） | 常用于分隔 URL 本身（例如在文本中标记链接），如果出现在 URL 内部可能被误解析 |
| `0x00 – 0x1F`，`0x7F` | 属于 **US-ASCII 非打印控制字符**（如换行、制表符、DEL），在 URL 中不允许出现，必须编码 |
| `> 0x7F` | 超出 7 位 ASCII 范围的字符（如扩展 ASCII 码），不属于 US-ASCII 可打印集，必须编码 |

### 特别注意

- `~`（波浪号）虽然在很多 Web 服务器中直接使用，但它属于受限字符，理论上最好编码为 `%7E`。不过由于历史原因，很多浏览器和服务器会直接接受 `~`。
- 中文字符等非 ASCII 字符（`> 0x7F`）在 URL 中必须按 UTF-8 或其他字符集编码后，再进行百分号编码。

---

## 三、最佳实践

- **对于保留字符**：如果它们不是用于其保留的语法目的（例如 `?` 不是作为 query 分隔符，而是作为数据），则必须编码。
- **对于受限字符**：应始终编码，除非你知道目标服务器能正确处理原始字符。
- **对于非 ASCII 字符**（如中文、日文、变音字母）：先使用 UTF-8 编码得到字节序列（例如 `中` → `E4 B8 AD`），再对每个字节进行百分号编码（`%E4%B8%AD`）。

---

| 类别 | 字符 | 处理方式 |
|------|------|----------|
| 保留字符 | `% / . .. # ? ; : $ , + @ & =` | 非保留用途时须编码 |
| 不安全/受限字符 | `{ } \| \ ^ ~ [ ] '` 等 | 建议始终编码 |
| 控制字符 | `0x00–0x1F`, `0x7F` | 必须编码 |
| 非 ASCII（>0x7F） | 中文、é、€ 等 | 先 UTF-8 再百分号编码 |

下面我把这段话的核心观点总结一下。

---

## 一、为什么直接写 `~` 也能访问？

你输入 `http://www.joes-hardware.com/~joe`，浏览器和服务器通常都能正确处理，不会出错。这是因为：

- 虽然 `~` 在理论上是“不安全/受限字符”（Table 2-3 中列出了 `{ } | \ ^ ~ [ ] '` 等），但在 **HTTP/HTTPS** 的实际传输中，大多数 Web 服务器和浏览器都能直接识别 `~`，并不会造成问题。
- 不同的传输协议对同一字符的容忍度不同。`~` 在 HTTP 中没问题，但在某些老旧邮件网关中可能被篡改。

但书中强调：**“能用”不等于“规范”**。对于追求健壮性和跨平台兼容的应用程序，仍然应该对不安全字符进行编码。

---

## 二、最佳实践：谁来编码？何时编码？

- **最佳位置**：**最初接收用户输入的客户端应用**（如浏览器）负责判断哪些字符需要编码。因为只有这个应用知道 URL 各部分的原始用途（例如用户是想在 path 中放一个 `~` 作为数据，还是作为 scheme 的一部分）。
- **目标**：将 URL 转换为 **规范形式（canonical form）**，即所有不安全字符均被编码。这样 URL 可以在不同应用间安全传递，不会因字符歧义而产生误解。

---

## 三、两个极端：不编码 vs. 全部编码

| 做法 | 风险 |
|------|------|
| **不编码不安全字符**（仅依赖“碰巧能工作”） | 在某些网络环境或协议中可能失败，且违反规范 |
| **对所有字符都编码（包括安全字符）** | 可能导致下游应用异常，因为一些应用可能认为安全字符不应编码，或者对编码后的安全字符解析错误（例如双重编码问题） |

因此推荐：**只编码那些确实不安全或保留用途之外的字符**，不要过度编码。

---

## 四、安全考虑：恶意编码绕过过滤

- 攻击者可能故意对**安全字符**（如字母）也进行编码，例如将 `a` 编码为 `%61`，以绕过某些基于模式匹配的 URL 过滤系统（如家长控制、防火墙）。
- 对策：任何对 URL 进行解析或匹配的应用，都应该**先解码 URL**，再对解码后的内容进行模式匹配，否则可能被欺骗。

---

文中提醒：以上讨论主要针对**客户端应用**。对于代理等中间设备，修改（如编码）URL 可能带来其他问题，将在第 6 章“In-Flight URI Modification”中讨论。

---

## 总结

- `~` 等不安全字符在 HTTP 中常被接受，但从健壮性角度应编码。
- 最佳做法：客户端编码不安全字符，得到规范形式的 URL。
- 不要过度编码（安全字符保留原样）。
- 解析 URL 的程序应**先解码**再匹配，以防绕过攻击。

- **file** 省略 host 时代表本机。

## 二、URL 的局限性与未来

### 1. URL 是地址，不是名字

- URL 绑定了资源**当前的位置**（特定的主机、端口、路径）。
- **问题**：如果资源被移动（例如换了服务器或目录），原来的 URL 就会失效，而且无法通过原 URL 找到新位置。

### 2. URN（统一资源名称）

- **目标**：为资源提供一个**永久的、位置无关的名字**，就像一个人的姓名一样，无论他搬到哪里，都能通过这个名字找到他。
- IETF 一直在推动 URN 标准。

### 3. PURL（持久统一资源定位符）

- 一种**使用 URL 模拟 URN 功能**的过渡方案。
- 原理：引入一个**中间解析服务器**，资源实际 URL 会变化，但 PURL 不变。客户端请求 PURL，解析服务器返回当前有效的实际 URL（通过 HTTP 重定向）。
- 示例图 Figure 2-6（未提供）展示了这个流程。
- 更多信息：http://purl.oclc.org

### 4. 为什么 URN 尚未普及？

- 标准化过程缓慢。
- 需要大量基础设施变更。
- URL 已经深入人心，迁移成本太高。
- 短期内 URL 仍将是主流。
