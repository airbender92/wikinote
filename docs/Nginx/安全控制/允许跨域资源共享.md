# 7.2 允许跨域资源共享（CORS）**超级通俗彻底讲懂**
我保证用**最简单、最实用、不绕弯**的方式讲👇

# 核心一句话
**CORS = 浏览器的安全规则，不让随便跨域名调用接口。
你必须在 Nginx 里加响应头，告诉浏览器：“允许这个域名跨域调用我”。**

---

# 先给你看这段配置在干嘛
```nginx
map $request_method $cors_method {
    OPTIONS 11;
    GET     1;
    POST    1;
    default 0;
}
```
作用：
- 把 **OPTIONS** 标记为 11（预检请求）
- 把 **GET、POST** 标记为 1
- 其他请求 0

```nginx
if ($cors_method ~ '1') {
    add_header Access-Control-Allow-Methods "GET,POST,OPTIONS";
    add_header Access-Control-Allow-Origin "*.example.com";
    add_header Access-Control-Allow-Headers "...";
}
```
作用：
- 允许跨域
- 允许的来源：`*.example.com`
- 允许的请求头、方法

```nginx
if ($cors_method = '11') {
    add_header Access-Control-Max-Age 1728000;
    return 204;
}
```
作用：
- 处理 OPTIONS **预检请求**
- 告诉浏览器缓存 20 天
- 直接返回 204，不进后端

---

# 什么是跨域？
前端域名：`https://a.com`  
接口域名：`https://b.com`  

浏览器会阻止，因为：
## **不同域名 = 跨域**

你必须在 Nginx 配置 CORS 头，浏览器才允许调用。

---

# 三个最重要的跨域头（必须记住）
### 1. **Access-Control-Allow-Origin**
允许哪些域名跨域  
最常用：
```
Access-Control-Allow-Origin *
```
（允许所有人）

### 2. **Access-Control-Allow-Methods**
允许哪些请求方法：GET, POST, PUT, DELETE...

### 3. **Access-Control-Allow-Headers**
允许哪些自定义请求头

---

# 什么是 OPTIONS 预检请求？
浏览器在跨域时，会**先发一个 OPTIONS 请求**，问服务器：
- 你允许我跨域吗？
- 你允许哪些方法？
- 允许哪些头？

服务器必须正确响应，否则浏览器**直接拦截请求**。

---

# 这段配置的完整逻辑（超级清晰）
1. 请求进来
2. 如果是 **GET/POST** → 加跨域头
3. 如果是 **OPTIONS** → 返回跨域允许 + 204
4. 其他方法 → 不加跨域头（默认禁止）

---

# 生产环境最标准、最简单的 CORS 配置（直接复制用）
```nginx
location / {
    # 允许所有域名跨域（生产可改成具体域名）
    add_header Access-Control-Allow-Origin *;
    add_header Access-Control-Allow-Methods "GET,POST,PUT,DELETE,OPTIONS";
    add_header Access-Control-Allow-Headers "Content-Type,Authorization";

    # 处理预检 OPTIONS
    if ($request_method = OPTIONS) {
        add_header Access-Control-Max-Age 1728000;
        add_header Content-Length 0;
        add_header Content-Type text/plain;
        return 204;
    }
}
```

**CORS = 跨域访问许可
OPTIONS = 浏览器预检
add_header = 告诉浏览器允许跨域
Nginx 配置好，前端就能跨域调接口了**
