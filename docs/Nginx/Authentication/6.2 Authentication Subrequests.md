# 6.2 子请求认证（Authentication Subrequests）
## **超级通俗、彻底讲懂**

我直接用**最直白、最实用、不绕弯**的方式给你讲透👇

# 核心一句话
**auth_request = 让 Nginx 先偷偷请求一个“认证接口”，
接口返回 200 → 允许访问；
返回 401/403 → 直接拒绝。
这是 Nginx 对接你自己的登录系统、SSO、鉴权服务的终极方式。**

---

# 1. 完整流程（一秒懂）
1. 用户请求 `/private/`
2. **Nginx 自动先发送一个子请求到 `/auth`**
3. `/auth` 去请求你的后端鉴权服务（验证token、session、cookie）
4. **鉴权服务返回 200 → 允许访问**
5. **返回 401/403 → 拒绝访问**
6. 用户完全不知道中间发生了这一步

---

# 2. 你给的配置逐行解释
```nginx
# 受保护的目录
location /private/ {
    auth_request /auth;               # 关键：先去 /auth 验证
    auth_request_set $auth_status $upstream_status;  # 把认证接口的状态码存成变量
}

# 内部认证点（客户端不能直接访问）
location = /auth {
    internal;                         # 标记为内部接口，外部无法访问
    proxy_pass http://auth-server;    # 转发到你的鉴权服务
    
    proxy_pass_request_body off;      # 不发送请求体（鉴权一般不需要body）
    proxy_set_header Content-Length "";
    proxy_set_header X-Original-URI $request_uri;  # 把原始地址传给鉴权服务
}
```

---

# 3. 最关键规则（必须记住）
### **鉴权服务返回什么，Nginx 就怎么处理：**
- **2xx → 认证通过 ✅**
- **401 → 未登录 ❌**
- **403 → 无权限 ❌**

**Nginx 完全信任你的鉴权服务。**

---

# 4. 为什么要加这些配置？
## `proxy_pass_request_body off;`
- 鉴权一般**不需要请求体**
- 关掉可以**减少带宽、提高速度**

## `Content-Length ""`
- 因为没发body，必须清空长度头，否则协议错误

## `X-Original-URI $request_uri`
- 让鉴权服务知道**用户访问的是哪个地址**
- 方便做**权限判断**（比如：用户能不能访问 /admin）

---

# 5. 这个功能有多强大？
你可以用它做：
- **登录验证**
- **Token 验证**
- **Cookie 验证**
- **Session 验证**
- **RBAC 权限控制**
- **SSO 单点登录**
- **OAuth2**
- **自定义鉴权系统**

**所有复杂逻辑都交给你的后端服务，Nginx 只负责“拦不拦”。**

---

# 6. 超级实用场景示例
1. 用户访问 `/admin`
2. Nginx 自动请求你的 `http://auth-server`
3. 你的服务检查 Cookie/Token
4. 是管理员 → 返回 200
5. 不是 → 返回 403
6. Nginx 拦截或放行

---

# 7. 这是生产环境最常用的认证方式
比 Basic Auth 强大一万倍，
是**现代架构必备**的 Nginx 鉴权模式。

---

**auth_request = 先请求你的鉴权接口
200 = 过
401/403 = 拒绝
internal = 防止外部直接访问
所有复杂登录逻辑都交给后端服务**
