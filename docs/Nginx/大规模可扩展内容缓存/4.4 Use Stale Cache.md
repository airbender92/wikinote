# 4.4 使用陈旧缓存（Use Stale Cache）· 终极通俗解释
你刚才担心的“请求冲击后台”，这一节就是**终极解决方案**！

# 核心一句话
**后端挂了、超时、报错、甚至正在更新缓存时，Nginx 不把请求转发到后端，直接返回旧的缓存给用户。
用户无感，后端零压力，简直是高可用神器。**

---

# 1. 配置直接翻译
```nginx
proxy_cache_use_stale 
    error 
    timeout 
    invalid_header 
    updating 
    http_500 
    http_502 
    http_503 
    http_504 
    http_403 
    http_404 
    http_429;
```

每个参数意思：
- `error`：后端连不上
- `timeout`：请求超时
- `invalid_header`：响应头非法
- `updating`：**缓存正在更新中**
- `http_xxx`：后端返回这些错误码

只要满足任意一条，**直接返回旧缓存，不打扰后端**。

---

# 2. 最关键的：`updating`（解决你刚才的担忧）
```nginx
proxy_cache_use_stale updating;
```

### 作用：
**缓存过期、正在重新获取时 → 直接返回旧缓存，不让请求去后端排队。**

### 效果：
- 缓存过期
- 第一个请求**后台悄悄更新缓存**
- **其他所有请求直接返回旧数据，完全不排队、不冲击后端**
- 用户拿到页面，速度极快
- 后端压力 = **0**

这就是**彻底解决惊群、彻底防冲击**的终极配置。

---

# 3. 这个机制为什么牛逼？
- 后端宕机 → 页面照样打开
- 后端超载 → 不给它加压力
- 缓存更新瞬间 → 不拥堵、不雪崩
- 用户完全无感知
- 给运维抢救时间

---

# 4. 和缓存锁的搭配（生产标配）
```nginx
proxy_cache_lock on;
proxy_cache_use_stale updating error timeout http_503;
```

### 效果拉满：
1. 只让一个请求去后台更新
2. 其他请求**直接返回旧缓存**，不等、不排队、不冲击
3. 后端压力极小
4. 服务永远可用

---

**proxy_cache_use_stale = 后端不行就用旧缓存
updating = 缓存更新时也直接用旧的，完全不堵后端
这是高并发、高可用架构的神级配置！**
